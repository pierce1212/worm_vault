# **第 1 阶段：先建立全局图，半天**
先搞清楚 UDS 是什么：

- UDS = Unified Diagnostic Services，主要标准是 `ISO 14229`
- 常跑在 `CAN + ISO-TP` 上，也可以跑在 `DoIP`
- 典型用途：读写数据、读故障码、清故障码、刷写、例程控制、安全解锁、会话切换

你要先记住这条链路：

```text
应用层：UDS, ISO 14229
传输层：ISO-TP, ISO 15765-2
网络/数据链路：CAN / CAN FD / Ethernet DoIP
```


# **第 2 阶段：先学 ISO-TP，1 天**
UDS 报文大于 8 字节时，必须靠 ISO-TP 分包。这个一定要先会，否则后面看不懂刷写和长响应。

<mark style="background: #FF5582A6;">重点掌握：</mark>

- **<mark style="background: #BBFABBA6;">单帧 `Single Frame`</mark>**
	- 一条消息仅用一帧即可完成传输，即为单帧（SingleFrame）
	- ![](assets/2026.06.01%20学习建议/file-20260702135226637.png)

	<mark style="background: #FFF3A3A6;">普通 CAN </mark>: 最多 7 字节 UDS 数据。Single Frame PCI 常见是 0 x 0 .低 4 bit 的 `L` 表示长度,比如 0322 F 186.
	<mark style="background: #FF5582A6;">CAN FD</mark>: `00` 告诉 CanTp：这是 CAN FD 单帧的长格式 。23 实际 UDS 长度是 35.
	
```
PCI 是 **Protocol Control Information**，中文可以叫 **协议控制信息**
这是一帧什么类型？
后面有多少诊断数据？
是不是还有后续帧？
这是第几帧？
要不要继续发？

PCI 头是真正上 CAN/CAN FD 总线的字节，接收方必须靠它完成 CanTp 解包。
```

| 字节                | 含义                            |
| ----------------- | ----------------------------- |
| `00`              | CAN FD Single Frame 扩展长度格式    |
| `23`              | UDS payload 长度，`0x23 = 35` 字节 |
| `22`              | UDS SID：ReadDataByIdentifier  |
| `F1 86 F1 90 ...` | 一次读取多个 DID                    |
| 后面的 `00`          | padding，不属于 UDS 数据            |

| PCI    | 帧类型                   | 作用                |
| ------ | --------------------- | ----------------- |
| `0x0?` | Single Frame, SF      | 单帧，数据一次发完         |
| `0x1?` | First Frame, FF       | 首帧，后面还有连续帧        |
| `0x2?` | Consecutive Frame, CF | 连续帧               |
| `0x3?` | Flow Control, FC      | 流控帧，告诉对方继续发/等待/停止 |

<mark style="background: #BBFABBA6;">- 物理寻址 vs 功能寻址</mark>

物理寻址是一对一，功能寻址是一对多广播。功能寻址通常只用于请求，不适合需要长响应的服务。
物理寻址：
		tester -> ECU: 0 x 72 B  10 03
		ECU -> tester: 0 x 7 AB  50 03
含义是：我就找这个 ECU，所以这个 ECU 可以回响应。
功能寻址：
		tester -> all ECUs: 0 x 7 DF  3 E 80
含义是：谁支持这个功能谁处理。功能寻址通常用于广播类请求，比如 tester present、清 DTC、读某些公共信息。

# **第 3 阶段：学 UDS 报文格式，半天**
核心格式：

```text
请求：SID + 参数
正响应：SID + 0x40 + 数据
负响应：0x7F + 原始 SID + NRC
```

例子：

```text
请求扩展会话：10 03
正响应：50 03 ...
负响应：7F 10 22
```

你必须熟悉负响应码 NRC，尤其是：

- `0x10` general reject
- `0x11` service not supported
- `0x12` sub-function not supported
- `0x13` incorrect message length
- `0x22` conditions not correct
- `0x31` request out of range
- `0x33` security access denied
- `0x35` invalid key
- `0x36` exceeded number of attempts
- `0x78` response pending

# **第 4 阶段：优先掌握最常用服务，2-3 天**
按这个顺序学，不要按标准目录顺序学。

第一组：诊断入口类

```text
0x10 Diagnostic Session Control
0x11 ECU Reset
0x27 Security Access
0x3E Tester Present
0x28 Communication Control
0x85 Control DTC Setting
```

第二组：数据读写类

```text
0x22 Read Data By Identifier
0x2E Write Data By Identifier
0x2F Input Output Control By Identifier
```

第三组：故障码类

```text
0x19 Read DTC Information
0x14 Clear Diagnostic Information
```

第四组：例程控制

```text
0x31 Routine Control
```

第五组：刷写类，最后学

```text
0x34 Request Download
0x36 Transfer Data
0x37 Request Transfer Exit
0x31 Erase / Checksum Routine
0x11 Reset
```

# **第 5 阶段：重点理解会话、安全、时序，1-2 天**
UDS 不是“发什么回什么”这么简单，很多服务依赖状态机。

必须理解：

- 默认会话 `Default Session`
- 扩展会话 `Extended Session`
- 编程会话 `Programming Session`
- 安全等级 `Security Level`
- `Seed / Key`
- `P2` 和 `P2*` 响应时间
- `S3 Server` 会话保持时间
- `Tester Present 0x3E` 保活

一个典型流程：

```text
10 03        进入扩展会话
27 01        请求 seed
27 02 key    发送 key
22 F1 90     读取 VIN 或其他 DID
3E 00        保活
```

**第 6 阶段：做实战练习，3-5 天**

- `python-can`

- `udsoncan`

练习任务：

