## 1. 给初学者的直白解释

CanSM 是 CAN State Manager，中文可以叫 CAN 状态管理模块。它不是负责打包信号的，也不是负责 NM 报文内容的，而是**<mark style="background: #BBFABBA6;">负责把本 ECU 的 CAN 通道切到正确的通信状态</mark>**。

最直白地说：

```text
ComM 说：我要 FullCom
**CanSM 就去让 CAN 收发器 Normal、CAN Controller Started、CanIf/Com 可以收发**

ComM 说：我要 NoCom
CanSM 就去停 Controller、让收发器 Standby/Sleep、关闭 PDU 收发

底层说：BusOff 了
**CanSM 就负责通知 ComM/BswM，并按快慢恢复策略重启 CAN Controller**
```

所以 CanSM 管的是“本 ECU 的 CAN 硬件和通信通道状态”。NM 管的是“网络上大家醒不醒、睡不睡”。ComM 是上层通信总开关，CanSM 是 CAN 这一路总线的执行者。

可以这样分工：

| 模块 | 直白职责 |
|---|---|
| ComM | 上层通信请求总管，决定要 Full/Silent/No 通信 |
| CanSM | 执行 CAN 通道状态切换，控制 Controller/Transceiver/CanIf |
| CanIf | CAN 抽象接口，真正调 Can Driver/CanTrcv |
| Can Driver | 控制 MCU 内部 CAN 控制器 |
| CanTrcv | 控制外部 CAN 收发器，例如 TJA1145 |
| CanNm | 发收 NM 报文，管理网络唤醒和休眠 |
| BswM | 根据模式变化执行规则和 ActionList |

## 5. CanSM 的正常通信流程

### 5.1 从 NoCom 到 FullCom

典型调用链：

```text
应用/诊断/NM 需要通信
  -> ComM_RequestComMode(User, COMM_FULL_COMMUNICATION)
  -> ComM_MainFunction
  -> CanSM_RequestComMode(Channel, COMM_FULL_COMMUNICATION)
  -> CanSM_CurrNw_Mode_en = CANSM_BSM_S_PRE_FULLCOM
  -> CanSM_MainFunction
  -> CanSM_NetworkModeTrans
  -> CanSM_NO2FULL_COM
  -> CanSM_NormalTrcv
  -> CanIf_SetTrcvMode(... NORMAL)
  -> CanSM_StartCtrl
  -> CanIf_SetControllerMode(... CAN_CS_STARTED)
  -> CanIf_SetPduMode / Com I-PDU group 允许收发
  -> ComM_BusSM_ModeIndication(... COMM_FULL_COMMUNICATION)
  -> BswM_CanSM_CurrentState(... CANSM_BSWM_FULL_COMMUNICATION)
```

在当前工程里，`CanSM_RequestComMode()` 收到 `COMM_FULL_COMMUNICATION`，如果当前处于 `CANSM_BSM_S_NOCOM`，会把当前网络模式改为：

```c
CanSM_CurrNw_Mode_en[lnetwork] = CANSM_BSM_S_PRE_FULLCOM;
```

然后 `CanSM_MainFunction()` 每 10ms 调状态机，把 Transceiver 和 Controller 拉起来。

### 5.2 从 FullCom 到 NoCom

典型调用链：

```text
应用/ComM 释放通信
  -> ComM_RequestComMode(User, COMM_NO_COMMUNICATION)
  -> ComM_MainFunction
  -> CanSM_RequestComMode(Channel, COMM_NO_COMMUNICATION)
  -> CanSM_CurrNw_Mode_en = CANSM_BSM_S_PRE_NOCOM
  -> CanSM_MainFunction
  -> CanSM_NetworkModeTrans
  -> BswM_CanSM_CurrentState(... CANSM_BSWM_NO_COMMUNICATION)
  -> 停 Controller
  -> Controller Sleep
  -> Transceiver Standby/Sleep
  -> CanSM_PreNoCom_Exit
  -> ComM_BusSM_ModeIndication(... COMM_NO_COMMUNICATION)
```

当前工程配置了 `CanSMTransceiverId`，所以 NoCom 过程不仅要停 CAN Controller，还会处理 TJA1145 收发器模式。

### 5.3 FullCom 与 SilentCom

CanSM 支持 3 个 ComM 通信模式：

| ComM 模式 | CanSM 直白动作 |
|---|---|
| `COMM_FULL_COMMUNICATION` | Rx/Tx 都打开 |
| `COMM_SILENT_COMMUNICATION` | 通常 Rx 可用，Tx 关闭 |
| `COMM_NO_COMMUNICATION` | 通信关闭，Controller/Trcv 进入低功耗或停止状态 |

BusOff 时 CanSM 会把通道切到 SilentCom/BOR 相关状态，这是工程上非常重要的点：BusOff 恢复期间通常还能接收报文，但不能正常发送报文。

## 6. BusOff 处理机制

### 6.1 BusOff 是什么

BusOff 是 CAN Controller 自己因为发送错误太多，被迫退出总线。根本条件是：

```text
TEC > 255
```

这里的 TEC 是 Transmit Error Counter，发送错误计数。重点：

- BusOff 是本节点自己的 CAN Controller 出问题。
- 其他节点发错误帧，不会直接让本节点 BusOff。
- BusOff 后 ECU 的 MCU 还在跑，OS 任务还在调度，只是对应 CAN Controller 暂时不参与总线通信。

### 6.2 当前工程 BusOff 调用链

有道云笔记和当前工程代码可以对应成：

```text
CAN Controller 检测 BusOff
  -> Can Driver 中断或轮询处理
  -> CanIf_ControllerBusOff(controllerId)
  -> CanSM_ControllerBusOff(controllerId)
  -> CanSM_BusOff_Indicated_ab[network] = TRUE
  -> CanSM_MainFunction 10ms 周期处理
  -> CanSM_BusOffTransitions(network)
```

当前工程 `CanSM_ControllerBusoff.c` 会根据 ControllerId 找到 network，然后设置 BusOff 标志。`CanSM_MainFunction()` 周期检查：

```c
if (CanSM_BusOff_Indicated_ab[CanSM_NetworkIdx_u8] == TRUE)
{
    CanSM_BusOffTransitions(CanSM_NetworkIdx_u8);
}
```

### 6.3 BusOff 发生后 CanSM 做什么

典型动作：

1. 关闭 Tx，避免故障节点继续干扰总线。
2. 通知 ComM：进入 `COMM_SILENT_COMMUNICATION`。
3. 通知 BswM：进入 `CANSM_BSWM_BUS_OFF` 或相关状态。
4. 重启 CAN Controller。
5. 根据 BusOff 连续次数选择 L1 快恢复或 L2 慢恢复。
6. 等恢复时间到，再重新允许 Tx。
7. 等 `CanSMBorTimeTxEnsured`，确认一段时间内没再次 BusOff，就认为恢复成功。

当前工程配置：

| 参数 | 值 | 直白解释 |
|---|---:|---|
| `CanSMBorTimeL1` | `0.1 s` | 前 10 次以内，100ms 后尝试恢复发送 |
| `CanSMBorCounterL1ToL2` | `10` | 连续 BusOff 达到 10 次，进入慢恢复 |
| `CanSMBorTimeL2` | `1.0 s` | 慢恢复阶段，1s 后再尝试恢复 |
| `CanSMBorTimeTxEnsured` | `1.0 s` | 恢复后 1s 内不再 BusOff，认为恢复稳定 |

### 6.4 为什么需要快慢恢复

有道云笔记里总结得很清楚：从 CAN 协议角度，BusOff 后重新参与通信至少要等待连续 128 个 11 bit 隐性位。但工程上不会只按这个最短物理时间来恢复。

原因：

- 偶发干扰导致的 BusOff，应该尽快恢复，所以有 L1 快恢复。
- 如果故障一直存在，节点反复快速加入总线会继续干扰其他节点，所以要转 L2 慢恢复。
- OEM 会根据网络负载、功能安全、诊断策略定义 L1/L2 时间和阈值。

当前工程 L1=100ms、L2=1s，是比较典型的“先快速尝试，连续失败后降低干扰”的策略。

## 7. CanSM 和 Transceiver/Controller 的关系

CanSM 的关键价值在这里：它不是只管 MCU 内部 CAN Controller，还要协调外部 CAN Transceiver。

当前工程配置：

```text
CanSMTransceiverId -> CanIfTrcvCfg_1145
CanSMControllerId  -> CanIfCtrlDrvCfg/ETAS_CAN
```

所以 FullCom 时要保证：

```text
TJA1145 / HS-CAN = Active/Normal
CAN Controller = STARTED
CanIf PduMode = ONLINE
```

NoCom/休眠时要保证：

```text
CAN Controller = STOPPED/SLEEP
Transceiver = STANDBY/SLEEP 或可唤醒状态
CanIf PduMode = OFFLINE
```

有道云笔记中的工程问题非常关键：Transceiver 模式如果选错，可能出现“ECU 上电了，但 CAN 通道不在线”。例如 TJA1145 的 HS-CAN 如果处于 Offline/Offline Bias，收发器不做数字/模拟信号转换，即使 CAN Controller 以为自己在发，也可能发不出去，甚至引发 BusOff。

排查 CAN 不在线时，不要只看 CanSM 状态是否 FullCom，还要看：

- TJA1145 真实模式是否 Active/Normal。
- Vcc 是否稳定，是否触发欠压导致 HS-CAN 进入 Offline Bias。
- CanIf_SetTrcvMode 是否返回 OK。
- CanSM_TransceiverModeIndication 是否回来。
- CanIf_SetControllerMode 是否返回 OK。
- CanSM_ControllerModeIndication 是否回来。

## 8. 当前工程 CanSM 和其他模块的接口

### 8.1 ComM -> CanSM

ComM 请求 CAN 通道模式：

```c
CanSM_RequestComMode(NetworkHandleType network, ComM_ModeType ComM_Mode)
```

当前工程 `CanSM_GetHandle()` 会用 ComM channel id 找 CanSM network：

```c
if (CanSM_ConfigSet_pcst->CanSM_NetworkConf_pcst[loop].ComM_channelId_uo == ComMChannelId)
```

当前只有一个 ComM channel，id 为 0。

### 8.2 CanSM -> ComM

CanSM 状态切换完成后通知 ComM：

```c
ComM_BusSM_ModeIndication(channel, mode)
```

这就是 ComM 判断 CAN 通道是否真的进入 Full/Silent/No 的关键回调。

### 8.3 CanSM -> BswM

CanSM 会通知 BswM 当前状态：

```c
BswM_CanSM_CurrentState(channel, CANSM_BSWM_...)
```

BswM 再根据规则执行 ActionList，例如打开/关闭 PDU Group、通知模式、执行初始化等。

### 8.4 CanIf -> CanSM

CanIf 通知 CanSM 底层状态变化：

- `CanSM_ControllerModeIndication`
- `CanSM_TransceiverModeIndication`
- `CanSM_ControllerBusOff`
- `CanSM_ClearTrcvWufFlagIndication`
- `CanSM_CheckTransceiverWakeFlagIndication`

这些 Indication 如果没回来，CanSM 可能卡在 WAIT 子状态，然后靠 `CanSMModeRequestRepetitionTime` 和 `CanSMModeRequestRepetitionMax` 重试。

## 9. 常见工程问题

### 9.1 ComM 请求 FullCom，但 CAN 不发报文

优先检查：

1. `ComM_RequestComMode()` 是否真的请求了 FullCom。
2. `CanSM_RequestComMode()` 是否被调用。
3. `CanSM_CurrNw_Mode_en[0]` 是否从 `NOCOM` 进入 `PRE_FULLCOM` 再到 `FULLCOM`。
4. `CanSM_NormalTrcv()` 是否成功。
5. `CanIf_SetTrcvMode(... NORMAL)` 是否 OK。
6. `CanSM_TransceiverModeIndication()` 是否回来。
7. `CanIf_SetControllerMode(... STARTED)` 是否 OK。
8. `CanSM_ControllerModeIndication()` 是否回来。
9. CanIf PDU Mode 是否 Tx/Rx Online。
10. Com/PduR 的 I-PDU Group 是否开启。

### 9.2 CanSM 卡在 PreFullCom 或 PreNoCom

常见原因：

- Controller mode indication 没回来。
- Transceiver mode indication 没回来。
- `CanIf_SetControllerMode()` 返回 `E_NOT_OK`。
- `CanIf_SetTrcvMode()` 返回 `E_NOT_OK`。
- `CanSMModeRequestRepetitionMax = 5` 次重试耗尽。
- CanSM 主函数没按 10ms 调度。

### 9.3 BusOff 恢复不了

常见原因：

- 物理层故障仍存在，例如 CANH/CANL 短路、终端电阻问题、收发器欠压。
- Controller 重启失败。
- 收发器没有在 Active/Normal。
- BusOff 一恢复就再次发送错误帧，连续进入 L2 慢恢复。
- 测试脚本制造 BusOff 后没有恢复 CANoe Channel。
- DEM/DTC 只记录了故障，但没有解决底层恢复链路。

### 9.4 BusOff 后还能收到报文吗

通常 BusOff 恢复期间 Tx 会被关闭，但 Rx 可能仍然可用，具体取决于 Controller/CanIf/CanSM 的状态和供应商实现。有道云笔记里也强调：快/慢恢复期内可能可以接收报文。

所以看到“BusOff 后还能收到其他节点报文”并不一定矛盾；真正要确认的是本节点 Tx 是否被关闭、何时重新放开。

### 9.5 BusOff DTC 谁上报

有两种常见做法：

- CanSM 作为 BSW 调 `Dem_ReportErrorStatus()`。
- SWC 通过 RTE 调 `Dem_SetEventStatus()`。

当前工程 CanSM 配置中 BusOff DEM Event 引到了：

```text
DTC_0xC11001_Event
```

所以这里要结合 DEM 配置确认是否由 CanSM 上报、是否有去抖/使能条件、是否连续多次 BusOff 才报 FAILED。

### 9.6 Transceiver 模式导致 ECU 不在线

这是工程上很容易误判的问题。表象是：

- CanSM/ComM 看起来请求了 FullCom。
- 其他节点在总线上正常发。
- 本 ECU 不发，甚至误报其他节点丢失。

可能根因：

- TJA1145 HS-CAN 没进入 Active。
- Vcc 不稳定导致 HS-CAN Offline Bias。
- Transceiver SPI 配置错误。
- CanSM 认为模式切换成功，但硬件实际不工作。

排查时必须读收发器状态寄存器，不能只看软件变量。

## 10. 易错点清单

- 把 CanSM 和 CanNm 混在一起：CanSM 管本 ECU CAN 硬件状态，CanNm 管网络 NM 报文和网络睡醒。
- 以为 ComM FullCom 就一定能发报文：还要 CanSM、CanIf、CanTrcv、Can Driver 都成功。
- 以为 BusOff 是别人发错误帧导致自己 BusOff：BusOff 本质是本节点发送错误计数 TEC 超过阈值。
- 只测 BusOff DTC，不测快慢恢复：DTC 上报只是诊断结果，恢复机制才是通信鲁棒性的核心。
- 忽略 Transceiver：TJA1145 模式或供电异常会直接导致 CAN 通道不在线。
- 忽略 Indication：CanSM 很多状态切换依赖 Controller/Transceiver mode indication。
- PN support 误解：当前 CanSM PN support 打开，但 CanNm global PN 关闭，不能当完整 PN 网络分析。
- ModeRequest 时间单位误解：ARXML 是秒，生成代码按 10ms tick 换算。
- BusOff 后马上期待 Tx 恢复：当前 L1 也要等 100ms，连续 BusOff 后 L2 要等 1s。

- `CanSM_Num_Unsuccessful_ModeReq_au8[0]`

期望状态：

```text
NOCOM -> PRE_FULLCOM -> FULLCOM
```

### 11.2 NoCom/休眠调试

目标：确认释放通信后 CAN 通道关闭。

断点/观察：

- `CanSM_RequestComMode(... COMM_NO_COMMUNICATION)`
- `CanSM_NetworkModeTrans`
- `CanSM_DeInitPnSupported` 或 `CanSM_DeInitPnNotSupported`
- `CanSM_StopCtrl`
- `CanSM_SleepCtrl`
- `CanSM_StandbyTrcv`
- `CanSM_PreNoCom_Exit`

关键变量：

- `CanSM_CurrNw_Mode_en[0]`
- `CanSM_PreNoCom_Substates_en[0]`
- `CanSM_NoComRequested_ab[0]`

期望状态：

```text
FULLCOM/SILENTCOM -> PRE_NOCOM -> NOCOM
```

### 11.3 BusOff 调试

目标：确认 BusOff 触发、通知、快慢恢复都正常。

断点/观察：

- `CanIf_ControllerBusOff`
- `CanSM_ControllerBusOff`
- `CanSM_BusOffTransitions`
- `CanSM_BOR_MultipleCtrlsStop`
- `CanSM_BOR_CtrlsStart`
- `CanSM_BOR_CtrlsStart_SilentCom`
- `CanSM_GetBusoff_Substate`
- `ComM_BusSM_ModeIndication`
- `BswM_CanSM_CurrentState`
- `Dem_ReportErrorStatus` 或 DEM 相关封装

关键变量：

- `CanSM_BusOff_Indicated_ab[0]`
- `CanSM_currBOR_State_en[0]`
- `CanSM_BusOff_Cntr_au8[0]`
- `CanSM_BORMode_au8[0]`
- `CanSM_TimerConfig_ast[0].cntTick_u16`
- `CanSM_BOR_SilentCom_ab[0]`

期望行为：

```text
BusOff
  -> Tx 关闭 / SilentCom
  -> BOR L1 100ms
  -> 如果连续失败达到 10 次，进入 BOR L2 1s
  -> 恢复后等待 TxEnsured 1s
  -> 回到 FullCom 或按请求进入其他模式
```

## 12. 和 NM 文档的衔接

NM 文档里讲的是网络如何醒、如何睡；CanSM 文档讲的是本 ECU 的 CAN 通道如何真的打开/关闭。两者的衔接点是 ComM：

```text
应用/NM/诊断请求通信
  -> ComM 决策 Full/Silent/No
  -> CanSM 执行 CAN 硬件状态
  -> CanNm 在硬件可用后发收 NM 报文
```

所以排查“为什么 NM 不发”时，除了看 `Nm_NetworkRequest` / `CanNm_NetworkRequest`，还必须确认：

- CanSM 已经 FullCom。
- Controller 已 STARTED。
- Transceiver 已 Normal。
- CanIf PDU Mode 已 ONLINE。

否则 CanNm 就算状态机想发，底层也可能发不出去。

## 13. 关键变量和含义

CanSM 的变量可以按三层理解：第一层是“配置表”，决定这个网络有哪些 controller、transceiver、BusOff 时间；第二层是“请求和当前模式”，决定 CanSM 想去哪、现在在哪；第三层是“回调标志和定时器”，决定状态机能不能继续往下走。

| 变量/配置项 | 所在位置 | 直白解释 | 调试时怎么看 |
|---|---|---|---|
| `CanSM_ConfigSet[]` | `BasicSoftware/src/bsw/CanSM/CanSM_PBcfg.c` | CanSM 总配置入口。初始化时 CanSM 从这里拿网络配置、控制器映射、通道数量。 | 如果 CanSM 初始化后所有状态都不对，先确认 `CanSM_Init()` 用的是这套配置。 |
| `CanSM_Config` | `BasicSoftware/src/bsw/CanSM/CanSM_Cfg.h` | 宏定义，指向 `CanSM_ConfigSet[0]`。 | 多配置集工程里要注意当前激活的是哪一套；本工程就是第 0 套。 |
| `CanSM_Network_Config_acst[]` | `CanSM_PBcfg.c` | 每个 CanSM 网络的配置表。本工程只有 `Can_Network`。 | 这里能直接看到 L1/L2 恢复时间、BusOff DTC、Transceiver、ComM channel。 |
| `CanSM_NetworktoCtrl_Config_acu8[]` | `CanSM_PBcfg.c` | CAN Controller 到 CanSM Network 的映射。 | `CanSM_ControllerBusOff(ControllerId)` 进来后，就是靠它找到哪个网络 BusOff。 |
| `CanSM_Ctrl_Can_Network_Config_au8[]` | `CanSM_PBcfg.c` | 当前网络包含哪些 CAN Controller。 | FullCom/NoCom 切换时，CanSM 会遍历这里的 controller 去 start/stop。 |
| `CanSM_Network_pcst` | CanSM 内部全局指针 | 指向当前正在使用的网络配置表。 | 单步状态机时经常看到它，重点看 `ComM_channelId_uo`、`Trcv_hndle_u8`、`SizeofController_u8`。 |
| `CanSM_ConfigSet_pcst` | CanSM 内部全局指针 | 指向当前 CanSM 配置集。 | 如果它是空或没初始化，很多 API 会报未初始化或直接不动作。 |
| `CanSM_CurrNw_Mode_en[network]` | CanSM 运行时状态 | CanSM 当前网络状态，是看 CanSM 卡在哪一步的第一变量。 | 常见值包括 `CANSM_BSM_S_NOCOM`、`CANSM_BSM_S_PRE_FULLCOM`、`CANSM_BSM_S_FULLCOM`、`CANSM_BSM_S_PRE_NOCOM`、`CANSM_BSM_S_SILENTCOM`、`CANSM_BSM_S_SILENTCOM_BOR`。 |
| `CanSM_ReqComM_Mode_en[network]` | CanSM 运行时请求 | ComM 请求 CanSM 去哪个通信模式。 | ComM 请求 FullCom 后，它应变成 `COMM_FULL_COMMUNICATION`；释放后应变成 `COMM_NO_COMMUNICATION`。 |
| `CanSM_BusSMMode_au8[network]` | CanSM 对外模式缓存 | CanSM 准备通知给 ComM 的当前 BusSM 模式。 | `ComM_BusSM_ModeIndication()` 前后看它，能判断 CanSM 是否已经认为模式切换完成。 |
| `CanSM_PreFullCom_Substates_en[network]` | PreFullCom 子状态 | 从 NoCom 到 FullCom 时的细分步骤，比如 transceiver normal、controller stopped、controller started、PDU online。 | 卡在 PreFullCom 时重点看它，比只看 `CanSM_CurrNw_Mode_en` 更具体。 |
| `CanSM_PreNoCom_Substates_en[network]` | PreNoCom 子状态 | 从 FullCom/SilentCom 到 NoCom 的细分步骤，比如 controller stop、transceiver standby、PN 清标志。 | 休眠失败或下电前 CAN 还在线时重点看它。 |
| `CanSM_Ctrl_ModeInd_ab[network]` | Controller 模式确认标志 | CanIf/CanDrv 完成 controller 模式切换后，通过回调告诉 CanSM，CanSM 把它置位。 | 如果一直为 `FALSE`，状态机通常会等到重试超时，说明 controller mode indication 没回来。 |
| `CanSM_Trcv_ModeInd_ab[network]` | Transceiver 模式确认标志 | CanIf/CanTrcv 完成收发器模式切换后，通过回调告诉 CanSM。 | ECU 不在线但软件请求 FullCom 时，重点看 transceiver 是否真的进入 NORMAL。 |
| `CanSM_TimerConfig_ast[network].cntTick_u16` | CanSM 定时器计数 | CanSM 内部等待/重试/BOR 使用的 tick 计数。 | 本工程 MainFunction 10ms，所以计数 10 大约是 100ms，100 大约是 1s。 |
| `CanSM_TimerConfig_ast[network].stTimer` | CanSM 定时器状态 | 表示定时器取消、运行、到期等。 | 状态机卡住时看它是否一直未到期，或已经到期但没有触发下一步。 |
| `CanSM_Num_Unsuccessful_ModeReq_au8[network]` | 模式请求失败计数 | 请求 controller/trcv 模式失败或等不到确认时累加。 | 达到 `CanSMModeRequestRepetitionMax=5` 后，状态机会走错误处理或停在异常路径。 |
| `CanSM_ModeRepeatReq_Time_u16` | 生成/内部时间值 | 模式请求重试周期。本工程 ARXML 是 0.07s，按 10ms 主函数约等于 7 tick。 | 如果模式切换比预期慢，看这个时间和 MainFunction 周期是否匹配。 |
| `CanSM_currBOR_State_en[network]` | BusOff Recovery 当前状态 | BusOff 恢复状态机的核心变量。 | BusOff 后应进入检测/恢复相关状态，恢复完成后回到 `CANSM_S_NO_BUS_OFF` 或继续 FullCom。 |
| `CanSM_BusOff_Cntr_au8[network]` | BusOff 次数计数 | 用来判断当前处于 L1 快恢复还是 L2 慢恢复。 | 本工程 `CanSMBorCounterL1ToL2=10`，连续 BusOff 超过阈值后会转慢恢复。 |
| `CanSM_BusOff_Indicated_ab[network]` | BusOff 指示标志 | `CanSM_ControllerBusOff()` 收到 CanIf 通知后置位，主函数再处理。 | 触发 BusOff 后先看它是否置位；没置位说明 CanIf 到 CanSM 的回调链断了。 |
| `CanSM_BORMode_au8[network]` | BOR 使能状态 | 表示 BusOff Recovery 当前是否允许执行。 | 如果被应用或内部逻辑禁用，BusOff 后可能不会按预期恢复。 |
| `CanSM_BOR_SilentCom_ab[network]` | SilentCom 下 BOR 标志 | Silent Communication 场景下发生 BusOff 的特殊处理标志。 | 如果当前不是 FullCom 而是 SilentCom，BusOff 调试要看它。 |
| `CanSM_GetBusOffDelay_Value_u8[network]` | 可选 BusOff delay 值 | 用户自定义 BusOff 延迟接口相关变量。 | 本工程主要看配置里的 L1/L2；若启用用户 delay，再看这个值是否被应用更新。 |
| `BusOffEventId_uo` | `CanSM_NetworkConf_tst` 成员 | BusOff 关联的 Dem EventId。本工程指向 `DTC_0xC11001_Event`。 | BusOff 有恢复动作但没有 DTC，检查这个 EventId 和 Dem 配置。 |
| `Trcv_hndle_u8` | `CanSM_NetworkConf_tst` 成员 | 当前网络绑定的 transceiver handle。本工程是 `CanIfTrcvCfg_1145`。 | FullCom 不在线、休眠唤醒异常时，必须确认这个 handle 和 CanIf/CanTrcv 配置一致。 |
| `ComM_channelId_uo` | `CanSM_NetworkConf_tst` 成员 | CanSM 网络对应的 ComM channel。 | 如果 ComM 请求进错通道，CanSM 状态不会按预期变化。 |

最常用的调试判断顺序是：

1. 看 `CanSM_ReqComM_Mode_en[0]` 是否收到 ComM 的目标模式。
2. 看 `CanSM_CurrNw_Mode_en[0]` 当前停在哪个主状态。
3. 如果停在 PreFullCom/PreNoCom，再看对应子状态和 `CanSM_Ctrl_ModeInd_ab[0]`、`CanSM_Trcv_ModeInd_ab[0]`。
4. 如果是 BusOff，先看 `CanSM_BusOff_Indicated_ab[0]`，再看 `CanSM_currBOR_State_en[0]`、`CanSM_BusOff_Cntr_au8[0]`、`CanSM_TimerConfig_ast[0]`。
5. 最后确认 `ComM_BusSM_ModeIndication()`、`BswM_CanSM_CurrentState()`、`Dem_ReportErrorStatus()` 有没有被调用。

## 14. 总结

对初学者，CanSM 可以记成一句话：

```text
CanSM 是 ComM 和 CAN 硬件之间的状态执行器，同时负责 BusOff 恢复。
```

当前工程的 CanSM 主线：

```text
ComM 请求模式
  -> CanSM_RequestComMode
  -> CanSM_MainFunction 10ms 状态机
  -> CanIf_SetTrcvMode / CanIf_SetControllerMode
  -> ComM_BusSM_ModeIndication / BswM_CanSM_CurrentState
```

当前工程 BusOff 主线：

```text
CanIf_ControllerBusOff
  -> CanSM_ControllerBusOff
  -> CanSM_BusOffTransitions
  -> L1 100ms / L2 1s 快慢恢复
  -> TxEnsured 1s 后确认恢复
```

最常见的工程风险不是“不知道 CanSM 有哪些状态”，而是只看软件状态，不看 Transceiver/Controller/CanIf indication 和物理层条件。调 CanSM 时，软件状态机、硬件模式、总线波形必须一起看。
