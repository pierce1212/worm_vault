# 硬件原理

![](assets/2026.06.08%20RSM8900开发/file-20260608095406378.png)
这张图是 `RS4TC8900QCE` RTC 芯片的外围电路。按功能块看最清楚。

**1. 供电**

左边 `V_5V_RTC` 给芯片供电：

```text
V_5V_RTC -> VBAT pin3
V_5V_RTC -> VDD  pin2
GND      -> GND  pin9
```

`C337`、`C338` 都是 `100nF` 去耦电容，作用是给 RTC 电源滤波，抑制电源毛刺。

这里 `VBAT` 和 `VDD` 接到同一路 `V_5V_RTC`，说明这版硬件没有单独备用电池，RTC 不是真正靠纽扣电池保持时间。图上标的 `0.7~1.5uA@stb` 是 RTC 备用/待机状态电流量级。

**2. I2C 通信**

右边是 I2C：

```text
SCL pin5 -> SCL_RTC
SDA pin7 -> SDA_RTC
```

`R322` 和 `R323` 是 I2C 上拉电阻：

```text
R322 1K：SCL 上拉到 VCC_5V_2
R323 1K：SDA 上拉到 VCC_5V_2
```

I2C 是开漏总线，只能拉低，不能主动输出高电平，所以必须有上拉电阻。

`TP330`、`TP332` 是测试点，方便用示波器/逻辑分析仪看 `SCL_RTC`、`SDA_RTC` 波形。

和代码对应：

```c
#define RSM_IIC_SLAVE_ADDRESS 0x32u
```

MCU 通过这两根线访问地址 `0x32` 的 RTC。

**3. 中断输出**

左下角：

```text
RTC_INT -> INT pin10
```

旁边注释：

```text
定时中断输出
OD门
HiZ to low
```

意思是 `INT` 是开漏输出：

```text
无中断：高阻态 Hi-Z
有中断：拉低 Low
```

所以这个信号需要上拉，可能是 MCU 内部上拉，也可能在别的页面有外部上拉。RTC 的闹钟、周期定时、更新事件都可以通过这个脚通知 MCU。

代码里的这些位和它有关：

```c
RSM8900_BTC_CTRL_AIE   // Alarm interrupt enable
RSM8900_BTC_CTRL_TIE   // Timer interrupt enable
RSM8900_BTC_CTRL_UIE   // Update interrupt enable

RSM8900_BTC_FLAG_AF    // Alarm flag
RSM8900_BTC_FLAG_TF    // Timer flag
RSM8900_BTC_FLAG_UF    // Update flag
```

**4. FOE / FOUT**

```text
FOE pin1
FOUT pin4
```

`FOE` 是频率输出使能脚。图中 `FOE` 通过 `R328 1K` 下拉到 GND，所以默认关闭频率输出。

`FOUT` 打叉，表示没有连接。因此当前硬件没有使用 RTC 的频率输出功能。

**5. T1/T2**

```text
T1 pin6
T2 pin8
```

都打叉，表示未连接。一般是测试脚或未使用功能脚。

**6. 这张图的整体含义**

一句话讲：

```text
RS4TC8900QCE 由 5V_RTC 供电，
通过 SCL_RTC/SDA_RTC 和 MCU 做 I2C 通信，
RTC_INT 用开漏方式输出中断，
FOUT/T1/T2 当前未使用。
```

对你当前 `RSM_IIC.c` 来说，最关键的是右边这部分：

```text
SCL_RTC / SDA_RTC
+ R322/R323 上拉
+ RTC 地址 0x32
```

也就是软件里的 `I2c_SyncTransmit()` 最终就是在这两根线上产生 I2C 波形去读写 RTC 寄存器。


## RTC唤醒
![](assets/2026.06.08%20RSM8900开发/file-20260608102136047.png)

这张图是 **RTC 唤醒电路**。作用是：当 RTC 芯片的 `RTC_INT` 产生中断时，通过三极管电路把这个低电平中断转换成一个唤醒信号 `O_S_INH_RTC`，去唤醒后级电源/系统。

**整体信号链**

```text
RTC_INT
  -> Q303/Q305 整形反相
  -> Q304 拉低驱动
  -> Q302 高边开关导通
  -> VBAT 输出到 O_S_INH_RTC
```

也就是：

```text
RTC 中断有效
  -> O_S_INH_RTC 被拉高
  -> 系统被 RTC 唤醒
```

**1. RTC_INT 是低有效开漏信号**

左下角：

```text
RTC_INT
```

来自 RTC 芯片的中断脚。前面那张图已经标了：

```text
OD 门
HiZ to low
```

意思是：

```text
无中断：RTC_INT 高阻态
有中断：RTC_INT 拉低
```

所以这里加了 `R325 100K` 上拉到 `V_5V_RTC`：

```text
R325：RTC_INT 上拉电阻
```

**无中断时，`RTC_INT` 被拉成高电平；有中断时，RTC 芯片把它拉低。**

**2. Q303 + Q305 把 RTC_INT 转成中间控制信号**

左中间：

```text
Q303：PNP，BC807
Q305：NPN，BC817
```

它们组成一个类似反相/整形电路。

当 **无中断** 时：

```text
RTC_INT = 高
Q303 关断
Q305 导通
TP331 被拉低
```

当 **有中断** 时：

```text
RTC_INT = 低
Q303 导通
Q305 关断
TP331 被拉高
```

所以 `TP331` 是一个和 `RTC_INT` 相反的信号：

```text
RTC_INT 高 -> TP331 低
RTC_INT 低 -> TP331 高
```

`R318 68K` 是 Q303 的上拉/限流电阻。

**3. Q304 是低边驱动**

中间：

```text
Q304：NPN，BC817
R326：10K，基极电阻
R329：22K，基极下拉
```

`TP331` 通过 `R326` 控制 Q304。

当 RTC 中断有效时：

```text
TP331 高
Q304 导通
Q304 集电极被拉低
```

也就是 Q304 会把 `TP326` 附近节点拉到 GND。

**4. Q302 是高边开关**

右上：

```text
Q302：PNP，BC807
R317：47K，上拉到 VBAT
R319：1K，基极限流
```

Q302 的作用是把 `VBAT` 接到右侧输出。

默认情况下：

```text
R317 把 Q302 基极拉到 VBAT
Q302 关断
O_S_INH_RTC 没有被 RTC 拉高
```

当 RTC 中断有效时：

```text
Q304 导通
Q302 基极被拉低
Q302 导通
VBAT -> Q302 -> TP328 -> D301 -> O_S_INH_RTC
```

所以 `O_S_INH_RTC` 被拉高，形成唤醒信号。

**5. D301 是隔离二极管**

```text
D301：BAS100CS
```

它把 Q302 输出和 `O_S_INH_RTC` 隔离开。

作用通常是：

```text
允许 RTC 唤醒信号送到 O_S_INH_RTC
防止 O_S_INH_RTC 反向灌回 RTC 唤醒电路
```

也方便多个唤醒源做“二极管或”连接。

**6. ADC_RTC 是检测电压**

右下角：

```text
R327 51K
R330 10K
C339 10nF
ADC_RTC
```

这是一个电阻分压 + 滤波。

分压比例是：

```text
10K / (51K + 10K) = 0.164
```

图上红字 `0.164` 就是这个比例。

也就是说：

```text
ADC_RTC = TP328 电压 * 0.164
```

如果 `TP328 = 5V`：

```text
ADC_RTC ≈ 0.82V
```

如果 `TP328 = 12V`：

```text
ADC_RTC ≈ 1.97V
```

MCU 可以通过 `ADC_RTC` 判断 RTC 唤醒输出是否存在、VBAT 是否正常，或者记录唤醒来源状态。`C339` 用来滤波，防止 ADC 读数抖动。

**7. 工作状态总结**

无 RTC 中断：

```text
RTC_INT 高
Q305 导通
TP331 低
Q304 关断
Q302 关断
O_S_INH_RTC 不被拉高
ADC_RTC 低
```

有 RTC 中断：

```text
RTC_INT 低
Q303 导通
TP331 高
Q304 导通
Q302 基极被拉低
Q302 导通
VBAT 输出到 O_S_INH_RTC
ADC_RTC 检测到分压电压
```

一句话：这张图就是把 RTC 的 **低有效开漏中断** 转换成一个由 `VBAT` 供电的 **高有效唤醒信号**，同时用 `ADC_RTC` 给 MCU 做电压检测。

### 为什么RTC_INT低有效和V5v_RTC同时发生表现为低？

**因为这里的 `V_5V_RTC` 不是“强推高”，而是通过 `R325 100K` 弱上拉；`RTC_INT` 低有效时，是 RTC 芯片内部开漏管把这根线 强拉到 GND。**

等效电路可以看成：

```text
V_5V_RTC
   |
  R325 100K
   |
RTC_INT 节点
   |
RTC 内部开漏 NMOS 导通
   |
  GND
```

当没有中断时：

```text
RTC_INT 输出高阻 Hi-Z
R325 把 RTC_INT 拉到高电平
RTC_INT = High
```

当 RTC 中断有效时：

```text
RTC_INT 内部开漏管导通
RTC_INT 被拉到 GND
RTC_INT = Low
```

**这时虽然上面还有 `V_5V_RTC`，但它要经过 `100K` 电阻，驱动能力很弱。**拉低路径是晶体管直接到地，阻抗很低，所以 低电平赢。**

电流大概是：

```text
I = 5V / 100K = 50uA
```

也就是：

```text
V_5V_RTC -> R325 -> RTC_INT 开漏管 -> GND
```

只流很小的电流，不会短路。

所以结论是：

```text
RTC_INT 低有效 + V_5V_RTC 上拉同时存在时
= 开漏输出拉低覆盖弱上拉
= 节点表现为低电平
```

**这就是开漏信号的典型用法：上拉负责“无动作时为高”，器件动作时只负责“拉低”。**

### 为什么TP331可以输出一个跟RTC_INT相反的信号？

因为 `TP331` 这个点是由 **一上一下两只三极管共同控制** 的，而这两只三极管都受 `RTC_INT` 控制，但导通条件正好相反。

可以把左边简化成这样：

```text
             V_5V_RTC
                |
              R318
                |
              Q303 PNP
                |
              TP331
                |
              Q305 NPN
                |
               GND

Q303 base <- RTC_INT
Q305 base <- RTC_INT
```

**Q303 是 PNP 管**

PNP 的特点是：

```text
基极比发射极低约 0.7V -> 导通
基极接近发射极电压 -> 关断
```

所以：

```text
RTC_INT = 低 -> Q303 导通 -> TP331 被往上拉
RTC_INT = 高 -> Q303 关断
```

**Q305 是 NPN 管**

NPN 的特点是：

```text
基极比发射极高约 0.7V -> 导通
基极为低 -> 关断
```

Q305 的发射极接地，所以：

```text
RTC_INT = 高 -> Q305 导通 -> TP331 被拉到 GND
RTC_INT = 低 -> Q305 关断
```

所以两个状态合起来就是：

| RTC_INT | Q303 PNP | Q305 NPN | TP331 |
|---|---|---|---|
| 高，无中断 | 关断 | 导通 | 低 |
| 低，有中断 | 导通 | 关断 | 高 |

所以 `TP331` 就天然变成了 `RTC_INT` 的反相信号：

```text
RTC_INT 高 -> TP331 低
RTC_INT 低 -> TP331 高
```

关键点是：`RTC_INT` 同时控制了一个“上拉开关”Q303 和一个“下拉开关”Q305。高电平时下拉开关打开，低电平时上拉开关打开，于是 TP331 就反相了。


### 集电极和发射极为什么这么命名？
这个问题问得很好，因为“三极管的电流方向”和“电子运动方向”很容易把人绕晕。

对 **NPN 三极管**来说：

```text
N  P  N
E  B  C
发射极  基极  集电极
```

名字来自它在正常放大区里的物理动作：

```text
发射极 E：发射载流子
集电极 C：收集载流子
基极 B：控制有多少载流子通过
```

对 NPN 来说，主要载流子是 **电子**。

正常工作时：

```text
BE 结正偏
BC 结反偏
```

也就是：

```text
Base 比 Emitter 高大约 0.7V
Collector 比 Base 更高
```

这时发生的事情是：

```text
Emitter 向 Base 发射大量电子
电子穿过很薄的 Base
Collector 把这些电子收集走
```

所以叫：

```text
Emitter = 发射极
Collector = 集电极
```

注意，这里说的是 **电子流**：

```text
电子实际运动方向：Emitter -> Base -> Collector
```

但我们平时画电路用的是 **传统电流方向**，传统电流方向和电子方向相反：

```text
传统电流方向：Collector -> Emitter
电子运动方向：Emitter -> Collector
```

这就是最容易迷惑的地方：  
你看到电流好像从 `C` 流到 `E`，于是会疑惑“为什么不叫 C 发射、E 收集？”  
答案是：==**名字是按 载流子物理注入/收集过程 起的，不是按传统电流箭头起的。**==

还有一个关键点：`E` 和 `C` 不是随便对称命名的，它们结构不同。

```text
Emitter：掺杂很重，负责高效率注入电子
Base：很薄、掺杂轻，负责控制
Collector：面积更大、耐压更高，负责收集电子和承受功耗
```

所以虽然 NPN 看起来是 `N-P-N` 对称结构，但实际制造上不对称。把 `C` 和 `E` 反过来用也可能导通，但放大能力很差，耐压也不对。

一句话记忆：

```text
NPN 中，发射极发射电子，集电极收集电子；
传统电流方向虽然是 C -> E，但电子实际是 E -> C。
```

回到你那个 RTC 唤醒电路里，像 `Q305` 这种 NPN，发射极接地，基极被拉高时导通，集电极就能把上面的节点拉低。它的“集电极”之所以叫集电极，是因为在器件内部它负责收集从发射极注入并穿过基区的电子。

### Q304是NPN型的，也就是基极要比发射极高高0.7V才是导通，但是发射极连接这VBAT，怎么可能比VBAT高呢？
你这里把 **Q304** 和右上角 **Q302** 的连接看混了。

在这张图里：

```text
Q304 是 NPN，BC817
Q304 的发射极 E 接 GND
Q304 的集电极 C 接到 Q302 的基极控制节点
Q304 的基极 B 通过 R326 接 TP331
```

也就是 Q304 实际是这样：

```text
        到 Q302 基极
            |
            C
          Q304  NPN
            E
            |
           GND

TP331 -- R326 -- B
```

所以 Q304 导通条件是：

```text
Vb > Ve + 0.7V
```

因为 Q304 的 `Ve = GND = 0V`，所以只要：

```text
TP331 ≈ 0.7V 以上
```

Q304 就能导通。

它不是发射极接 `VBAT`。  
**接 VBAT 的是 Q302，不是 Q304。**

右上角 Q302 是 PNP，高边开关：

```text
Q302 发射极 E 接 VBAT
Q302 集电极 C 接 TP328 / O_S_INH_RTC
Q302 基极 B 通过 R317 拉到 VBAT
```

Q304 的作用是：当它导通时，把 Q302 的基极往地拉。

```text
Q304 导通
  -> Q302 基极被拉低
  -> Q302 的发射极在 VBAT
  -> Q302 的 Veb 形成约 0.7V
  -> Q302 PNP 导通
  -> VBAT 输出到 O_S_INH_RTC
```

所以整个过程是：

```text
TP331 高
  -> Q304 NPN 导通
  -> Q304 集电极拉低
  -> Q302 PNP 基极被拉低
  -> Q302 导通
  -> O_S_INH_RTC 被拉高
```

如果 Q304 的发射极真的接 VBAT，那你说得对：它的基极必须比 VBAT 高约 0.7V 才能导通，电路就不合理了。但图里不是这样，**Q304 发射极在下面接 GND，Q302 发射极才接 VBAT**。

### 也就是说NPN和PNP的电位是不一样的？
对，可以这么理解：**NPN 和 PNP 的发射极、集电极在常用电路里的电位方向是相反的**。

但更准确地说，不是“名字方向不同”，而是它们正常导通时的偏置方向不同。

**NPN**

NPN 常见用法是低边开关：

```text
VCC
 |
负载
 |
 C
 NPN
 E
 |
GND
```

导通条件：

```text
B 比 E 高约 0.7V
```

导通后传统电流方向：

```text
C -> E
```

所以 NPN 通常：

```text
C 接较高电位
E 接较低电位
```

**PNP**

PNP 常见用法是高边开关：

```text
VCC
 |
 E
 PNP
 C
 |
负载
 |
GND
```

导通条件：

```text
B 比 E 低约 0.7V
```

导通后传统电流方向：

```text
E -> C
```

所以 PNP 通常：

```text
E 接较高电位
C 接较低电位
```

放到一起记：

```text
NPN：C 高，E 低；B 比 E 高 0.7V 导通
PNP：E 高，C 低；B 比 E 低 0.7V 导通
```

再对应你图里的电路：

```text
Q304 是 NPN：
E 接 GND
C 接 Q302 基极节点
B 被 TP331 拉高时导通
作用：把 Q302 基极拉低
```

```text
Q302 是 PNP：
E 接 VBAT
C 接 O_S_INH_RTC 输出
B 被 Q304 拉低时导通
作用：把 VBAT 送到 O_S_INH_RTC
```

一句话记忆：
**==NPN 像“低边拉地开关”，基极给高就开；**
**PNP 像“高边供电开关”，基极拉低就开。==**




# I2C总线协议
I2C 是一种很常见的板级低速串行总线，用两根线连接 MCU 和外设，比如 EEPROM、RTC、温度传感器、PMIC、IO 扩展芯片等。
**核心概念**
I2C 只有两根信号线：

- SCL：时钟线，由主机产生时钟。
- SDA：数据线，主机和从机共用。

它是主从式总线。通常 MCU 是 Master，外设是 Slave。主机发起通信，决定访问哪个从机、读还是写、传多少字节。

I2C 线是开漏/开集电极结构，所以总线需要上拉电阻。设备只能主动拉低，不能主动输出高电平；高电平靠上拉电阻恢复。这也是为什么 I2C 可以多设备共用同一组 SCL/SDA。
## 为什么要在 IIC 信号线上加上拉电阻。？

主要原因就是 IIC 芯片的 SDA 和 SCL 的引脚是开漏输出，就是只有一个 NMOS 管，不像推挽输出有两个 MOS 管。
![](assets/2026.06.08%20RSM8900开发/file-20260608100109758.png)

当芯片 SDA 和 SCL 的引脚输出 MOS 管导通，IIC 信号线电平为低电平。
![](assets/2026.06.08%20RSM8900开发/file-20260608100221606.png)
当芯片 SDA 和 SCL 的引脚输出 MOS 管关闭，如果没有上拉电阻，IIC 信号线是处于一个高阻状态，电平是未知的，开漏输出是没有高电平的输出能力的。
![](assets/2026.06.08%20RSM8900开发/file-20260608100243819.png)
所以加上上拉电阻后，当芯片 SDA 和 SCL 的引脚输出 MOS 管关闭，IIC 信号线上的电平就是一个确切的高电平。
![](assets/2026.06.08%20RSM8900开发/file-20260608100456243.png)

## **一次写操作**

写 RTC 寄存器通常是：

`START Slave Address + Write Register Address Data0 Data1 ... STOP`
你当前代码里的 RSM_IIC_Send_Data() 就是这么做的：先把寄存器地址 cmd 放到发送缓冲区第一个字节，后面跟实际数据，然后一次性发出去。
## **一次读操作**

读寄存器一般分两步：

`START Slave Address + Write Register Address REPEATED START Slave Address + Read Data0 Data1 ... STOP`

为什么读之前还要先写？因为要先告诉从机“我要从哪个寄存器开始读”。

## **ACK/NACK**

**I2C 每发送 8 位数据后，接收方要回一个 ACK，表示“我收到了”。如果没有 ACK，说明可能**：

- 地址不对。
- 外设没上电。
- 总线被拉死。
- 时序/速率不匹配。
- 从机内部忙。

所以调 I2C 时，逻辑分析仪里最关键的就是看地址后面有没有 ACK。

### ACK/NACK是由硬件完成的
你在 `CDD_I2c.c` 这一层确实看不到 ACK 的逐位处理，因为 ACK/NACK 是 **LPI2C 硬件和底层 IP driver** 处理的，不是在 `I2c_SyncTransmit()` 里手写出来的。

代码分层是这样的：

```text
RSM_IIC.c
  -> I2c_SyncTransmit()

CDD_I2c.c
  -> 检查参数/通道/busy状态
  -> I2c_Ipw_SyncTransmit()

I2c_Ipw.c
  -> Lpi2c_Ip_MasterSendDataBlocking()
  -> Lpi2c_Ip_MasterReceiveDataBlocking()

LPI2C IP driver / 硬件寄存器
  -> 发送地址、数据
  -> 自动采样 ACK/NACK
  -> 发现 NACK/超时/仲裁错误后返回错误状态
```

你可以看 [I2c_Ipw.c](e:/github/ECAS_RTA_S32K324GHS_Heating/BasicSoftware/integration/mcal/src/modules/I2c/src/I2c_Ipw.c:397)：

```c
Lpi2c_Ip_MasterSendDataBlocking(...)
```

以及 [I2c_Ipw.c](e:/github/ECAS_RTA_S32K324GHS_Heating/BasicSoftware/integration/mcal/src/modules/I2c/src/I2c_Ipw.c:401)：

```c
Lpi2c_Ip_MasterReceiveDataBlocking(...)
```

ACK 处理就在这些更底层的 LPI2C IP 函数/硬件状态机里。

所以你看到的 `I2c_SyncTransmit()` 只负责：

```text
准备这次是 SEND 还是 RECEIVE
检查通道是否忙
调用底层 blocking 传输
拿到底层返回值
```

它不会自己写：

```text
发送 8 bit
等待第 9 个 ACK bit
```

因为这个由硬件完成。

**==在你当前 RTC 代码里，NACK 检测依赖这个返回链路==**：

```text
LPI2C硬件检测到NACK
  -> Lpi2c_Ip_MasterSendDataBlocking 返回非 E_OK
  -> I2c_Ipw_SyncTransmit 转成 E_NOT_OK
  -> I2c_SyncTransmit 返回 E_NOT_OK
  -> RSM_IIC_Send_Data / Receive_Data 返回 E_NOT_OK
```

另外，你这里有一个配置字段：

```c
RequestData.ExpectNack = FALSE;
```

在 [RSM_IIC.c](e:/github/ECAS_RTA_S32K324GHS_Heating/CDD/RTC/Src/MainFunction/RSM8900/RSM_IIC.c:306) 这些地方设置了它。意思是：**这次传输不期望 NACK**。如果底层检测到 NACK，就应该作为失败处理。

一句话：**ACK 不在 `CDD_I2c.c` 里显式写出来，它由 LPI2C 硬件/IP 层自动处理，应用层通过 `I2c_SyncTransmit()` 返回值知道是否 ACK 成功。**

## 对于I2C协议，时钟信号是谁提供的？一个时钟周期代表一个bit？时钟是一直有的吗？

I2C 的时钟信号 **SCL 是主机提供的**。

在你这个项目里，S32K324 作为 I2C Master，RTC 芯片 RSM8900 作为 I2C Slave，所以：

```text
S32K324 / LPI2C Master  -> 提供 SCL 时钟
RSM8900 RTC Slave       -> 跟随 SCL 收发 SDA 数据
```

**一个时钟周期是不是代表一个 bit？**

基本可以这么理解：**SCL 的一个完整周期传输 SDA 上的 1 bit 数据**。

典型过程：

```text
SCL:  _|‾|_|‾|_|‾|_|‾|_
SDA:   D0  D1  D2  D3
```

I2C 规定 SDA 数据在 SCL 高电平期间保持稳定，接收方通常在 SCL 高电平期间采样 SDA。

一个字节传输不是 8 个周期就结束，而是：

```text
8 bit 数据 + 1 bit ACK/NACK = 9 个 SCL 周期
```

例如主机写一个字节给 RTC：

```text
SCL 周期 1~8:  发送 8 bit 数据
SCL 周期 9:    从机拉低 SDA 表示 ACK
```

**时钟是一直有的吗？**

不是。I2C 的 SCL **不是一直输出的连续时钟**。

它只在一次 I2C 通信期间由 Master 产生，比如：

```text
Start -> 地址 -> ACK -> 寄存器地址 -> ACK -> 数据 -> ACK -> Stop
```

通信结束后，SCL 和 SDA 通常都会回到高电平空闲状态：

```text
Idle: SCL = High, SDA = High
```

**所以它不是像 SPI 某些配置那样可以一直有时钟，也不是 MCU 系统时钟那种持续跑的时钟。I2C 是“需要通信时才打时钟”。**

**补充一个关键点：Clock Stretching**

虽然 SCL 由 Master 提供，但 I2C 允许 Slave 在某些情况下把 SCL 拉低，要求 Master 等一下，这叫 clock stretching。

也就是说：

```text
Master 想释放 SCL 为高
Slave 如果还没准备好，可以继续拉低 SCL
Master 检测到 SCL 还低，就等待
```

不过很多简单 RTC 芯片不一定会用 clock stretching，具体看手册。

一句话总结：

```text
I2C 的 SCL 由主机产生；
正常情况下 1 个 SCL 周期传输 1 bit；
每个字节后还有 1 个 ACK/NACK 周期；
SCL 只在通信期间出现，空闲时保持高电平。
```
## 读是需要先写再读，写也是两次传输吗？

不一样。

**读寄存器：通常是“两段传输”**

因为主机要先告诉 RTC：“我要从哪个寄存器开始读”，所以先写寄存器地址，再读数据：

```text
START
SlaveAddr + W
RegisterAddr
REPEATED START
SlaveAddr + R
Read Data
NACK/ACK
STOP
```

对应你代码里的 [RSM_IIC_Receive_Data](e:/github/ECAS_RTA_S32K324GHS_Heating/CDD/RTC/Src/MainFunction/RSM8900/RSM_IIC.c:291)：

```c
// 1. 先发送寄存器地址 cmd
I2c_SyncTransmit(0U, &RequestData);

// 2. 再读取数据
I2c_SyncTransmit(0U, &RequestData);
```

所以读确实是先写再读。

**写寄存器：通常是“一段传输”**

写的时候，主机可以在一次写事务里同时发：

```text
寄存器地址 + 数据
```

总线形式：

```text
START
SlaveAddr + W
RegisterAddr
Data0
Data1
...
STOP
```

对应你代码里的 [RSM_IIC_Send_Data](e:/github/ECAS_RTA_S32K324GHS_Heating/CDD/RTC/Src/MainFunction/RSM8900/RSM_IIC.c:364)：

```c
RSM_tx_data[0] = cmd;        // 寄存器地址
RSM_tx_data[1] = pData[0];   // 数据
...
I2c_SyncTransmit(0U, &RequestData);
```

这里只调用了一次 `I2c_SyncTransmit()`，所以写不是两次传输，而是一次传输，只是这一次传输的 buffer 里包含：

```text
RegisterAddr + Data
```

总结：

```text
读寄存器 = 写寄存器地址 + 读数据        -> 两次 I2C request，中间 Repeated Start
写寄存器 = 写寄存器地址和数据一起发    -> 一次 I2C request
```

## RSM8900 RTC Driver说明

本文说明当前工程中 `CDD/RTC/Src/MainFunction/RSM8900` 下 RTC 驱动的函数功能、运行逻辑、调用关系和寄存器设置。
相关文件：
- `RSM8900.c` / `RSM8900.h`：RTC功能层，提供初始化、时间读写、闹钟、周期唤醒、寄存器读写接口。
- `RSM_IIC.c` / `RSM_IIC.h`：I2C传输层，封装对RS4TC8900Q的寄存器读写。
### 1. 总体结构

驱动分为两层：

```text
应用/上层模块
    |
    | 调用 RSM8900_Init / RSM8900_SetTime / RSM8900_GetTime
    |      RSM8900_SetAlarm / RSM8900_GetAlarm
    |      RSM8900_SetCycWake / RSM8900_SetCycWakeMinutes
    v
RSM8900 RTC功能层
    |
    | 调用 RSM8900_ReadReg / RSM8900_WriteReg
    |      RSM8900_ReadOneReg / RSM8900_WriteOneReg
    v
RSM_IIC I2C传输层
    |
    | 调用 I2c_SyncTransmit
    v
CDD_I2c驱动 / LPI2C硬件
```

I2C设备地址在 `RSM_IIC.c` 中定义：

```c
#define RSM_IIC_SLAVE_ADDRESS 0x32u
```

寄存器地址在 `RSM8900.h` 中定义。时间日历寄存器有两组地址映射：

- `0x00~0x06`：Basic Time and Calendar区域，当前驱动主要使用这一组。
- `0x10~0x16`：Extension区域中的镜像时间寄存器，当前只使用了扩展区的 `TEMP/BACKUP` 等寄存器地址。

### 2. 关键寄存器

#### 2.1 时间日历寄存器

| 宏                   |     地址 | 功能            | 当前代码使用                 |
| ------------------- | -----: | ------------- | ---------------------- |
| `RSM8900_BTC_SEC`   | `0x00` | 秒，BCD格式       | `SetTime/GetTime/Init` |
| `RSM8900_BTC_MIN`   | `0x01` | 分，BCD格式       | `SetTime/GetTime/Init` |
| `RSM8900_BTC_HOUR`  | `0x02` | 时，BCD格式       | `SetTime/GetTime/Init` |
| `RSM8900_BTC_WEEK`  | `0x03` | 星期，单bit有效     | `SetTime/GetTime/Init` |
| `RSM8900_BTC_DAY`   | `0x04` | 日期，BCD格式      | `SetTime/GetTime/Init` |
| `RSM8900_BTC_MONTH` | `0x05` | 月，BCD格式       | `SetTime/GetTime/Init` |
| `RSM8900_BTC_YEAR`  | `0x06` | 年，BCD格式，00~99 | `SetTime/GetTime/Init` |

时间数据以BCD格式存放，例如十进制 `23` 写入寄存器为 `0x23`。星期寄存器使用单bit表示：

```text
bit0 Sunday
bit1 Monday
bit2 Tuesday
bit3 Wednesday
bit4 Thursday
bit5 Friday
bit6 Saturday
```

当前代码的 `pTime->week` 使用 `0~6`，写寄存器时执行：

```c
date[RSM8900_BTC_WEEK] = 1 << (pTime->week);
```

#### 2.2 闹钟寄存器

| 宏                               |     地址 | 功能      |
| ------------------------------- | -----: | ------- |
| `RSM8900_BTC_ALARM_MIN`         | `0x08` | 闹钟分钟    |
| `RSM8900_BTC_ALARM_HOUR`        | `0x09` | 闹钟小时    |
| `RSM8900_BTC_ALARM_WEEK_OR_DAY` | `0x0A` | 闹钟星期或日期 |

闹钟的第三个字段由 `RSM8900_BTC_EXT.WADA` 决定：

- `WADA = 0`：按星期匹配，写入 `1 << al_week`。
- `WADA = 1`：按日期匹配，写入 `al_day`。

#### 2.3 Timer定时唤醒寄存器

| 宏 | 地址 | 功能 |
|---|---:|---|
| `RSM8900_BTC_TIMER_CNT_0` | `0x0B` | Timer计数低8位 |
| `RSM8900_BTC_TIMER_CNT_1` | `0x0C` | Timer计数高4位 |
| `RSM8900_BTC_EXT` | `0x0D` | Timer时钟源/使能等扩展控制 |
| `RSM8900_BTC_FLAG` | `0x0E` | 中断/状态标志 |
| `RSM8900_BTC_CTRL` | `0x0F` | 中断使能/控制 |

当前定时唤醒使用分钟为单位，默认 `120min`：

```c
#define RSM8900_CYCWAKE_DEFAULT_MIN 120u
#define RSM8900_TIMER_COUNTER_MAX   4095u
```

Timer计数器是12位：

```c
timer_cnt[0] = (uint8)(wakeup_min & 0xFFu);
timer_cnt[1] = (uint8)((wakeup_min >> 8u) & 0x0Fu);
```

#### 2.4 EXT寄存器 `0x0D`

|  bit | 宏                       | 当前用途           |
| ---: | ----------------------- | -------------- |
| bit0 | `RSM8900_BTC_EXT_TSEL0` | Timer源选择       |
| bit1 | `RSM8900_BTC_EXT_TSEL1` | Timer源选择       |
| bit2 | `RSM8900_BTC_EXT_FSEL0` | 频率输出选择，当前未主动配置 |
| bit3 | `RSM8900_BTC_EXT_FSEL1` | 频率输出选择，当前未主动配置 |
| bit4 | `RSM8900_BTC_EXT_TE`    | Timer Enable   |
| bit5 | `RSM8900_BTC_EXT_USEL`  | Update Timer选择 |
| bit6 | `RSM8900_BTC_EXT_WADA`  | 闹钟按星期/日期选择     |
| bit7 | `RSM8900_BTC_EXT_TEST`  | 测试位，当前未主动配置    |

`RSM8900_SetCycWakeMinutes()` 中的设置逻辑：

```c
extreg &= ~(TSEL0 | TSEL1 | TE | USEL);
extreg |=  (TSEL0 | TSEL1 | TE | USEL);
```

也就是：

- `TSEL0 = 1`
- `TSEL1 = 1`
- `TE = 1`
- `USEL = 1`

该配置沿用旧项目中已实现的周期唤醒逻辑，用于分钟级Timer唤醒。

#### 2.5 FLAG寄存器 `0x0E`

|  bit | 宏                       | 含义          | 当前处理                      |
| ---: | ----------------------- | ----------- | ------------------------- |
| bit0 | `RSM8900_BTC_FLAG_VDET` | 电压检测标志      | Init中检测，置位则清标志            |
| bit1 | `RSM8900_BTC_FLAG_VLF`  | 低电压/时钟有效性标志 | Init中检测，置位则清标志            |
| bit3 | `RSM8900_BTC_FLAG_AF`   | Alarm Flag  | SetAlarm清除，GetAlarm判断     |
| bit4 | `RSM8900_BTC_FLAG_TF`   | Timer Flag  | Init判断是否定时唤醒，SetCycWake清除 |
| bit5 | `RSM8900_BTC_FLAG_UF`   | Update Flag | Init中检测，置位则清标志            |
|      |                         |             |                           |

`RSM8900_Init()` 中如果检测到 `TF = 1`，会设置：

```c
RSM8900_TimerWakeup = 1u;
```

上层可以通过该变量判断本次启动是否来自RTC Timer唤醒。

#### 2.6 CTRL寄存器 `0x0F`

| bit | 宏 | 含义 | 当前处理 |
|---:|---|---|---|
| bit0 | `RSM8900_BTC_CTRL_RESET` | 软件复位 | 当前未主动使用 |
| bit3 | `RSM8900_BTC_CTRL_AIE` | Alarm Interrupt Enable | SetAlarm置位，SetCycWake清除 |
| bit4 | `RSM8900_BTC_CTRL_TIE` | Timer Interrupt Enable | SetCycWake置位 |
| bit5 | `RSM8900_BTC_CTRL_UIE` | Update Interrupt Enable | SetAlarm置位，SetCycWake清除 |
| bit6 | `RSM8900_BTC_CTRL_CSEL0` | 时钟选择 | Init清CTRL时写入 |
| bit7 | `RSM8900_BTC_CTRL_CSEL1` | 时钟选择 | 当前未主动置位 |

### 3. 函数说明

#### 3.1 `RSM8900_Init(void)`

功能：

- 写入默认时间。
- 读取 `RSM8900_BTC_FLAG` 判断状态标志。
- 如果 `VDET/VLF/AF/TF/UF` 任一置位，则清 `CTRL/EXT/FLAG`。
- 如果 `TF` 置位，则认为本次启动来自RTC Timer唤醒，设置 `RSM8900_TimerWakeup = 1u`。
- 写 `RSM8900_EXT_BACKUP = 0x08`，关闭备用电池供电相关电压检测功能。

调用关系：

```text
RSM8900_Init
  -> RSM8900_WriteReg(RSM8900_BTC_SEC, default_time, 7)
       -> RSM_IIC_Send_Data
  -> RSM8900_ReadOneReg(RSM8900_BTC_FLAG, &data)
       -> RSM_IIC_Receive_Data
  -> RSM8900_WriteOneReg(RSM8900_BTC_CTRL, RSM8900_BTC_CTRL_CSEL0)
  -> RSM8900_WriteOneReg(RSM8900_BTC_EXT, 0)
  -> RSM8900_WriteOneReg(RSM8900_BTC_FLAG, 0)
  -> RSM8900_WriteOneReg(RSM8900_EXT_BACKUP, 0x08)
```

注意：

- 当前初始化每次都会写默认时间。如果产品需要保存RTC真实时间，建议只在 `VLF` 等失效标志置位时才写默认时间。
- `RSM8900_TimerWakeup` 只在检测到 `TF` 时置 `1`，代码中没有自动清零接口。

#### 3.2 `RSM8900_MainFunc(void)`

功能：

- 当前为空函数，保留周期任务接口。

调用关系：

```text
RSM8900_MainFunc
  -> no operation
```

#### 3.3 `RSM8900_SetTime(RSM8900_time* pTime)`

功能：

- 校验时间合法性。
- 将十进制时间转换为BCD。
- 从 `0x00` 开始连续写入7个时间日历寄存器。

输入结构：

```c
typedef struct
{
    uint8  sec;
    uint8  min;
    uint8  hour;
    uint8  week;   /* 0~6 */
    uint8  date;
    uint8  month;
    uint16 year;
} RSM8900_time;
```

运行逻辑：

```text
RSM8900_SetTime
  -> RTC_Time_Vaild(pTime)
  -> DECtoBCD(sec/min/hour/date/month/year)
  -> week转换为单bit
  -> RSM8900_WriteReg(RSM8900_BTC_SEC, date, 7)
```

寄存器设置：

| 地址 | 写入内容 |
|---:|---|
| `0x00` | `DECtoBCD(sec)` |
| `0x01` | `DECtoBCD(min)` |
| `0x02` | `DECtoBCD(hour)` |
| `0x03` | `1 << week` |
| `0x04` | `DECtoBCD(date)` |
| `0x05` | `DECtoBCD(month)` |
| `0x06` | `DECtoBCD(year % 100)` |

返回值：

- `E_OK`：时间合法且写入已发起。
- `E_NOT_OK`：时间非法。

#### 3.4 `RSM8900_GetTime(RSM8900_time* pTime)`

功能：

- 从 `0x00` 开始连续读取7个时间日历寄存器。
- 屏蔽无关位。
- 将BCD转换为十进制。
- 将星期单bit转换为 `0~6`。
- 再次校验时间合法性。

运行逻辑：

```text
RSM8900_GetTime
  -> RSM8900_ReadReg(RSM8900_BTC_SEC, data, 7)
  -> BCDtoDEC(data[0] & 0x7F) -> sec
  -> BCDtoDEC(data[1] & 0x7F) -> min
  -> BCDtoDEC(data[2] & 0x3F) -> hour
  -> RSM8900_GetWeekDay(data[3] & 0x7F) -> week
  -> BCDtoDEC(data[4] & 0x3F) -> date
  -> BCDtoDEC(data[5] & 0x1F) -> month
  -> BCDtoDEC(data[6]) -> year
  -> RTC_Time_Vaild(pTime)
```

返回值：

- `E_OK`：读取出的时间合法。
- `E_NOT_OK`：读取出的时间非法。

#### 3.5 `RSM8900_SetAlarm(RSM8900_alarm* pAlarm)`

功能：

- 设置RTC Alarm中断。
- 写入闹钟分钟、小时、星期/日期。
- 清除 `AF`。
- 使能 `AIE` 和 `UIE`。

输入结构：

```c
typedef struct
{
    uint8 al_min;
    uint8 al_hour;
    uint8 al_week;
    uint8 al_day;
} RSM8900_alarm;
```

运行逻辑：

```text
RSM8900_SetAlarm
  -> RSM8900_ReadReg(RSM8900_BTC_EXT, ctrl, 2)
       ctrl[0] = EXT
       ctrl[1] = FLAG
  -> al_min/al_hour转换为BCD
  -> 根据 EXT.WADA 判断第三个字段写星期还是日期
  -> RSM8900_ReadOneReg(RSM8900_BTC_CTRL, &ctrlreg)
  -> 如果 AIE/UIE 已置位，先清除并写回CTRL
  -> RSM8900_WriteReg(RSM8900_BTC_ALARM_MIN, alarmvals, 3)
  -> 清 FLAG.AF
  -> 置 CTRL.AIE 和 CTRL.UIE
```

寄存器设置：

| 寄存器 | 设置 |
|---|---|
| `ALARM_MIN(0x08)` | `DECtoBCD(al_min)` |
| `ALARM_HOUR(0x09)` | `DECtoBCD(al_hour)` |
| `ALARM_WEEK_OR_DAY(0x0A)` | `1 << al_week` 或 `al_day` |
| `FLAG(0x0E)` | 清 `AF` |
| `CTRL(0x0F)` | 置 `AIE/UIE` |

注意：

- 当前代码在 `WADA = 1` 时直接写入 `al_day`，没有执行BCD转换；如果芯片手册要求日期闹钟为BCD，需要确认这里是否应改为 `DECtoBCD(al_day)`。

#### 3.6 `RSM8900_GetAlarm(RSM8900_alarm* pAlarm)`

功能：

- 读取Alarm设置。
- 判断Alarm中断是否使能。
- 判断 `AF` 是否触发。

运行逻辑：

```text
RSM8900_GetAlarm
  -> RSM8900_ReadReg(RSM8900_BTC_ALARM_MIN, alarmvals, 3)
  -> RSM8900_ReadReg(RSM8900_BTC_EXT, ctrl, 2)
  -> 解析分钟/小时
  -> 根据 EXT.WADA 解析星期或日期
  -> RSM8900_ReadOneReg(RSM8900_BTC_CTRL, &ctrlreg)
  -> 如果 CTRL.AIE=1 且 FLAG.AF=1，返回1
  -> 否则返回0
```

返回值：

- `1`：Alarm中断使能且Alarm Flag已触发。
- `0`：未触发或Alarm中断未使能。

#### 3.7 `RSM8900_SetCycWake(void)`

功能：

- 使用全局变量 `RSM8900_Cycwake_min` 配置周期定时唤醒。
- 默认周期为 `120min`。

运行逻辑：

```text
RSM8900_SetCycWake
  -> RSM8900_SetCycWakeMinutes(RSM8900_Cycwake_min)
```

推荐调用时机：

- 系统准备进入休眠前调用，用RTC Timer配置下一次唤醒时间。

#### 3.8 `RSM8900_SetCycWakeMinutes(uint16 wakeup_min)`

功能：

- 配置RTC倒计时Timer。
- 清除旧的 `TF`。
- 使能Timer和Timer中断。

参数：

- `wakeup_min`：定时唤醒周期，单位分钟。
- 有效范围：`1~4095`。

运行逻辑：

```text
RSM8900_SetCycWakeMinutes
  -> 参数范围检查
  -> 拆分12位Timer Counter
  -> 读取 EXT/CTRL/FLAG
  -> EXT: 配置TSEL0/TSEL1/TE/USEL
  -> CTRL: 清AIE/UIE，置TIE
  -> FLAG: 清TF
  -> 写 TIMER_CNT_0/TIMER_CNT_1
  -> 写 FLAG
  -> 写 CTRL
  -> 写 EXT
```

寄存器设置：

| 寄存器 | 设置 |
|---|---|
| `TIMER_CNT_0(0x0B)` | `wakeup_min`低8位 |
| `TIMER_CNT_1(0x0C)` | `wakeup_min`高4位 |
| `FLAG(0x0E)` | 清 `TF` |
| `CTRL(0x0F)` | 清 `AIE/UIE`，置 `TIE` |
| `EXT(0x0D)` | 置 `TSEL0/TSEL1/TE/USEL` |

返回值：

- `E_OK`：参数合法并完成寄存器写入。
- `E_NOT_OK`：参数为0或大于4095。

### 4. I2C读写逻辑

#### 4.1 读寄存器

`RSM_IIC_Receive_Data(cmd, pData, length)` 采用两段式同步传输：

```text
1. I2C write:
   SlaveAddress = 0x32
   DataBuffer   = &cmd
   BufferSize   = 1
   RepeatedStart = TRUE

2. I2C read:
   SlaveAddress = 0x32
   DataBuffer   = pData
   BufferSize   = length
   RepeatedStart = TRUE
```

也就是先发送寄存器地址，再读取指定长度的数据。

#### 4.2 写寄存器

`RSM_IIC_Send_Data(cmd, pData, length)` 会组包：

```text
RSM_tx_data[0] = cmd
RSM_tx_data[1] = pData[0]
RSM_tx_data[2] = pData[1]
...
```

然后一次同步发送：

```text
SlaveAddress = 0x32
BufferSize   = 1 + length
DataBuffer   = RSM_tx_data
```

最大数据长度：

```c
#define RSM_MAX_IIC_DATA_LEN    16U
#define RSM_MAX_IIC_BUFFER_LEN  (1U + RSM_MAX_IIC_DATA_LEN)
```

## 6. 当前实现注意事项

- `RSM8900_Init()` 当前每次都会写默认时间，这可能覆盖RTC实际保持的时间。
- `RSM8900_TimerWakeup` 置位后没有驱动内清除函数，上层使用后可按项目需要手动清零或增加接口。
- `RSM8900_SetCycWakeMinutes()` 会清 `AIE/UIE` 并置 `TIE`，因此配置Timer唤醒会关闭Alarm/Update中断使能。
- `RSM8900_SetAlarm()` 会置 `AIE/UIE`，如果同时使用Timer唤醒和Alarm，需要明确中断组合策略。
- `RSM_IIC_Receive_Data()` 和 `RSM_IIC_Send_Data()` 声明返回 `void`，但参数非法时使用了 `return false;`，从C语义上建议后续改成 `return;` 或将函数改成返回状态。
- `RTC_Time_Vaild()` 函数名拼写为 `Vaild`，当前保持现状以兼容已有调用。
- 日期合法性未区分月份天数和闰年。

# EB 配置
![](assets/2026.06.08%20RSM8900开发/file-20260608175427524.png)

![](assets/2026.06.08%20RSM8900开发/file-20260609092506702.png)
这是当前 `I2cChannel_0` 的 Master 通道配置，整体意思是：**S32K324 作为 I2C 主机，通过 LPI2C + DMA，以 400 kHz 速率访问 RTC 芯片**。

关键项解释如下。

`I2cClockRef = AIPS_SLOW_CLK`

这是 LPI2C 外设的输入时钟来源。I2C 的 SCL 最终就是基于这个时钟分频生成的。

也就是：

```text
AIPS_SLOW_CLK -> LPI2C分频/时序配置 -> SCL输出
```

`I2c Asynchronous Method = LPI2C_USING_DMA`

表示异步传输方式使用 DMA。  
如果调用 `I2c_AsyncTransmit()`，数据收发会通过 DMA 搬运，CPU不用逐字节搬数据。

但你当前 `RSM_IIC.c` 里实际用的是：

```c
I2c_SyncTransmit(0U, &RequestData);
```

也就是同步传输。是否真正走 DMA，要看底层 MCAL 对 SyncTransmit 是否也复用这个配置。至少从上层代码看，你现在不是主动用异步 DMA 接口。

`I2c TX DMA Channel` / `I2c RX DMA Channel`

发送和接收使用的 DMA 逻辑通道，当前都配成：

```text
dmaLogicChannel_I2C_RTC
```

这表示 I2C RTC 通道的 TX/RX DMA 都引用这个 MCL DMA 逻辑通道。一般更常见的是 TX/RX 分开两个 DMA channel；这里能不能共用要看 MCAL 配置是否合法以及底层是否要求独立通道。

`I2c Prescaler = LPI2C_MASTER_PRESC_DIV_2`

LPI2C 主时钟预分频为 2。  
如果 `AIPS_SLOW_CLK` 是 `80 MHz`，那么进入 I2C 时序模块的时钟大约是：

```text
80 MHz / 2 = 40 MHz
```

实际还要看工程里的 MCU 时钟配置。

`Glitch Filter SDA/SCL = 0`

SDA/SCL 毛刺滤波周期为 0，表示没有额外数字滤波。  
如果总线环境干净、线短、速率 400 kHz，一般可以；如果波形有毛刺或干扰，可以考虑打开，比如 1~2 cycles，但会影响时序裕量。

`Pin Low Timeout = 0`

引脚低电平超时关闭。  
如果 SDA/SCL 被外设或异常状态一直拉低，驱动不会因为这个配置自动报 pin low timeout。

`Bus Idle Timeout = 0`

总线空闲超时关闭。  
不会根据总线 idle 时间触发超时检测。

`Data Valid Delay = 1`

数据有效延迟，影响 SDA 数据相对 SCL 的采样/保持时序。当前是 1 个 LPI2C 内部周期。

`Setup Hold Delay = 2`

START/STOP 或数据建立保持相关的延迟，当前是 2 个周期。

`Clock High Period = 21`

SCL 高电平周期配置为 21 个内部时钟周期。

`Clock Low Period = 26`

SCL 低电平周期配置为 26 个内部时钟周期。

这两个值决定 SCL 的高低电平宽度。低电平比高电平略长，是很常见的 I2C 配置。

`Baud Rate = 400000.0`

目标 I2C 速率是：

```text
400 kHz
```

这属于 Fast-mode。RTC 芯片如果支持 400 kHz，就可以使用；如果芯片只支持 Standard-mode 100 kHz，就需要改成 `100000`。

和你当前代码对应起来：

```c
RequestData.SlaveAddress = 0x32;
RequestData.HighSpeedMode = TRUE;
RequestData.RepeatedStart = TRUE;
I2c_SyncTransmit(0U, &RequestData);
```

含义是：

- 使用 I2C Channel 0
- 访问 7-bit 地址 `0x32` 的 RTC
- 当前配置目标速率 400 kHz
- 读寄存器时先写寄存器地址，再 repeated start 读数据
- 写寄存器时一次发送寄存器地址 + 数据

需要重点确认两点：

1. RTC 手册是否支持 `400 kHz` I2C。
2. TX/RX DMA 是否允许共用 `dmaLogicChannel_I2C_RTC`，否则建议 TX/RX 分开配置。
