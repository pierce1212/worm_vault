# REG_PORT

## REG_PORT功能是不是浪费了？


![](assets/2026.06.03%20功能安全-REG_PORT%20XRDC/file-20260604093930140.png)
### S32K324芯片工作在什么模式下？
是的，S32K324 这类 S32K3 MCU 的 CPU 核心主要工作在 Supervisor 模式下。更准确地说：

    S32K324 用的是 Arm Cortex-M7 核，不是 Cortex-A/R 那种传统意义上的 ARM Supervisor/User/FIQ/IRQ 多模式体系。Cortex-M7 的权限模型通常叫：

    - Thread mode
    - Handler mode
    - Privileged
    - Unprivileged

    所以如果你问“是不是 Supervisor”，从车载/MCAL/AUTOSAR工程语境里可以近似理解为：
    绝大多数底层代码、启动代码、MCAL、OS 内核、驱动初始化都运行在 privileged/supervisor 权限下。
    但严格 ARM Cortex-M7 术语不是 “Supervisor mode”，而是 “Privileged Thread mode / Handler mode”。

#### S32K324 CPU 实际有哪些模式？

    Cortex-M7 主要有两个运行模式：
    1. Thread mode
    普通代码运行的模式。
    比如：
    - main()
    - AUTOSAR BSW 初始化
    - SWC Runnable
    - 普通应用函数
    - 大部分任务 Task


    Thread mode 又分：
    - Privileged Thread mode
    - Unprivileged Thread mode

    2. Handler mode
    异常/中断服务运行的模式。
    比如：
    - Reset_Handler
    - NMI_Handler
    - HardFault_Handler
    - SysTick_Handler
    - eMIOS interrupt
    - CAN interrupt
    - DMA interrupt
    - OS tick interrupt
    **Handler mode 永远是 privileged**


#### S32K324 上电复位后是什么权限？

    复位后从 Reset_Handler 开始执行，处于特权级。

    也就是：

    text
    Privileged


    启动阶段可以访问：

    - SCB
    - NVIC
    - MPU
    - Cache
    - Clock
    - MC_ME
    - MSCR
    - SIUL2
    - STM
    - eMIOS
    - FlexCAN
    - DMA
    - Flash controller
    - peripheral registers

    **这些都需要 privileged 权限。**

#### AUTOSAR 工程里一般怎么用？

    如果是常见 S32K324 AUTOSAR/MCAL 工程，通常是：

    
    Bootloader: privileged
    Startup code: privileged
    MCAL init: privileged
    BSW: privileged
    OS kernel: privileged
    ISR: privileged
    大部分 SWC Runnable: 也可能 privileged
    
```
    如果项目启用了 **AUTOSAR OS 的内存保护、应用隔离、MPU**，那么才可能有：
    **Trusted OS-Application: privileged**
    **Non-trusted OS-Application: unprivileged**
    **也就是说，是否有“用户态/非特权态”，取决于 AUTOSAR OS 配置。**

```
    很多量产项目为了安全隔离会配置 MPU/OS-Application，但很多普通 demo 或早期开发工程，几乎所有东西都跑在 privileged 下。
#### 什么时候会从 privileged 切到 unprivileged？

    Cortex-M7 通过 CONTROL 寄存器控制 Thread mode 权限。

    关键位是：
    CONTROL.nPRIV
    含义：
    text
    CONTROL.nPRIV = 0  -> Thread mode privileged
    CONTROL.nPRIV = 1  -> Thread mode unprivileged
    注意：
    - Handler mode 永远 privileged
    - unprivileged Thread mode 不能自己改回 privileged
    - 需要通过 SVC 异常进入 Handler mode，再由 OS 帮它切回或执行特权服务
    这就是 RTOS/AUTOSAR OS 实现用户态系统调用的基础。
#### SVC 是什么？

    SVC，全称 Supervisor Call。
    在 Cortex-M 上，SVC 是一种异常：
    asm
    svc #imm
    执行后 CPU 进入：
    SVC_Handler
    Handler mode
    Privileged
    用途是让非特权应用请求内核服务，比如：
    - 启动任务
    - 切换上下文
    - 访问受保护资源
    - 调用 OS service
    - 请求特权驱动操作

**==这也是为什么你会听到 “Supervisor” 这个词。不是因为 Cortex-M 有传统 SVC mode，而是因为 SVC 异常承担了“请求 supervisor 服务”的功能。==**


 ==S32K324 是 Cortex-M7 多核架构，具体型号可能是双核/锁步相关配置。每个 Cortex-M7 core 都有自己的：

    text
    Thread mode
    Handler mode
    CONTROL
    MSP/PSP
    NVIC exception handling
    每个核复位/启动后都是 privileged。
    但外设访问、内存保护、跨核资源访问还会受到：

    - MPU
    - XRDC / resource domain controller
    - MC_ME core control
    - Clock / peripheral gating
    - AUTOSAR OS protection
    - Safety/security 配置
的限制。所以不能只看 CPU mode，还要看==**系统级访问控制。**

#### 嵌入式中的线程的含义？一个OS算一个线程吗？
 在 AUTOSAR Classic 里，你平时说的 Task，基本可以理解成“嵌入式里的线程”。但它通常不是 PC/Linux 那种完整线程，而是一个由 AUTOSAR OS 调度的执行上下文。
	一个 Core 上运行 OS，OS 根据 Task 优先级、激活状态、事件、ScheduleTable/Alarm 去调度多个 Task。不是“整个 Core 是一个线程”，而是：text
    一个 Core 上有一个 OS
    这个 OS 管理多个 Task
    每个 Task 可以近似理解成一个线程
#### 嵌入式里为什么很少说进程？

因为像 S32K324 这种 MCU 通常没有 MMU，只有 MPU。没有 MMU 就很难实现 Linux 那种完整“进程隔离”。S32K324 Cortex-M7 有 MPU，可以做内存保护，但没有 Linux 那种完整虚拟内存进程模型。所以 Classic AUTOSAR 里一般不说：text process
而说：
	text
	Task
	ISR
	OS-Application
	Core
	Runnable
如果非要类比：
text
AUTOSAR Task ≈ thread
AUTOSAR OS-Application ≈ protection domain / lightweight process-like group
SWC Runnable ≈ function called inside a task。

# XRDC

==**MDAC==**控制访问Master的所在Domain，同一时刻一个Master只能处在一个Domain，一个Domain可以有多个Master，K3最大可以配置4个Domian。不同的的PN所支持的Domian数量是不一样的，具体可参考RM手册中的表Number of domains。

**==MRC==****：控制3个Domain对每块Memory（起始地址可配置，最大可配置36块Memory区间）的读写权限，另外还可以给每块Memory加硬件信号量（SEMA42，在后面的文章中会介绍到）。

==**PAC**==：控制3个Domain对外设的读写权限，另外还可以给每个外设加硬件信号量（SEMA42，在后面的文章中会介绍到）。
## 资源访问过程
*一个Master对外设或者Memory发起读写申请。*
*XRDC的Master控制侧截断申请，并加上MDAC里的跟Domain有关的信息。*
*加上额外信息的读写申请发送到内部crossbar总线上继续传输。*
*当读写申请来到了XRDC的Slave控制侧时，XRDC再次截断读写申请并将之前XRDC强加到读写申请的额外信息与Slave的权限控制信息进行对比（这一步叫做==**DdACP==**评估，记住这个词，后面的文章会出现）。*
*如果对比成功，则成功操作并返回读写信息给Master，如果对比不成果，则触发BusFault_Handler中断（需通过写SCB->SHCSR使能，否则触发HardFault_Handler中断）。*

# MPU
## MPU和OS的内存保护是一回事吗？
MPU 配置 = 硬件保护资源怎么划分。
OS 内存保护 = AUTOSAR OS 利用 MPU 去实现 Task/ISR/Application 级别的访问控制。 你现在在 EB tresos 里配置 MPU，后续继续用 OS 的内存保护，这个方向是对的；但要注意不要把“MCAL/芯片级 MPU 初始化”和“OS Memory Protection 配置”混成两个互相打架的配置。
 MPU 是硬件
    S32K3 里面的 MPU/系统内存保护单元负责定义：
    - 哪些地址范围可读/可写/可执行
    - 哪些 master/core 可以访问
    - 访问违反后产生 fault
    - region/descriptor 的起始地址、大小、权限
    它本身不认识 AUTOSAR Task、OS-Application、Trusted/NonTrusted 这些概念。
    
OS 内存保护是软件机制
    AUTOSAR OS 的 Memory Protection 关心的是：
    - 某个 Task/ISR 属于哪个 OS-Application
    - 这个 OS-Application 可以访问哪些 code/data/peripheral
    - Task 切换时，**OS 是否重新装载/切换 MPU 配置**
    - **非法访问后调用 ProtectionHook，可能终止 Task/Application 或复位*
    也就是说，OS Memory Protection 通常是“通过配置和动态切换 MPU region 来实现的”。

| -------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| `Name`               | 配置容器名字，方便 EB tresos/生成代码识别，例如 `RegionConfig_0`。它不是硬件寄存器字段。                                                                        |
| `Region Number`      | MPU Region 编号。Cortex-M7 支持多个 Region，编号越大通常优先级越高；地址重叠时高编号 Region 覆盖低编号 Region。`0` 是最低优先级。                                          |
| `Start Address`      | 该 Region 的起始地址，也就是 MPU `RBAR` 基地址。必须按 Region Size 对齐。当前 `0x0`。                                                                    |
| `End Address`        | 该 Region 的结束地址，通常是包含式结束地址。当前 `0xFFFFFFFF`。                                                                                        |
| `Region Size`        | Region 覆盖大小，等于 `End - Start + 1`。当前 `0x100000000`，也就是完整 4GB 地址空间。MPU Region 大小必须是 2 的幂。                                           |
| `Memory type`        | 内存属性类型，决定该区域按普通内存、设备寄存器、强顺序内存等方式访问。当前 `MPU_M7_MEM_STRONG_ORDER` 表示强顺序、不可缓存、访问顺序严格，通常很慢，多用于特殊区域或默认保护背景。                            |
| `Access Rights`      | 访问权限，区分特权态 `Privileged` 和非特权态 `Unprivileged`，并控制读/写/执行。当前 `MPU_M7_PRIV_UNPRIV_NONE` 表示特权态和非特权态都不能读、写、执行。                          |
| `Outer Cache Policy` | 外部缓存策略。只有当 `Memory type = MPU_M7_MEM_NORMAL_CACHEABLE` 时才真正参与配置；当前 Memory type 不是这个，所以这里的 `NO_CACHE` 实际基本无效。                      |
| `Inner Cache Policy` | 内部缓存策略，主要对应 Cortex-M7 内部 D-Cache/I-Cache 相关属性。同样只有 `MPU_M7_MEM_NORMAL_CACHEABLE` 时生效。                                             |
| `Subregion Mask`     | 子区域屏蔽。一个 MPU Region 可分成 8 个等大小 subregion，mask 某一位为 `1` 表示禁用对应 subregion。当前 `0` 表示 8 个子区域都启用。当前 4GB Region 下，每个 subregion 是 512MB。 |
| `Shareable`          | 是否设置为 Shareable 区域，用于多核、多 bus master、DMA 等共享访问场景的内存一致性/顺序属性。当前未勾选，即 non-shareable。                                                |

**Memory type**
`Memory type` 决定 CPU 访问这段地址时采用什么内存属性：是否缓存、是否允许写缓冲、是否允许访问重排、是否按外设寄存器处理。它不是在说“这段地址是 Flash 还是 RAM”，而是在告诉 Cortex-M7 MPU：“访问这段地址要按什么规则来”。

| Memory Type                       | 含义                                                                                            | 典型用途                                                  |
| --------------------------------- | --------------------------------------------------------------------------------------------- | ----------------------------------------------------- |
| `MPU_M7_MEM_STRONG_ORDER`         | 强顺序内存。不可缓存，访问顺序最严格，CPU 不会随意合并、推测或重排访问。性能最保守。                                                  | 默认兜底保护区、极特殊的强顺序访问区域。一般不用于普通 Flash/RAM。                |
| `MPU_M7_MEM_DEVICE_SHARED`        | 设备类型内存，Shared Device。不可缓存，按外设寄存器/MMIO 的语义访问，访问顺序比普通内存更严格。                                     | 多核或多个 bus master 可能访问的外设寄存器区。                         |
| `MPU_M7_MEM_NORMAL_IO_WR_THROUGH` | 普通内存，Inner/Outer 都是 Write-through，No write allocate。读可以走 cache；写入会同时更新 cache 和实际内存。           | 普通 RAM/Flash，想要一定 cache 性能，但希望写入更快同步到内存。              |
| `MPU_M7_MEM_NORMAL_IO_WR_BACK1`   | 普通内存，Inner/Outer 都是 Write-back，No write allocate。写入可先留在 cache，之后再回写内存；写 miss 不分配 cache line。  | CPU 频繁访问的 RAM，性能好，但 DMA 共享时要做 cache clean/invalidate。 |
| `MPU_M7_MEM_NORMAL_IO_NO_CACHE`   | 普通内存，但不可缓存。注意它仍是 Normal memory，不是 Device memory。                                              | DMA buffer、共享 RAM、通信 buffer，不想处理 cache 一致性时很常用。       |
| `MPU_M7_MEM_NORMAL_IO_WR_BACK2`   | 普通内存，Inner/Outer 都是 Write-back，Write and read allocate。读 miss、写 miss 都可能分配 cache line，性能通常最好。 | CPU 独占或主要由 CPU 使用的 SRAM、stack、heap、普通数据区。             |
| `MPU_M7_MEM_DEVICE_NOSHARE`       | 设备类型内存，Non-shareable Device。不可缓存，按外设寄存器语义访问。                                                  | 普通外设寄存器地址空间，比如 UART、SPI、CAN、ADC、MCAL 外设寄存器。           |
| `MPU_M7_MEM_NORMAL_CACHEABLE`     | 普通可缓存内存，但具体缓存策略不由这个枚举固定，而是继续看下面的 `Outer Cache Policy` 和 `Inner Cache Policy`。                 | 想单独配置 inner/outer cache policy 的普通 RAM/Flash 区域。      |

这里名字里的 `NORMAL_IO` 容易误会，它不是“外设 I/O”的意思，更接近 “Inner/Outer cache attributes”。所以：

**外设寄存器不要选 `NORMAL_IO_*`，应该选 `DEVICE_NOSHARE` 或 `DEVICE_SHARED`。**  
**DMA buffer 不想维护 cache 时，选 `NORMAL_IO_NO_CACHE`。**  
**普通 RAM 想要高性能，选 `NORMAL_IO_WR_BACK2` 或 `NORMAL_CACHEABLE + write-back`。**  
**代码 Flash 通常选 Normal + cacheable，并配合访问权限 `RX`。**  
**默认全地址禁止访问区可以选 `STRONG_ORDER + PRIV_UNPRIV_NONE`。**


下面这几项可以分成三组看：权限、缓存策略、区域细分/共享属性。
**Access Rights**

记法里：

`PRIV` = 特权态访问权限  
`UNPRIV` = 非特权态访问权限  
`R` = 读  
`W` = 写  
`X` = 执行代码  
`NONE` = 不能访问

| 选项 | 含义 |
|---|---|
| `MPU_M7_PRIV_UNPRIV_NONE` | 特权态、非特权态都不能读/写/执行。常用于默认禁止访问区域。 |
| `MPU_M7_PRIV_RW_UNPRIV_NONE` | 特权态可读写，不能执行；非特权态无权限。适合 OS/驱动私有数据区。 |
| `MPU_M7_PRIV_RWX_UNPRIV_NONE` | 特权态可读写执行；非特权态无权限。一般少用，因为数据区最好不要可执行。 |
| `MPU_M7_PRIV_RW_UNPRIV_R` | 特权态可读写，非特权态只读，均不可执行。适合用户态可读的配置/常量数据。 |
| `MPU_M7_PRIV_RWX_UNPRIV_RX` | 特权态可读写执行，非特权态可读执行。可用于需要特权态更新、用户态执行的代码区，但要谨慎。 |
| `MPU_M7_PRIV_RW_UNPRIV_RW` | 特权态、非特权态都可读写，不能执行。适合普通用户态 RAM 数据区。 |
| `MPU_M7_PRIV_RWX_UNPRIV_RWX` | 特权态、非特权态都可读写执行。最宽松，不建议用于量产安全配置。 |
| `MPU_M7_PRIV_R_UNPRIV_NONE` | 特权态只读，非特权态无权限。适合特权态私有常量区。 |
| `MPU_M7_PRIV_RX_UNPRIV_NONE` | 特权态可读执行，非特权态无权限。适合内核/驱动代码区。 |
| `MPU_M7_PRIV_R_UNPRIV_R` | 特权态、非特权态都只读，不能执行。适合全局常量数据。 |
| `MPU_M7_PRIV_RX_UNPRIV_RX` | 特权态、非特权态都可读执行，不能写。适合普通应用代码区。 |

**常用搭配：代码区用 `RX`，数据区用 `RW` 且不要 `X`，外设寄存器用 `RW` 且不要 `X`，默认兜底区用 `NONE`。**

**Outer / Inner Cache Policy**

**这两个只有在 `Memory type = MPU_M7_MEM_NORMAL_CACHEABLE` 时才生效；如果你选的是 `DEVICE_*`、`STRONG_ORDER`、`NORMAL_IO_NO_CACHE` 之类，NXP 驱动会走 Memory Type 固定属性，Cache Policy 基本不参与最终配置。**

| 选项 | 含义 |
|---|---|
| `MPU_M7_CACHE_POLICY_NO_CACHE` | 不缓存。CPU 每次访问都直接走到内存/总线，性能低但一致性简单。 |
| `MPU_M7_CACHE_POLICY_W_BACK_WR_ALLOCATE` | Write-back + read/write allocate。性能最好，读 miss 和写 miss 都可能分配 cache line；写入先留在 cache，之后再回写内存。 |
| `MPU_M7_CACHE_POLICY_W_THROUGH_NO_W_ALLOCATE` | Write-through + no write allocate。写 cache 时同时写内存，写 miss 不分配 cache line。一致性比 write-back 简单。 |
| `MPU_M7_CACHE_POLICY_W_BACK_NO_W_ALLOCATE` | Write-back + no write allocate。读可缓存，写 miss 不分配 cache line；写中的 cache line 后续再回写。 |

`Inner Cache Policy` 通常更直接影响 Cortex-M7 核内 cache 行为。  
`Outer Cache Policy` 是外层内存系统/cache 属性；如果芯片没有独立外部 cache，通常保持和 Inner 一致即可。

**Subregion Mask**

一个 MPU Region 会被分成 8 个等大小的 subregion。`Subregion Mask` 是 8 bit：

**`0` 表示全部启用。**  
**某一 bit 为 `1` 表示禁用对应 subregion。**  
**bit0 对应最低地址的第 0 个 subregion，bit7 对应最高地址的第 7 个 subregion**。

例如一个 `256KB` Region，每个 subregion 是 `32KB`。  
`SubregionMask = 0x03` 表示禁用最低地址的两个 `32KB`，也就是前 `64KB` 不受这个 Region 保护。

**Shareable**

`Shareable = true` 表示这个区域可能被多个 bus master/核/DMA 共享访问，CPU 会按共享内存属性处理访问顺序和一致性。  
`Shareable = false` 表示非共享区域，适合 CPU 独占的普通 RAM/Flash。

**经验上：外设寄存器、DMA 共享 buffer、多核共享 RAM 更倾向于 `Shareable`；单核私有 stack/heap/普通变量区通常不用勾。**

重点提醒：你当前这个 `RegionConfig_0` 是“全地址空间 + 无任何访问权限”。如果 MPU 初始化时只有这一个 Region 生效，CPU 后续访问 Flash/RAM/外设都会触发 MemManage Fault。它更像是“默认禁止访问”的底层保护 Region，正常还需要再配置更高编号的 Region 去开放 Flash、RAM、外设等白名单区域。

## 如何确定中断编号？
![](assets/2026.06.03%20功能安全-REG_PORT%20XRDC/file-20260604093930051.png)
看两个地方就能确定：

1. 芯片头文件的 `IRQn_Type` 枚举  
   位置：`E:\github\ECAS_RTA_S32K324GHS_Heating\BasicSoftware\integration\mcal\src\modules\BaseNXP\header\S32K324_COMMON.h`

   这里是芯片真实外设 IRQ 号，例如：
   ```c
   PMC_IRQn  = 52
   CMU0_IRQn = 84
   CMU1_IRQn = 85
   CMU2_IRQn = 86
   ```

2. RTA-OS 的 `Interrupt_xx` 一般是“向量表编号”，不是外设 IRQ 号  
   Cortex-M 外设中断从向量表第 16 项开始，所以：

   ```text
   Interrupt_xx = IRQn + 16
   IRQn = Interrupt_xx - 16
   ```
你如果要配 PMC：
```text
PMC_IRQn = 52
Interrupt = 52 + 16 = Interrupt_68
```

你如果要配 CMU clock fail 相关中断：

```text
CMU0_IRQn = 84 -> Interrupt_100
CMU1_IRQn = 85 -> Interrupt_101
CMU2_IRQn = 86 -> Interrupt_102
```

工程里也能用生成文件反向验证这个规律：`Os_ConfigInterrupts.h` 里有类似：

```c
#define Os_IntID_Dma0_Ch0_IRQHandler      (20)
#define Os_IntChannel_Dma0_Ch0_IRQHandler (4)
```

这里 `IntID=20` 是 `Interrupt_20`，`IntChannel=4` 是真实 `DMATCD0_IRQn=4`，刚好 `20 - 16 = 4`。所以以后看到 OS 下拉里的 `Interrupt_xx`，先减 16，再去 `S32K324_COMMON.h` 里查对应的 `xxx_IRQn`。
