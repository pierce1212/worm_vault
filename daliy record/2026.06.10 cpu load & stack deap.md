# 什么是 CPU 负载
CPU 的“负载”本质上是：**CPU 在一段时间内有多少比例没有空闲，而是在执行有效工作或等待必须由 CPU 处理的事务。**

在嵌入式/实时系统里，常见定义是：

```text
CPU负载 = 忙碌时间 / 总时间 × 100%
```

例如 10 ms 周期内，**CPU 有 3 ms 在跑任务、ISR、中断回调、调度等**，剩下 7 ms 空闲：

```text
CPU负载 = 3 / 10 × 100% = 30%
```

通常算作 CPU 负载的包括：

- **主循环里的业务代码执行时间**
- **RTOS task 的运行时间**
- **中断 ISR 执行时间**
- **定时器回调、通信回调、DMA 完成回调等 CPU 实际执行的时间**
- **操作系统调度、上下文切换等开销**
- **因等待锁、关中断、忙等轮询造成 CPU 无法空闲的时间**

通常**不算 CPU 忙碌**的是：

- CPU 进入 idle task
- WFI/WFE 低功耗等待
- 硬件外设自己工作，比如 DMA 搬数据、CAN 控制器收发、ADC 自动采样，前提是 CPU 没有参与轮询等待

在 RTOS 里，经常用 idle task 来估算：

```text
CPU负载 = 100% - Idle任务占用率
```

比如 Idle 占 65%，CPU 负载就是 35%。

要注意，CPU 负载不等于“代码多不多”，也不等于“功能复杂不复杂”。真正关键的是：**在规定时间窗口内，CPU 被占用了多久，以及最坏情况下会不会来不及完成实时任务。** 对汽车/MCU 项目来说，平均负载只是参考，最坏执行时间、峰值中断占用、任务周期裕量更重要。

# 代码实现
RTM 里 CPU 负载主要有两种口径：**系统总负载**和**单个 Task/ISR 负载**。

**1. 系统 CPU 负载**

你们工程里 RTM 的系统负载变量是：

```c
Rtm_MonSystemLoadPer64k_au16[RTM_NUM_CORES]
```

计算逻辑在 [Rtm_Mon.c](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/src/cobra/Rtm/src/Rtm_Mon.c:313) 的 `Rtm_Mon_Idle()`。

核心公式是：

```text
Idle比例 = 本次采样窗口内的Idle时间 / 本次采样窗口总时间

CPU负载 = 1 - Idle比例
```

源码里不是直接用百分比，而是用 **Per 64 k** 格式，也就是：

```text
0x0000 = 0%
0x8000 = 50%
0xFFFF ≈ 100%
```

等价公式：

```text
loadPer64k = 65536 - (idleDelta * 65536 / totalDelta)
```

换成人看的百分比：

```text
CPU负载百分比 = loadPer64k / 65536 × 100%
```

举例：

```text
采样窗口总时间 = 100 ms
Idle时间 = 30 ms

CPU负载 = 1 - 30/100 = 70%
loadPer64k = 0.7 × 65536 ≈ 45875
```

所以如果你在标定/测量界面看到 `Rtm_MonSystemLoadPer64k_au16[x] = 45875`，它大约就是：

```text
45875 / 65536 × 100% ≈ 70%
```

**2. Task/ISR 负载**

每个被 RTM 监控的 task 都有：

```c
Rtm_MonTask[index].loadPer64k_u16
```

每个被监控的 ISR 有：

```c
Rtm_MonIsr[index].loadPer64k_u16
```

计算逻辑在 `Rtm_TaskEnd()` / `Rtm_IsrEnd()`。

公式是：

```text
某Task负载 = 采样窗口内该Task累计运行时间 / 采样窗口总时间
某ISR负载  = 采样窗口内该ISR累计运行时间 / 采样窗口总时间
```

源码里的形式类似：

```c
load = Rtm_Mul64kDiv32(accuRT, stopwatch - beginTimestamp);
```

也就是：

```text
loadPer64k = accuRT × 65536 / sampleWindowTime
```

举例：某个 10 ms task 在 1 秒采样窗口内运行了 25 次，每次平均 200 us：

```text
累计运行时间 = 25 × 200 us = 5000 us = 5 ms
采样窗口 = 1000 ms

Task负载 = 5 / 1000 = 0.5%
loadPer64k = 0.005 × 65536 ≈ 328
```

**3. RTM 的采样周期**

你们工程配置里有：

```c
#define RTM_MON_SAMPLES_PER_MINUTE ...
```

RTM 不是每次主函数都刷新统计值，而是按“每分钟采样多少次”来更新一次统计结果。

逻辑大概是：

```text
Rtm_Mon_MainFunction 周期调用
到达采样间隔后：
    Rtm_MonSamples++
    更新系统负载
    任务/ISR在结束 hook 里更新各自统计值
```

所以看到的 CPU load 不是瞬时值，而是**上一个采样窗口内的平均负载**。

**4. 这个负载包含什么**

系统负载：

```text
CPU负载 = 非Idle时间
```

所以它包含：

- Task 执行时间
- ISR 执行时间
- OS 调度开销
- 中断嵌套带来的 CPU 占用
- 忙等、轮询等没有进入 idle 的时间

不包含：

- CPU 真正处于 idle 的时间
- DMA/外设自己工作但 CPU 没参与的时间

**5. 看值时的重点**

如果看总 CPU 负载：

```text
Rtm_MonSystemLoadPer64k_au16[core]
```

如果看某个任务占多少 CPU：

```text
Rtm_MonTask[index].loadPer64k_u16
```

换算都一样：

```text
百分比 = loadPer64k_u16 × 100 / 65536
```

所以 RTM 模块里的 CPU 负载，本质就是：

```text
一段采样时间内，CPU没有空闲的时间比例
```

而 Task/ISR 的负载是：

```text
一段采样时间内，这个Task/ISR消耗的CPU时间比例
```

## RTM 中的 tick 的周期

**PIT 2 的 40000 tick = 1 ms，是 OS 调度 tick。**  
**RTM 里 `curRT/averageRT` 的 tick，不是 PIT 2 累加出来的，而是 OS stopwatch，也就是 STM 0 free-running counter。**

```c
Rtm_GetExecutionTime()
  -> GetExecutionTime()
    -> Os_Cbk_GetStopwatch()
      -> Mcu_Rb_GetSysTicks()
        -> TARGET_GETSYSTICKS32BITS()
          -> Target_GetSysTicks32Bits()
            -> IP_STM_0->CNT
```

对应代码：

```c
#define Mcu_Rb_GetSysTicks() TARGET_GETSYSTICKS32BITS()
```

```c
uint32 Target_GetSysTicks32Bits(void)
{
    STMCNT_Test = IP_STM_0->CNT;
    return (IP_STM_0->CNT);
}
```

所以 **RTM 的 task/ISR 运行时间单位 = STM 0 counter tick**。

再看配置：

```c
#define RTM_TICKS_PER_SECOND 40000000
```

MCU 配置里 `STM0_CLK` 也是：

```text
STM0_CLK = 4.0E7
```

所以当前工程配置下：

```text
STM0 counter frequency = 40 MHz
1 tick = 1 / 40000000 s = 25 ns
40000 ticks = 1 ms
```

也就是说：**40000 ticks = 1 ms 这个换算结果是对的，但原因不是 PIT 2，而是 RTM/OS stopwatch 用的 STM 0 也是 40 MHz。**

两套 tick 要区分：

| 类型 | 来源 | 含义 | 当前频率 |
|---|---|---|---|
| OS counter tick | `PIT_2_ISR -> Gpt_PIT2_0_Notification -> IncrementCounter()` | 调度周期，1 tick = 1 ms | PIT 2 40 MHz，启动值 40000 |
| RTM runtime tick | `IP_STM_0->CNT` | 测量运行时间 | STM 0 40 MHz |
| `Rtm_MonTask.averageRT` | `GetExecutionTime()` | task 执行时间，单位是 STM 0 tick | 40000 tick = 1 ms |

所以你之前看到：

```text
OsTask_SysOsApp_BSW_10ms averageRT = 153717
```

换算仍然是：

```text
153717 / 40000000 = 0.003842925 s = 3.843 ms
```

负载率：

```text
3.843ms / 10ms = 38.43%
```

验证方法也很简单：在 `Gpt_PIT2_0_Notification()` 里临时记录两次 STM 0 差值：

```c
static uint32 last;
static uint32 diff;

uint32 now = IP_STM_0->CNT;
diff = now - last;
last = now;
```

如果 1 ms PIT 2 正常，`diff` 应该接近：

```text
40000
```

这样就能证明 **PIT 2 的 1 ms 和 STM 0 的 RTM tick 都是 40 MHz 对齐的**。

### 为什么函数的运行时间比 TASK 运行总时间还要大？
这张同步截图就解释清楚了：

```text
Rtm_MonProcess[7].averageRT = 185133
Rtm_MonTask[6].averageRT    = 155850
```

确实还是 **process 比 task 大**：

```text
185133 - 155850 = 29283 ticks
29283 / 40000 = 0.732 ms
```

这说明 `Rtm_MonProcess[7]` 不是严格意义上的“task 内净执行时间”，它统计的是：

```c
Rtm_ProcessEnd - Rtm_ProcessStart
```

**也就是墙钟 duration**。

而 `Rtm_MonTask[6].averageRT` 来自：

```c
GetExecutionTime()
```

这是 OS 的 task net runtime。它可能不包含 ISR 抢占时间，甚至不包含某些 OS 调度/监控开销。

所以现在可以这样理解：

| 项目                            |  ticks |       时间 | 含义                                                  |
| ----------------------------- | -----: | -------: | --------------------------------------------------- |
| `Rtm_MonTask[6].averageRT`    | 155850 | 3.896 ms | task 净运行时间                                          |
| `Rtm_MonProcess[7].averageRT` | 185133 | 4.628 ms | `rba_SchMAnalyzer_MainFunction` 从 start 到 end 的墙钟时间 |
| 差值                            |  29283 | 0.732 ms | 期间可能被 ISR/OS 抢占，或统计口径不同                             |

所以这不是算错，而是两个监控口径不同。

但结论仍然成立：

```text
OsTask_SysOsApp_BSW_10ms 平均净运行时间 ≈ 3.896ms
10ms task 负载 ≈ 38.96%
```

而 `rba_SchMAnalyzer_MainFunction` 的墙钟耗时已经到：

```text
4.628ms
```

说明它几乎贯穿了这个 task 的主要执行区间，并且期间还可能被中断打断。  
**排查方向还是它。**如果把 `rba_SchMAnalyzer_MainFunction()` 临时屏蔽，再看 `Rtm_MonTask[6].averageRT` 是否大幅下降，就能最终确认。

### SchMAnalyzer 的配置


我按 AUTOSAR Classic / embedded C 的角度看了工程里的实现。`rba_SchMAnalyzer` 本质上是一个 **SchM critical section 分析工具模块**，用来检查各个 BSW 模块的 `SchM_Enter/Exit` 临界区保护是否合适：有没有缺锁、锁太弱、锁太强、实际根本不需要锁等。

**它解决什么问题**
它不是业务功能模块，而是集成/调试分析模块。比如 CanIf、Com、CanNm、ComM 这些 BSW 的 SchM 临界区会调用：

```c
rba_SchMAnalyzer_LogCall(..., RBA_SCHMANALYZER_SCHM_TYPE_EMPTY/SPINLOCK/INTERRUPT);
```

然后 `rba_SchMAnalyzer_MainFunction()` 周期性汇总：某个临界区被哪些 task / ISR / core 调用过，再判断当前锁类型是否合理。

**为什么它看起来像 SWC**
它在 ARXML 里被包装成 CDD SWC runnable：

- [rba_SchMAnalyzer_Cfg_SWCD.arxml](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/src/rba_SchMAnalyzer_Cfg_SWCD.arxml:11) 定义了 `COMPLEX-DEVICE-DRIVER-SW-COMPONENT-TYPE`
- [rba_SchMAnalyzer_Cfg_SWCD.arxml](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/src/rba_SchMAnalyzer_Cfg_SWCD.arxml:60) 做了 SWC-BSW runnable mapping
- [rba_SchMAnalyzer_Cfg_BSWMD.arxml](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/src/rba_SchMAnalyzer_Cfg_BSWMD.arxml:33) 配的是 `0.01s` 周期，也就是 10 ms

所以它在 RTA-CAR 里像 SWC，但实际算法主体是 BSW 集成代码：[rba_SchMAnalyzer.c](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/src/rba_SchMAnalyzer.c:507)。

**核心工作流程**
1. BSW 的 SchM wrapper 里调用 `rba_SchMAnalyzer_LogCall()`。
2. `LogCall()` 读取当前 `CoreID`、当前 `ISR` 或 `TaskID`。
3. 它把“某个临界区在某个 task/ISR/core 被用过”记录到 bit flag 里。
4. `rba_SchMAnalyzer_MainFunction()` 遍历所有临界区。
5. 对每个临界区统计 task/ISR/core 使用情况。
6. 根据锁类型算结果，保存到 `rba_SchMAnalyzer_Results_Schm_xxx` 变量里。

关键代码在这里：

- 记录调用：[rba_SchMAnalyzer.c](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/src/rba_SchMAnalyzer.c:161)
- 收集元数据：[rba_SchMAnalyzer.c](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/src/rba_SchMAnalyzer.c:254)
- 判断冲突：[rba_SchMAnalyzer.c](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/src/rba_SchMAnalyzer.c:303)
- 周期主函数：[rba_SchMAnalyzer.c](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/src/rba_SchMAnalyzer.c:507)

**结果含义**
结果 enum 在 [rba_SchMAnalyzer.h](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/api/rba_SchMAnalyzer.h:68)：

| 结果 | 含义 |
|---|---|
| `NOT_USED` | 当前测试没覆盖到这个临界区 |
| `OK` | 锁配置合理 |
| `NOT_NEEDED` | 只在一个上下文用到，当前看不需要锁 |
| `LOCK_TOO_STRONG` | 用了 spinlock，但实际只在单核/同类上下文，可能过强 |
| `MISSING_ISR_DISABLE_LOCK` | 单核多个上下文访问，但锁为空，需要关中断类保护 |
| `LOCK_TOO_WEAK` | 多核访问却只用了关本核中断，锁太弱 |
| `MISSING_SPINLOCK` | 多核访问但锁为空，需要 spinlock |
| `NOT_COMPILED` | 该 SchM 没编进当前工程 |

**为什么它 CPU 占用很高**
你的工程里 `rba_SchMAnalyzer_MainFunction()` 每 10 ms 被调一次，原来在 [Rte.c](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/src/rte/gen/Rte.c:20970)。

它每次都会：

```c
for (critSectIdx = 0; critSectIdx < RBA_SCHMANALYZER_SCHM_ENTRY_LAST; critSectIdx++)
```

而你这里临界区 ID 到 `272_279`，也就是接近 280 个 entry。每个 entry 又扫描：

```c
core * taskFlags * isrFlags
```

所以它很重。你之前看到 `Rtm_MonProcess[7]` 几 ms 是合理的，这个模块不适合长期 10 ms 周期跑在量产任务里。它更像集成阶段打开、跑测试、导出报告，然后关闭或降低频率。

**LocksCfg 怎么来**
`LocksCfg` 不是你在 SWC Runnable 页面里配的那个东西。它是 SchMAnalyzer 的项目配置：

- [rba_SchMAnalyzer_LocksCfg.h](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/cfg/rba_SchMAnalyzer_LocksCfg.h:8) 定义 task/core/isr 数量、默认锁类型、临界区 ID 列表
- [rba_SchMAnalyzer_LocksCfg.h](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/cfg/rba_SchMAnalyzer_LocksCfg.h:34) 明确写了 `DO NOT CHANGE`，因为 Excel generator 依赖这个顺序
- [rba_SchMAnalyzer_LocksCfg.c](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/cfg/rba_SchMAnalyzer_LocksCfg.c:7) 当前只是空的 `InitNotCompiledSchmFunctions`

目录里还有工具：

- ELF 分析并生成 config：[SchMAnalyzer_Elf_Analyzer_And_Config_Generator](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/tools/SchMAnalyzer_Elf_Analyzer_And_Config_Generator/_README.txt:1)
- 导出结果生成 Excel：[SchmAnalyzer_ReportGenerator](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/integration/src/bsw/rba_SchMAnalyzer/tools/SchmAnalyzer_ReportGenerator/_README.txt:3)

所以结论是：**调度周期/是否调用可以在 RTE/OS 配置里管；LocksCfg 这种临界区列表和 not compiled 信息优先走工具生成，实在没有生成链路才手动维护，而且不要随便改 enum 顺序。**
是的，**如果你现在不是在做 SchM 临界区锁分析/多核并发验证，那大概率用不到它**。

## 它真的需要吗？

它不是 ECU 正常功能必需模块，更像一个集成阶段的分析工具。你现在关心 CPU 负载，`rba_SchMAnalyzer_MainFunction()` 反而会干扰测量，因为它每 10 ms 扫一遍所有 SchM 临界区，之前已经把 `OsTask_SysOsApp_BSW_10ms` 拉得很高。

还有一个关键点：你现在 `RBA_SCHMANALYZER_FEATURE` 已经是 `STD_OFF`，这只会让各个 `SchM_Enter_xxx()` 里的 `rba_SchMAnalyzer_LogCall()` 不编进去。但是我刚看了下，`Rte.c` 里现在还是有：

```c
rba_SchMAnalyzer_MainFunction();
```

也就是说，**只关 `RBA_SCHMANALYZER_FEATURE` 还不够彻底**。它不记录日志了，但 `MainFunction` 仍然可能被 10 ms task 调用，继续空跑扫描。

建议你这样处理：

| 场景 | 建议 |
|---|---|
| 正常功能调试、CPU 负载测试、接近量产 | 关掉，不需要跑 |
| 想检查 SchM 锁是否合理 | 临时打开，跑测试后导出报告 |
| 平时长期运行 | 不建议开，尤其不建议 10 ms 周期跑 |

最干净的做法是在 RTA-CAR 配置里把 `rba_SchMAnalyzer_MainFunction` 的 10 ms 调度去掉，然后重新生成 RTE。临时验证也可以在 [Rte.c](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/src/rte/gen/Rte.c:20970) 里注释掉，但这个文件是生成文件，后面重新生成会被覆盖。

如果你只是想赶紧降负载，保持：

```c
#define RBA_SCHMANALYZER_FEATURE STD_OFF
```

并且不要调用：

```c
rba_SchMAnalyzer_MainFunction();
```

这样才是真的关闭。
