## 1. 给初学者的直白解释

NM 是 Network Management，中文通常叫“网络管理”。它不是负责发送普通业务信号的模块，而是负责回答几个问题：
 **<mark style="background: #FF5582A6;">Nm/CanNm 发 NM 报文维持网络</mark>**
- 这个 CAN 网络现在要不要醒着？
- 本 ECU 是否还需要通信？
- 其他 ECU 是否还在发 NM 报文，说明网络还不能睡？
- 所有节点都不需要通信时，什么时候进入睡眠？

可以把 **NM 理解成“CAN 网络的作息管理员”**。**业务功能想通信时，一般不直接调用 CanNm，而是通过 ComM 请求 **`COMM_FULL_COMMUNICATION`。ComM 再驱动 CanSM、CanIf、CanNm 等模块，把控制器、收发器和 NM 状态一起拉到工作状态**。**业务释放通信后，ComM/NM 会等定时器到期，再让网络进入 Prepare Bus Sleep 和 Bus Sleep。**

从整车角度看，判断一个 ECU 是否“醒着”，很多时候不是看 MCU 供电没供电，而是看它是否在总线上外发报文。它不发应用报文、不发 NM 报文时，整车网络通常就把它当成休眠节点。注意：这不等于物理下电，Prepare Bus Sleep 阶段就可能已经不发报文，但硬件还没有完全掉电。

主动唤醒和被动唤醒要分清：

- 主动唤醒：本 ECU 自己有工作需求，例如 KL15、充电插枪、定时唤醒、传感器触发、诊断请求等。软件上通常体现为 ComM User 请求 FullCom，最终走到 `CanNm_NetworkRequest()`。
- 被动唤醒：本 ECU 自己没有需求，是因为收到别的节点的 NM 报文或总线唤醒事件才醒。软件上通常体现为 `CanNm_PassiveStartup()` 或 `Nm_NetworkStartIndication()` 相关链路。

本工程使用的是 CAN NM：

- `Nm`：AUTOSAR NM Interface，统一抽象层。
- `CanNm`：CAN 总线上的具体 NM 实现。
- `ComM`：通信管理，接收应用的通信请求。
- `CanSM`：CAN 状态管理，负责 CAN 控制器网络模式。
- `CanIf/CanDrv/CanTrcv`：底层发送接收、控制器和收发器。

## 4. 当前代码中 NM 的使用流程

### 4.1 应用层入口：NmUT

`NmUT` 是一个示例/测试性质的 SWC。它的 runnable 是：

```c
FUNC (void, NmUT_CODE) RE_NmUT_func(void)
```

配置中它由 `TE_ASW_Nm_10ms` 触发，周期 `0.01 s`。

`NmUT.c` 里定义了每个 NM 通道的状态缓存：

```c
Nm_Test_t Nm_Test[NM_NUMBER_OF_CHANNELS];
```

`NM_NUMBER_OF_CHANNELS` 来自 `Nm_Cfg.h`，当前是 `1`。

### 4.2 应用请求网络

`RE_NmUT_func()` 每 10ms 扫描 `Nm_Test[index_chanel]`：

- 如果 `request_b = TRUE`
  - 通过 `Rte_Write_PP_NmState_uint8(APP_READY_FOR_NM_REQUEST)` 通知上层/执行管理逻辑。
  - 代码注释写明：这是为了让 `ExeMgrUT` 请求 `COMM_FULL_COMM`。
- 如果 `release_b = TRUE`
  - 通过 `Rte_Write_PP_NmState_uint8(APP_READY_FOR_NM_RELEASE)` 通知释放通信。
  - **代码注释写明：这是为了让 `ExeMgrUT` 请求 `COMM_NO_COMM`。**

**也就是说，本工程 `NmUT` 自己没有直接调用 `ComM_RequestComMode()`，而是输出一个 `PP_NmState` 信号，让其他模块根据状态去请求 ComM。**

工程里 `NmUT.arxml` 已经给 `NmUT` 配了 ComM Client/Server 端口：

- `UR_ComMUser_Can_Network_ETAS_Channel`
- 操作包括 `GetCurrentComMode`、`GetMaxComMode`、`GetRequestedComMode`、`RequestComMode`

但当前 `NmUT.c` 中实际没直接调用这些 RTE Client/Server API。

**这里是一个关键工程判断：如果你在调试时只改 `Nm_Test[0].request_b = TRUE`，但没有模块消费 `APP_READY_FOR_NM_REQUEST` 并真正调用 ComM 请求，那么 NM 不一定会起来。`NmUT` 更像“测试/演示状态信号 SWC”，不是完整的 NM 请求控制器。**


#### 4.2.1 Client/Server 端口的配置是怎么实现的呢


**ComM 的服务端 Port 通常由工具根据 ComM 配置自动生成，不是你在 NmUT 里手动创建的。**

更准确地说：

```
ComM 模块配置了 ComMUser / ComMChannel
        |
        v
工具生成 ComM 的 SWCD 服务描述
        |
        v
出现 ComM 的 P-Port，例如：
UR_ComMUser_Can_Network_Channel_Can_Network
modeLimitation
        |
        v
应用 SWC 再创建 R-Port 去连接这些 P-Port
```

所以分两边看：

| 位置                            | 是否需要你配                                     |
| ----------------------------- | ------------------------------------------ |
| ComM 的服务端 P-Port              | 一般由 ComM 配置/工具生成                           |
| NmUT 的客户端 R-Port              | 需要在 SWC 里配置                                |
| NmUT R-Port 和 ComM P-Port 的连接 | 需要在 System/Composition 里连接，很多工具可自动创建或半自动创建 |
| Runnable 里的 Server Call Point | 需要配置，否则 RTE 不知道这个 runnable 会调用哪个服务操作       |
|                               |                                            |


是的，截图里这些 `/AUTOSAR_ComM/PortInterfaces/...` 基本可以理解为 **AUTOSAR/工具提供的标准 ComM 服务接口模板**，不是你针对 `NmUT` 手写出来的。

但要区分三个概念：

|名称|是不是默认/标准|作用|
|---|---|---|
|`ComM_UserRequest`|是，ComM 标准服务接口|给 SWC 请求通信模式，比如 `RequestComMode()`|
|`ComM_ECUModeLimitation`|是，ComM 标准服务接口|限制 ECU 通信模式、读/清 inhibit counter|
|`ComM_ChannelLimitation`|是，ComM 标准服务接口|限制某个 channel 的通信能力|
|`ComM_CurrentMode`|是，ComM 标准接口|通知/读取当前通信模式|
|`ComM_ChannelWakeup`|是，ComM 标准接口|和通道唤醒相关|
|`DcmCommunicationControl_...`|Dcm 相关标准/生成接口|诊断通信控制用，不是 NmUT 常规请求网络用的|

你截图里的这个窗口是在选 **PortInterface**，也就是“接口类型”。  
它还不是某一个具体端口实例。

可以这样理解：

```
PortInterface = 接口模板/函数菜单
Port = 某个 SWC 上真正创建出来的插座
Connector = 把两个插座连起来的线
RTE = 根据这些生成代码
```

比如你选了：

```
ComM_UserRequest
```

它只是说明这个端口支持这些操作：

```
RequestComMode()
GetCurrentComMode()
GetMaxComMode()
GetRequestedComMode()
```

然后在 `NmUT` 里创建 R-Port：

```
UR_ComMUser_Can_Network_ETAS_Channel
Interface = ComM_UserRequest
```

ComM 那边也有对应 P-Port：

```
UR_ComMUser_Can_Network_Channel_Can_Network
Interface = ComM_UserRequest
```

最后 System 里把它们连起来。

所以结论是：

**PortInterface 是默认/标准已有的；但具体端口 R-Port、P-Port，以及二者连接，不完全是“默认自动就有”，需要根据 ComM User/Channel 和 SWC 使用需求生成或配置。**






### 4.3 ComM 到 CanNm 的典型路径

典型路径如下：

```text
NmUT(swc)为了让 `ExeMgrUT` 请求 `COMM_NO_COMM
->
应用/ExeMgrUT
  -> ComM_RequestComMode(User, COMM_FULL_COMMUNICATION)
  -> ComM_MainFunction(ComM)
  -> CanSM_RequestComMode(channel, COMM_FULL_COMMUNICATION)
  -> **CanSM 拉起 CAN Controller / CanIf / Transceiver**
  -> Nm_NetworkRequest(channel)
  -> CanNm_NetworkRequest(channel)
  -> CanNm 进入 Repeat Message / Normal Operation
  -> 周期发送 NM Tx PDU 0x600
```

从有道云笔记补充的完整视角看，主动唤醒时还要把硬件链路串起来：

```text
本地唤醒源/KL15/诊断/传感器
  -> EcuM/BswM/应用确认唤醒有效
  -> ComM User 请求 FullCom
  -> CanSM 请求 CanIf/CanTrcv/Can Driver 工作
  -> Transceiver 进入 Normal
  -> CAN Controller 进入 STARTED
  -> Com/PduR/CanIf 打开对应 PDU Group 和 PDU Mode
  -> Nm_NetworkRequest
  -> CanNm_NetworkRequest
  -> CanNm 进入 Network Mode / Repeat Message State
```

所以排查“为什么 FullCom 了但没有报文”时，不能只看 CanNm。要同时看 CanSM 是否把 Controller/Transceiver 拉到可收发状态，以及 CanIf PDU Mode 是否允许 Tx/Rx。

释放网络时：

```text
应用/ExeMgrUT
  -> ComM_RequestComMode(User, COMM_NO_COMMUNICATION)
  -> ComM 进入 Ready Sleep
  -> Nm_NetworkRelease(channel)
  -> CanNm_NetworkRelease(channel)
  -> CanNm 等待 NM Timeout / Wait Bus Sleep
  -> Prepare Bus Sleep
  -> Bus Sleep
  -> ComM/CanSM 切到 NO_COMMUNICATION
```

### 4.4 NM 接收回调

`Nm_Cfg.c` 把用户回调挂到了 NM Interface：

```c
const Nm_UserCallbackType_tst Nm_UserCallbacks = {
    &Nm_RxIndicationCallback,
    &TestNm_RemoteSleepCancellation,
    &TestNm_RemoteSleepIndication,
    &Nm_StateChangeIndication
};
```

收到 NM PDU 后，`Nm_RxIndicationCallback()` 会设置标志：

```c
Nm_Test[NetworkHandle].IsNmReceived = TRUE;
```

`RE_NmUT_func()` 下一个 10ms 周期看到该标志后调用：

```c
Nm_GetUserData(index_chanel, Nm_Test[index_chanel].nm_rx_data);
```

这说明本工程支持通过 NM 报文携带 user data，并在应用侧读取。

### 4.5 NM 状态变化回调

`Nm_StateChangeIndication()` 会记录前后状态：

```c
Nm_Test[NetworkHandle].nm_PreviousState = nmPreviousState;
Nm_Test[NetworkHandle].nm_CurrentState = nmCurrentState;
```

并对几个状态做处理：

- `PREPARE_BUS_SLEEP -> BUS_SLEEP`
  - `shutdown_b = 1`
  - 写 `APP_NM_BUS_SLEEP`
- `NM_STATE_REPEAT_MESSAGE`
  - 写 NM user data 信号 `0xAA`
- `NM_STATE_PREPARE_BUS_SLEEP`
  - 写 NM user data 信号 `0xBB`
- `NM_STATE_NORMAL_OPERATION`
  - 写 NM user data 信号 `0xCC`

注意：代码里 `Nm_SetUserData()` 被注释掉了，实际发送 user data 走的是：

```c
Rte_Write_NmUT_PP_NM_Tx_Sig_NM_Tx_Sig(...)
```

这符合当前配置里的 `CanNmComUserDataSupport = true`：NM user data 通过 COM/PduR 路径送给 CanNm，而不是直接调用 `Nm_SetUserData()`。

### 4.6 Repeat Message 触发

`Trggr_RepeatState` 为真时，应用直接调用：

```c
Nm_RepeatMessageRequest(0);
```

当前只有 1 个 NM 通道，所以 channel handle `0` 是合理的。但如果以后增加多通道，这里硬编码 `0` 就会成为问题。

## 6. 常见工程问题

### 6.1 请求 FullCom 后不发 NM 报文

优先检查：

1. 应用是否真的调用了 `ComM_RequestComMode(User, COMM_FULL_COMMUNICATION)`。
2. `ComMUser` 是否绑定到正确的 `ComMChannel`。
3. `ComMChannel` 的 `ComMNmVariant` 是否是 `FULL`。
4. `NmChannelConfig` 是否引用了同一个 `ComMChannel`。
5. `CanNmTxPduRef` 是否连接到 CanIf Tx PDU。
6. CanSM 是否成功把 CAN controller 切到 started/online。
7. CanIf PDU mode 是否允许 Tx。

本工程里 `NmUT` 只是写 `PP_NmState`，没有直接调用 ComM。如果 ExeMgrUT 或其他逻辑没消费 `APP_READY_FOR_NM_REQUEST`，网络不会起来。

### 6.2 收到 NM 报文但 user data 读不到

检查：

- `NmUserDataEnabled = true`
- `CanNmUserDataEnabled = true`
- `CanNmComUserDataSupport = true`
- PduR/Com 的 NM UserData PDU 是否配置完整。
- Rx PDU 长度是否为 8 字节。
- NID/CBV 占用 byte0/byte1 后，真正 user data 从后续字节开始，不要把 byte0/byte1 当业务 user data。

### 6.3 网络不睡眠

常见原因：

- 某个 ComM User 一直请求 `COMM_FULL_COMMUNICATION`。
- CanNm 一直收到对端 NM 报文，所以认为网络仍活跃。
- CanNm `RepeatMessageTime`、`TimeoutTime`、`WaitBusSleepTime` 配得过长。
- Dcm ActiveDiagnostic 或通信控制把网络保持在 FullCom。
- `Nm_NetworkRelease()` 没被触发。

### 6.4 网络很快睡了

常见原因：

- 应用没有持续保持 FullCom 请求。
- `ComMTMinFullComModeDuration` 太短。
- `CanNmTimeoutTime` 太短。
- 对端没有发 NM，当前 ECU 又释放了网络。
- ComM 用户、通道、Nm 通道引用错了，导致请求没有作用到目标 CAN 网络。

### 6.5 Repeat Message 无效

`Nm_RepeatMessageRequest()` 不是任何状态都有效。CanNm 代码注释中说明它主要在 Network Mode 下、且不是 Repeat Message 状态时才有意义。常见问题：

- 当前还在 Bus Sleep 或 Prepare Bus Sleep。
- `NodeDetectionEnabled` 没开。
- 通道号写死为 0，多通道后请求错通道。

### 6.6 PNC 配置误解

本工程 ComM 有 `ComMPnc_0`，但 CanNm 侧：

- `CanNmGlobalPnSupport = false`
- `CanNmPnEnabled = false`
- `CANNM_GLOBAL_PN_SUPPORT = STD_OFF`

所以当前不能按“完整 PN 过滤网络唤醒”去理解 CanNm。ComM 的 PNC 配置存在，不代表 CanNm 的 Partial Networking 已启用。

## 7. 易错点清单

- 把 `Nm` 和 `CanNm` 混为一谈：`Nm` 是接口层，`CanNm` 是 CAN 实现。
- 应用直接调用 `CanNm_*`：工程里更推荐通过 `Nm_*` 或 ComM 路径，避免绕过抽象层。
- 以为 `NmUT` 已经请求了 ComM：当前代码只是写 `PP_NmState`，真正请求要看 ExeMgrUT/其他逻辑。
- 忘记 10ms 主函数：Nm、CanNm、ComM 主函数周期必须被 OS/RTE 调度，否则状态机不走。
- NID/CBV 位置用错：当前 byte0 是 NodeId，byte1 是 CBV。
- user data 写错路径：当前 `Nm_SetUserData()` 被注释，实际走 `Rte_Write_NmUT_PP_NM_Tx_Sig_NM_Tx_Sig()`。
- 硬编码 channel 0：当前可用，但扩展多网络会出问题。
- 只看 ComM FullCom，不看 CanSM/CanIf：FullCom 请求成功不等于 CAN 控制器已经 online。
- 忽略 DCM：诊断会通过 ComM 保持通信，导致网络不睡。
- 把 ComM PNC 当作 CanNm PN：当前 CanNm PN 是关闭的。

## 10. 有道云笔记补充：状态机和报文行为

CanNm 有 3 个大模式：

| 大模式 | 直白解释 | 报文行为 |
|---|---|---|
| `Bus Sleep Mode` | 网络睡着，等唤醒 | 通常不发 NM、不发应用；硬件可保留唤醒能力 |
| `Network Mode` | 网络醒着 | 细分为 RMS/NOS/RSS |
| `Prepare Bus Sleep Mode` | 准备睡，给大家停报文的缓冲时间 | 通常不发 NM、不发应用，但可接收唤醒相关报文 |

Network Mode 里又有 3 个状态：

| 状态 | 直白解释 | 典型行为 |
|---|---|---|
| `Repeat Message State` | 刚唤醒或被要求重新同步，先反复发 NM | NM 和应用报文都可以收发 |
| `Normal Operation State` | 本 ECU 仍主动需要网络 | 周期发送 NM，应用报文正常 |
| `Ready Sleep State` | 本 ECU 不主动需要网络，但网络还没完全睡 | 停发本节点 NM，应用报文仍可按需求收发，等待 Timeout |

常见跳转逻辑：

- `Bus Sleep / Prepare Bus Sleep -> Repeat Message`：收到有效 NM 报文、被动唤醒，或本地主动请求网络。
- `Repeat Message -> Normal Operation`：Repeat 定时器到期，并且本节点仍有主动网络请求。
- `Repeat Message -> Ready Sleep`：Repeat 定时器到期，本节点没有主动网络请求。
- `Normal Operation -> Ready Sleep`：本节点释放网络请求。
- `Ready Sleep -> Prepare Bus Sleep`：`CanNmTimeoutTime` 到期，期间没有足够的 NM 活动维持网络。
- `Prepare Bus Sleep -> Bus Sleep`：`CanNmWaitBusSleepTime` 到期，期间没有新唤醒。
- `Normal Operation / Ready Sleep -> Repeat Message`：收到 RMR bit 为 1 的 NM 报文，或某些 PN/多网络请求场景要求重新同步。

本工程配置对应关系：

| 状态机参数 | 当前工程值 | 工程含义 |
|---|---:|---|
| `CanNmRepeatMessageTime` | `3.0 s` | 进入 RMS 后至少待 3 秒，再根据是否有本地请求转 NOS/RSS |
| `CanNmTimeoutTime` | `2.0 s` | RSS/NOS 下用于判断网络是否还能保持 |
| `CanNmWaitBusSleepTime` | `4.0 s` | PBSM 下等待总线真正安静 |
| `CanNmMsgCycleTime` | `1.0 s` | 正常 NM 周期是 1 秒 |
| `CanNmImmediateNmTransmissions` | `0` | 当前没有配置快发 NM |

易错理解：

- `RepeatMessageTime` 不是“收到每帧 NM 都重置”的定时器，它主要约束停留在 RMS 的时间。
- `CanNmTimeoutTime` 会因为发送/接收 NM 或超时等条件被刷新，是判断网络是否还活跃的重要定时器。
- RSS 不等于总线已经睡了。RSS 下应用报文可能仍然能发，直到后续进入 PBSM/BSM。

## 11. 有道云笔记补充：NM 报文、CBV 和当前工程 0x600/0x677

典型 CAN NM PDU 是 8 字节：

| Byte | 常见含义 | 当前工程配置 |
|---|---|---|
| Byte0 | Source Node Identifier，源节点 ID | `CanNmPduNidPosition = CANNM_PDU_BYTE_0` |
| Byte1 | CBV，Control Bit Vector | `CanNmPduCbvPosition = CANNM_PDU_BYTE_1` |
| Byte2-Byte7 | User Data 或 PN 信息 | 当前 `CanNmComUserDataSupport = true` |

CBV 常用 bit：

| Bit | 名称 | 直白解释 |
|---:|---|---|
| 0 | RMR, Repeat Message Request | 请求同网段节点进入/回到 Repeat Message State |
| 3 | NM Coordinator Sleep Bit | 协调同步休眠相关 |
| 4 | AWB, Active Wakeup Bit | 标识本节点是否主动唤醒网络 |
| 6 | PNI, Partial Network Information | 标识 NM 报文里是否带 PN/PNC 信息 |

结合当前工程：

- Tx NM PDU 是 `NM_Tx_0x600_Can_Network_Channel`。
- Rx NM PDU 是 `NM_Rx_0x677_Can_Network_Channel`。
- 当前 NodeId 配置为 `0`，Byte0 会放 Node ID。
- 当前启用了 Active Wakeup Bit，但没有启用 PN。
- 当前 `NmUT` 在状态变化时写 `PP_NM_Tx_Sig` 为 `0xAA/0xBB/0xCC`，这属于工程测试/演示性质的 user data 写入。
- 因为 `Nm_SetUserData()` 被注释，当前 user data 主要走 COM 信号到 PduR/CanNm 的路径。

RMR 的常见误区：

- 不是所有节点收到 RMR=1 后都要把自己的 RMR 也置 1。发起请求的节点置位，其他节点是被动进入 RMS。
- 调 `CanNm_RepeatMessageRequest()` 通常要求当前处于 NOS 或 RSS，并且 `CanNmNodeDetectionEnabled = true`。
- 本工程 `Trggr_RepeatState` 触发 `Nm_RepeatMessageRequest(0)`，通道 0 当前合理，但多通道后必须改成明确的 channel 映射。

AWB 的常见误区：

- 主动唤醒网络时，AWB 应该置 1。
- 被动收到其他节点 NM 报文唤醒时，AWB 通常为 0。
- 当前工程 `CanNmActiveWakeupBitEnabled = true`，所以主动/被动唤醒测试时可以观察 CBV bit4 是否符合预期。

PNI 的常见误区：

- PNI=1 说明 NM PDU 带 PN 信息，不是“所有 NM 都必须置 1”。
- 当前工程 CanNm PN 关闭，所以 PNI/PNC 过滤不要作为当前项目唤醒主线分析。

## 12. 有道云笔记补充：一帧/两帧 NM 唤醒问题

工程里经常会遇到：“为什么我发了一帧 NM，DUT 没真正进入网络状态，要第二帧才行？”

直白解释：

```text
第一帧 NM 到达总线
  -> 可能只把 Transceiver/SBC/MCU 从低功耗里叫醒
  -> MCU 开始上电/复位/初始化
  -> CanTrcv 进入 Normal
  -> CAN Controller 初始化并 STARTED
  -> CanIf/CanNm 初始化完成
  -> 这时软件才有能力真正接收并处理后续 CAN/NM 帧
```

所以第一帧可能只是“硬件唤醒帧”，不是“被 CanNm 软件处理到的帧”。如果硬件没有 PN 过滤或没有把唤醒原因直接验证成有效 NM，那么需要第二帧或第三帧让软件栈真正收到，才会进入 `CanNm_PassiveStartup()` / `Nm_NetworkStartIndication()` 等链路。

当前工程相关点：

- 工程里有自研/集成的 `CANTrcv_Tja1145`，TJA1145 是支持复杂唤醒/PN 能力的典型收发器，但是否启用硬件 PN 过滤要看 CanTrcv 具体配置。
- 当前 CanNm 配置没有 PN，所以软件层面不能按 PN 过滤唤醒来分析。
- 如果测试“0x600/0x677 一帧唤醒”，要同时确认硬件唤醒源、EcuM wakeup validation、CanSM wakeup source、CanIf validation 是否完整。

结论：

- 主动唤醒节点通常要求快速发多帧 NM，目的是让同网段节点都能收到。
- 被动唤醒节点不一定能被第一帧软件处理到，特别是冷启动/低功耗场景。
- 如果需求强制“一帧 NM 唤醒”，通常需要硬件过滤、唤醒源验证、EcuM/CanIf/CanTrcv 配置配合，而不是只改 CanNm。

## 13. 有道云笔记补充：第一帧必须是 NM 吗

很多 OEM 会要求网络唤醒后，本节点第一帧外发报文必须是 NM。原因很简单：先让同网段节点同步进入网络状态，再发应用报文，网络行为更稳定。

通用建议：

- 优先通过 AUTOSAR 工具配置解决，例如 NM 发送优先级、Com I-PDU Group 启动顺序、CanSM/BswM ActionList 顺序、Immediate NM 发送配置。
- 不建议轻易改 Can Driver/CanIf 静态代码去拦截应用报文。
- 如果不得不临时做工程兜底，通常思路是在 Repeat Message State 下，只允许 NM CAN ID 先通过，首帧 NM 发出后再放开应用报文。但这属于高风险改法，需要完整验证。

当前工程要特别注意：

- `CanNmImmediateNmTransmissions = 0`，没有快发 NM。
- 应用报文和 NM 报文谁先发，可能取决于 Com/CanIf/PduR/CanSM/BswM 的启动顺序和周期任务调度。
- 如果测试中发现上电后第一帧不是 `0x600`，不要先改驱动，先看 ISOLAR 里的 CanNm immediate、Com I-PDU Group、BswM/CanSM 通信打开时序。

## 14. 有道云笔记补充：诊断报文能不能唤醒网络

结论要分两层：

- 诊断报文能不能唤醒 ECU：取决于硬件收发器/SBC 是否会被这类 CAN 帧唤醒，以及 EcuM 是否把它作为有效唤醒源验证。
- 诊断报文能不能让 NM 网络进入 Network Mode：软件上通常不应把诊断帧当 NM 帧去走 `CanNm_PassiveStartup()`，而是由 DCM/ComM 作为 User 请求 FullCom，最终走 `CanNm_NetworkRequest()`。

典型诊断保持网络流程：

```text
Dcm 收到诊断请求
  -> ComM_DCM_ActiveDiagnostic(NetworkId)
  -> ComM 请求 Full Communication
  -> CanSM/CanIf/CanTrcv 打开通信
  -> Nm_NetworkRequest / CanNm_NetworkRequest
  -> 网络进入 RMS/NOS
```

释放流程：

```text
诊断响应发送完成，且默认会话/无需继续保持
  -> Dcm_TpTxConfirmation
  -> ComM_DCM_InactiveDiagnostic(NetworkId)
  -> ComM 释放通信请求
  -> Nm_NetworkRelease / CanNm_NetworkRelease
```

当前工程里有 Dcm 和 ComM 交互代码，例如：

- `BasicSoftware\src\bsw\Dcm\src\Dsl\Dcm_Dsl_ComMInteraction.c`
- `BasicSoftware\src\bsw\ComM\src\ComM_Dcm_ActiveDiagnostic.c`
- `BasicSoftware\src\bsw\ComM\src\ComM_Dcm_InactiveDiagnostic.c`

所以遇到“网络不睡”的问题时，要检查诊断是否还在非默认会话、是否还在 NRC 0x78、是否 DCM 没有释放 ComM。

## 15. 有道云笔记补充：Transceiver/Controller/EcuM 的关系

NM 是软件状态机，但它必须依赖硬件链路真正可用：

```text
CAN 总线活动
  -> Transceiver 检测唤醒
  -> INH/SBC/电源管理唤醒 MCU
  -> EcuM 记录 Pending Wakeup
  -> EcuM_StartWakeupSources
  -> CanSM_StartWakeupSource
  -> CanTrcv Normal
  -> Can Controller STARTED
  -> CanIf_CheckValidation / EcuM_ValidateWakeupEvent
  -> ComM_EcuM_WakeUpIndication / BswM_EcuM_CurrentWakeup
```

排查顺序建议：

1. 先确认 MCU 是否真的被唤醒/上电。
2. 再确认 CanTrcv 是否 Normal。
3. 再确认 Controller 是否 STARTED。
4. 再确认 CanIf 是否允许 Rx/Tx。
5. 最后再看 CanNm 状态机。

不要把“ECU 上电”直接等同于“网络管理已进入 Network Mode”。ECU 可以已经上电，但 CanNm 仍在 Bus Sleep，因为唤醒事件还没验证或通信还没被允许。

## 16. 有道云笔记补充：PN/PNC 和当前工程差异

PNC 可以理解成“只叫醒一部分节点的小群”。PNI bit 表示 NM 报文里带 PN 信息，PN 信息通过掩码过滤判断当前节点是否相关。

典型 PN 判断：

```text
收到 NM PDU
  -> 看 CBV bit6 PNI 是否为 1
  -> 找 PN Info 区域
  -> 用本节点配置的 PnFilterMask 做按位与
  -> 结果非 0：相关 PNC，请求/保持唤醒
  -> 结果为 0：不相关，可以忽略该 PN 请求
```

当前工程虽然 ComM 配了 `ComMPnc_0`，但 CanNm PN 关闭：

- `CanNmGlobalPnSupport = false`
- `CanNmPnEnabled = false`
- `CANNM_GLOBAL_PN_SUPPORT = STD_OFF`
- `CANNM_PN_ENABLED = STD_OFF`

所以当前工程不能按“PNC 硬件/软件过滤网络唤醒”去解释。`ComMPnc_0` 只能说明 ComM 配置中存在 PNC 容器，不能说明 CanNm 已经使用 PN 功能。

## 17. 按当前工程做 NM 调试的推荐路径

### 17.1 主动唤醒测试

目标：确认本 ECU 主动请求网络后是否发 `0x600`。

检查顺序：

1. 触发应用请求，例如让 `Nm_Test[0].request_b = TRUE`。
2. 确认 `PP_NmState` 输出 `APP_READY_FOR_NM_REQUEST`。
3. 确认 ExeMgrUT 或其他逻辑真正调用 ComM 请求 FullCom。
4. 断点 `ComM_RequestComMode`，确认 User 是 `ComMUser_Can_Network_Channel_Can_Network`。
5. 断点 `CanSM_RequestComMode`，确认请求 `COMM_FULL_COMMUNICATION`。
6. 断点 `Nm_NetworkRequest` / `CanNm_NetworkRequest`。
7. CANoe/CANalyzer 观察 `0x600`。
8. 观察 `Nm_StateChangeIndication()` 是否进入 `NM_STATE_REPEAT_MESSAGE` / `NM_STATE_NORMAL_OPERATION`。

### 17.2 被动唤醒测试

目标：确认收到外部 NM 后本 ECU 状态是否起来。

检查顺序：

1. 外部发送有效 NM，例如当前工程期望 Rx PDU `0x677`。
2. 确认 CanTrcv/CanIf/Can Controller 已经能接收。
3. 断点 `CanNm_RxIndication`。
4. 断点 `Nm_RxIndicationCallback`，确认 `IsNmReceived = TRUE`。
5. 断点 `Nm_GetUserData()`，确认 user data 被读取。
6. 观察 `Nm_StateChangeIndication()` 状态变化。
7. 如果第一帧没触发软件链路，连续发送多帧 NM 验证是否是冷启动/初始化窗口问题。

### 17.3 释放与睡眠测试

目标：确认释放网络后能进入 Bus Sleep。

检查顺序：

1. 触发 `Nm_Test[0].release_b = TRUE`。
2. 确认 `APP_READY_FOR_NM_RELEASE` 被消费。
3. 断点 `ComM_RequestComMode(... COMM_NO_COMMUNICATION)`。
4. 断点 `Nm_NetworkRelease` / `CanNm_NetworkRelease`。
5. 等待 `CanNmTimeoutTime = 2s`、`CanNmWaitBusSleepTime = 4s`。
6. 观察 `Nm_StateChangeIndication()` 是否出现 `PREPARE_BUS_SLEEP -> BUS_SLEEP`。
7. 确认 `shutdown_b = 1` 和 `APP_NM_BUS_SLEEP` 输出。

如果不睡：

- 查 ComM 是否还有 User 请求。
- 查 DCM 是否还 ActiveDiagnostic。
- 查外部是否仍周期发送 `0x677`。
- 查 CanNm 是否一直收到 NM 导致 Timeout 被刷新。
- 查 BswM/ComM 是否没有允许进入 NoCom。

| `Nm_ChannelData_s[x].ChannelState_en`                   | `Nm_NetworkRamType_tst`                                               | Nm 内部“通道状态”。常见值可理解为初始化、正常、等待睡眠、总线睡眠。                               | 请求网络后应从初始化/睡眠方向进入正常；释放网络后应向等待睡眠/睡眠方向走。                                            |
| `Nm_ChannelData_s[x].ChannelMode_en`                    | `Nm_NetworkRamType_tst`                                               | Nm 当前“模式”。它更接近 AUTOSAR 对外说的 BusSleep、PrepareBusSleep、Network 这类模式。 | 如果 CAN 上已经有 NM 报文，但 ComM 没收到网络模式通知，要看这个值是否被回调更新。                                  |
| `Nm_ChannelData_s[x].ComMRequest_b`                     | `Nm_NetworkRamType_tst`                                               | ComM 或上层是否请求 Nm 保持网络。`TRUE` 表示有请求，`FALSE` 表示释放。                    | 网络不睡眠时重点看它是不是一直为 `TRUE`；网络不起时看它是否根本没置位。                                           |
| `Nm_ChannelData_s[x].RemoteSleepStatus_b`               | `Nm_NetworkRamType_tst`                                               | 远端节点是否看起来准备睡眠。                                                     | 本地释放后仍不睡，可能是远端一直发 NM，也可能这个状态一直没满足。                                                |
| `Nm_ChannelData_s[x].ShutdownDelayTimeStamp_u32`        | `Nm_NetworkRamType_tst`                                               | Nm 协调睡眠用的时间戳。                                                      | 怀疑睡眠延迟、等待时间不对时看它和系统 tick 的差值。                                                     |
| `Nm_GlobalData_s[]`                                     | `BasicSoftware/src/bsw/Nm/Nm_Cfg.c`                                   | Nm 的簇级运行时状态。多个 Nm channel 属于同一 cluster 时会用到。                       | 本工程主要一个 CAN 网络，但调试 coordinator/cluster 行为时要看它。                                    |
| `CanNm_ChannelConfigData_cs[]`                          | `BasicSoftware/src/bsw/CanNm_PreCompile_and_PB_Variant/CanNm_PBcfg.c` | CanNm 每个 CAN NM 通道的静态配置。报文周期、超时、NodeId、PDU、CBV 位置等都从这里来。           | 现场怀疑“配置没生效”时，看生成代码里的实际值，不只看 ISOLAR 界面。                                            |
| `CanNm_GlobalConfigData_cs`                             | `CanNm_PBcfg.c`                                                       | CanNm 全局配置入口，里面挂着通道配置表。                                            | 如果初始化异常或通道数不对，先确认这个全局配置是否被 `CanNm_Init()` 使用。                                     |
| `CanNm_Channel_Mapping_Table___KW_COMMON[]`             | `CanNm_PBcfg.c`                                                       | 把外部 PDU/通道映射到 CanNm 内部通道索引。                                        | 收到 0x677 但没有进正确通道时，查这张映射。                                                         |
| `CanNmMsgCycleTime` / 生成值 `1000000`                     | ARXML / `CanNm_PBcfg.c`                                               | NM 正常报文周期。本工程配置为 1s，生成代码里是微秒级数值。                                   | 抓 CANoe 时，0x600 周期不接近 1s，就要查调度、状态或配置。                                             |
| `CanNmTimeoutTime` / 生成值 `3000000`                      | ARXML / `CanNm_PBcfg.c`                                               | 等待远端 NM 报文的超时时间。本工程 3s。                                            | 远端停发后，本地不会立刻睡，要等相关超时和状态机走完。                                                       |
| `CanNmRepeatMessageTime` / 生成值 `1000000`                | ARXML / `CanNm_PBcfg.c`                                               | Repeat Message 状态持续时间。本工程 1s。                                      | 刚唤醒或 Repeat 请求后，会先经历这段更积极的 NM 行为。                                                 |
| `CanNmWaitBusSleepTime` / 生成值 `4000000`                 | ARXML / `CanNm_PBcfg.c`                                               | 总线安静后进入 BusSleep 前的等待时间。本工程 4s。                                    | “为什么释放后还没睡”通常先看这个时间，再看是否还有报文。                                                     |
| `NM_Tx_0x600_Can_Network_Channel`                       | CanNm Tx PDU 配置                                                       | 本 ECU 发送的 NM 报文 PDU，对应 CAN ID 0x600。                               | 主动请求网络后，用 CANoe 看 0x600 是否出现。                                                     |
| `NM_Rx_0x677_Can_Network_Channel`                       | CanNm Rx PDU 配置                                                       | 本 ECU 接收远端 NM 报文 PDU，对应 CAN ID 0x677。                              | 被动唤醒或远端保持网络时，看 0x677 是否进入 CanIf/CanNm。                                            |
| `NmUT_Test[0].request_b` / `release_b`                  | `ASW/SWC/NmUT/src/NmUT.c`                                             | NmUT 的测试开关，用来触发应用层“请求网络/释放网络”的意图。                                  | 注意它本身不等于已经调用 `ComM_RequestComMode()`，还要看后续谁消费 `APP_READY_FOR_NM_REQUEST/RELEASE`。 |
| `APP_READY_FOR_NM_REQUEST` / `APP_READY_FOR_NM_RELEASE` | NmUT 输出到应用/执行管理逻辑                                                     | 应用侧告诉系统“我要网络”或“我可以释放网络”的状态。                                        | 如果 NmUT 变量变了但 ComM 没动作，重点查这个状态有没有被 ExeMgrUT 或其他逻辑消费。                              |
| `Nm_RxIndicationCallback()`                             | Nm 回调配置                                                               | 收到 NM PDU 后的应用回调。                                                  | 用来确认“报文进 BSW 了，而且 Nm 把接收事件通知到了应用”。                                                |
| `Nm_StateChangeIndication()`                            | Nm 回调配置                                                               | Nm 状态变化通知。                                                         | 调试状态机最直观，建议在这里打断点或打印 old/new state。                                               |

最常用的判断顺序是：

1. 看 `NmUT_Test[0].request_b/release_b` 是否被置位。
2. 看是否真的调用到 `ComM_RequestComMode()`，再进入 `Nm_NetworkRequest()` 或 `Nm_NetworkRelease()`。
3. 看 `Nm_ChannelData_s[x].ComMRequest_b` 和 `ChannelState_en` 是否变化。
4. 看 `CanNm_ChannelConfigData_cs[]` 中 0x600/0x677、周期、超时是否和 ISOLAR 一致。
5. 最后用 CANoe 看 0x600 是否按周期发、0x677 是否能被收到。

## 19. 结论

对初学者最重要的是把 NM 拆成 4 条线：

1. 应用线：谁要通信，谁释放通信。对应 ComM User。
2. NM 状态线：RMS/NOS/RSS/PBSM/BSM 怎么跳。
3. 报文线：NM PDU 的 NID、CBV、UserData 怎么发、怎么收。
4. 硬件线：Transceiver、Controller、EcuM wakeup validation 是否真的让 CAN 可收发。

当前工程的 NM 主线是：

```text
NmUT/ExeMgrUT 发起请求
  -> ComM User 请求 FullCom/NoCom
  -> CanSM 控制 CAN 硬件通信状态
  -> Nm Interface 转调 CanNm
  -> CanNm 通过 0x600/0x677 管理网络
  -> NmUT 通过回调观察状态和 user data
```

当前工程不是 PN 主线，也不是 Passive Node 主线。调试时要重点盯住 `ComM_RequestComMode`、`CanSM_RequestComMode`、`Nm_NetworkRequest/Release`、`CanNm_RxIndication`、`Nm_StateChangeIndication`、`0x600/0x677` 这几处。
