# BswM

BswM 就是一个**模式仲裁器**：  
**接收各模块的模式输入 MRP -> 判断条件/规则 -> 执行动作列表 -> 切换系统状态或调用 BSW API。**

EcuM/CanIf/ComM/SWC 等模块上报模式
        |
        v
BswM_MRP_xxx 保存这些模式输入
        |
        v
BswM_MC_xxx 判断某个输入是否满足条件
        |
        v
BswM_LE_xxx 把多个条件 AND/OR 组合
        |
        v
BswM_Rule 判断是否触发
        |
        v
BswM_ActionList 执行动作，比如切换 BSWM_MDG、请求 ComM、通知 RTE

## BswM Mode Condition

```
BswM_MC_AppModeReq_AppRun
```

含义是：

```
判断 BswM_MRP_SWC_AppMode 当前是否等于 APP_RUN
```

右边几个字段这样看：

```
ShortName = BswM_MC_AppModeReq_AppRun
```

只是这个条件的名字。

```
BswMConditionType = BSWM_EQUALS
```

表示判断方式是“等于”。

```
BswMConditionMode = BswM_MRP_SWC_AppMode
```

表示它判断的是哪一路 MRP，也就是：

```
BswM_Cfg_GenericReqModeInfo_ast[BSWM_IDX_BSWM_MRP_SWC_APPMODE].dataMode_u16
```

不过你截图里还没展开下面的具体值，通常它还会有一个 `BswMConditionValue` 或类似字段，对应：

```
APP_RUN
```

生成代码里大概就是这种判断：

```
RTE_MODE_MDG_ECUM_STATE_ECUM_STATE_APP_RUN
==
BswM_Cfg_GenericReqModeInfo_ast[BSWM_IDX_BSWM_MRP_SWC_APPMODE].dataMode_u16
```

也就是：

```
SWC_AppMode == APP_RUN ?
```

它本身不会设置模式，只是一个判断条件。后面会被 `BswMLogicalExpression` 引用，比如：

```
AppModeReq_AppRun
AND
BswM_MDG == POST_RUN
```

之类的组合条件。组合条件成立后，`BswMRule` 才会触发 `ActionList`，去真正执行模式切换、ComM 请求、EcuM 动作等。

所以这类对象的层级是：

```
BswMModeRequestPort
    保存当前模式值

BswMModeCondition
    判断某个 MRP 当前是否等于某个模式

BswMLogicalExpression
    把多个 ModeCondition 用 AND/OR 组合

BswMRule
    根据 LogicalExpression 决定是否执行

BswMActionList
    真正做动作
```

你这个 `BswM_MC_AppModeReq_AppRun` 就是：**判断 ExeMgr/SWC 有没有请求 APP_RUN。**

## **BswMRule**
`LogicalExpression` 和 `BswMActionList` 是通过 **BswMRule** 对应起来的。

结构是：

```
BswMRule
    |
    +-- LogicalExpression
    |
    +-- TrueActionList
    |
    +-- FalseActionList
```

也就是说：**LogicalExpression 本身不直接绑定 ActionList，中间一定有一个 Rule。**

你工程里生成代码在这里：

[BasicSoftware/src/bsw/BswM_PreCompile_and_PB_Variant/BswM_PBcfg.c (line 115)](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/src/bsw/BswM_PreCompile_and_PB_Variant/BswM_PBcfg.c:115)

比如这个规则：

```
{       /* Rule Structure for the configuration container BswM_AR_AppRun2PostRun */
    BSWM_FALSE,
    &BswM_Cfg_LE_BswM_LE_AppRun2PostRun,
    &BswM_Cfg_AL_cast[3],    /* trueActionList */
    NULL_PTR
},
```

这就表示：

```
Rule = BswM_AR_AppRun2PostRun
LogicalExpression = BswM_LE_AppRun2PostRun
TrueActionList = BswM_Cfg_AL_cast[3]
FalseActionList = NULL
```

再去下面查 `BswM_Cfg_AL_cast[3]`，在同一个文件里：

```
/* ActionListName : BswM_AL_BswMSwitchPostRun */
...
3, /* Unique Number for ActionList */
```

所以完整对应关系就是：

```
BswM_LE_AppRun2PostRun 成立
        |
        v
BswM_AR_AppRun2PostRun 这个 Rule 被触发
        |
        v
执行 BswM_AL_BswMSwitchPostRun
```

你关心休眠的话，几个关键对应关系是：

```
BswM_AR_AppRun2PostRun
    LE: BswM_LE_AppRun2PostRun
    AL: BswM_AL_BswMSwitchPostRun

BswM_AR_PostRun2PrepShutdown
    LE: BswM_LE_PostRun2PrepShutdown
    AL: BswM_AL_BswMSwitchPrepShutdown

BswM_AR_Shutdown
    LE: BswM_LE_PrepShutdown2Shutdown
    AL: BswM_AL_BswMSwitchShutdown
```

再看 `AppRun2PostRun` 的条件，在 [BswM_Cfg_LE.h (line 39)](E:/github/ECAS_qirui_Single_Chamber/BasicSoftware/src/bsw/BswM_PreCompile_and_PB_Variant/BswM_Cfg_LE.h:39)：

```
SWC_AppMode == POST_RUN
&& BswM_MDG == APP_RUN
&& WKSRC_IGN == NONE
&& WKSRC_CAN == NONE
```

成立后执行 `BswM_AL_BswMSwitchPostRun`，里面会做：

```
StopPduGroup
ReqNoComm
CommNotAllowed
BswMPostRun
```

其中最后的 `BswMPostRun` 通常就是把 `BswM_MDG` 切到 `POST_RUN`。 

## `BswMModeRequestPorts`
别的模块通知 BswM 当前状态。
- `BswM_MRP_BSWM_MDG`：BswM/EcuM 当前大状态，比如 `RUN`、`APP_RUN`、`POST_RUN`、`PREP_SHUTDOWN`
- `BswM_MRP_SWC_AppMode`：应用层请求的模式，比如 `APP_RUN`、`POST_RUN`
- `BswM_MRP_WKSRC_IGN`：IGN 唤醒源状态
- `BswM_MRP_WKSRC_CAN`：CAN 唤醒源状态
- `BswM_MRP_ComM_Can_Network_ETAS`：ComM 通信状态，比如 `COMM_NO_COMMUNICATION`
- `BswM_MRP_NVM_ReadAllComplete` / `WriteAllComplete`：NvM 读写完成状态
- `DCM_*`：诊断会话、通信控制、ECU reset 请求


## 关键变量解释


BswM 调试时，**最关键的不是看所有变量，而是看三类：输入值、规则状态、延迟请求状态。**

**第一优先级：MRP 输入变量**

这些是 BswM 判断规则的原始输入。下电问题优先看这些。

```
BswM_Cfg_GenericReqModeInfo_ast[BSWM_IDX_BSWM_MRP_BSWM_MDG].dataMode_u16
BswM_Cfg_GenericReqModeInfo_ast[BSWM_IDX_BSWM_MRP_BSWM_MDG].isValidModePresent_b
```

系统当前 BswM/EcuM 大状态。  
例如 `RUN`、`APP_RUN`、`POST_RUN`、`PREP_SHUTDOWN`、`SHUTDOWN`。

```
BswM_Cfg_GenericReqModeInfo_ast[BSWM_IDX_BSWM_MRP_SWC_APPMODE].dataMode_u16
BswM_Cfg_GenericReqModeInfo_ast[BSWM_IDX_BSWM_MRP_SWC_APPMODE].isValidModePresent_b
```

应用请求状态。  
下电时关键是它要变成 `POST_RUN`。

```
BswM_Cfg_EcuMWkpSrcInfo_ast[BSWM_IDX_BSWM_MRP_WKSRC_IGN].dataState
BswM_Cfg_EcuMWkpSrcInfo_ast[BSWM_IDX_BSWM_MRP_WKSRC_IGN].isValidModePresent_b

BswM_Cfg_EcuMWkpSrcInfo_ast[BSWM_IDX_BSWM_MRP_WKSRC_CAN].dataState
BswM_Cfg_EcuMWkpSrcInfo_ast[BSWM_IDX_BSWM_MRP_WKSRC_CAN].isValidModePresent_b
```

IGN/CAN 唤醒源状态。  
下电时通常都要是：

```
ECUM_WKSTATUS_NONE
```

```
BswM_Cfg_ComMIndicationModeInfo_ast[BSWM_IDX_BSWM_MRP_COMM_CAN_NETWORK_ETAS].dataMode_u8
BswM_Cfg_ComMIndicationModeInfo_ast[BSWM_IDX_BSWM_MRP_COMM_CAN_NETWORK_ETAS].isValidModePresent_b
```

CAN 通信状态。  
下电时通常要是：

```
COMM_NO_COMMUNICATION
```

**第二优先级：Rule 状态**

规则状态数组：

```
BswM_Prv_RuleState_aen[index]
```

下电重点看这些 index：

```
BswM_Prv_RuleState_aen[0]   // AppRun2PostRun
BswM_Prv_RuleState_aen[17]  // PostRun2AppRun
BswM_Prv_RuleState_aen[18]  // PostRun2PrepShutdown
BswM_Prv_RuleState_aen[19]  // Run2AppRun
BswM_Prv_RuleState_aen[20]  // Run2PrepShutdown
BswM_Prv_RuleState_aen[21]  // PrepShutdown2Shutdown
```

判断方式：

```
MRP 值正确，但 RuleState 没变 TRUE
=> LogicalExpression 没成立，或者规则没被触发评估

RuleState 变 TRUE，但动作没执行
=> 看 ActionList / Action / UserCallout
```

**第三优先级：Deferred 延迟请求**

如果某个 MRP 是 Deferred，不会立即执行规则，而是等 BswM_MainFunction 处理。

看：

```
BswM_Prv_DeferredRuleEvaluation_b[0]
BswM_Prv_DeferredRuleEvaluation_b[18]
BswM_Prv_DeferredRuleEvaluation_b[20]
BswM_Prv_DeferredRuleEvaluation_b[21]
```

以及：

```
BswM_Prv_DeferredReqstInfo_ast
BswM_Prv_flgDeferredReqstProgress_b
BswM_Prv_flgDelayDeferredReqst_b
```

判断方式：

```
MRP 已更新，但 Rule 没立刻变化
=> 看是不是 Deferred 请求还没被 BswM_MainFunction 消化
```

**启动/下电相关可选变量**

NvM：

```
BswM_Cfg_NvMJobModeInfo_ast[BSWM_IDX_BSWM_MRP_NVM_READALLCOMPLETE].dataMode_en
BswM_Cfg_NvMJobModeInfo_ast[BSWM_IDX_BSWM_MRP_NVM_WRITEALLCOMPLETE].dataMode_en
```

诊断复位：

```
BswM_Cfg_GenericReqModeInfo_ast[BSWM_IDX_BSWM_MRP_DCM_MODENOTIFICATION_ECURESET].dataMode_u16
BswM_Cfg_DcmComModeRequestModeInfo_ast[BSWM_IDX_BSWM_MRP_DCM_COMCONTROL_CAN_NETWORK_ETAS].dataMode_u8
```

**下电调试最小 Watch 清单**

如果只想看最少变量，就看这 10 个：

```
BswM_Cfg_GenericReqModeInfo_ast[0].dataMode_u16
BswM_Cfg_GenericReqModeInfo_ast[0].isValidModePresent_b

BswM_Cfg_GenericReqModeInfo_ast[4].dataMode_u16
BswM_Cfg_GenericReqModeInfo_ast[4].isValidModePresent_b

BswM_Cfg_EcuMWkpSrcInfo_ast[0].dataState
BswM_Cfg_EcuMWkpSrcInfo_ast[1].dataState

BswM_Cfg_ComMIndicationModeInfo_ast[0].dataMode_u8

BswM_Prv_RuleState_aen[18]
BswM_Prv_RuleState_aen[20]
BswM_Prv_RuleState_aen[21]
```

其中：

```
GenericReqModeInfo_ast[0] = BswM_MRP_BSWM_MDG
GenericReqModeInfo_ast[4] = BswM_MRP_SWC_AppMode
EcuMWkpSrcInfo_ast[0]     = WKSRC_CAN
EcuMWkpSrcInfo_ast[1]     = WKSRC_IGN
ComMIndicationModeInfo_ast[0] = CAN Network ComM
```

**调试判断顺序**

```
1. SWC_AppMode 是否变 POST_RUN
2. IGN/CAN wake 是否都是 NONE
3. ComM 是否 NO_COMMUNICATION
4. BswM_MDG 是否按 RUN -> PREP_SHUTDOWN -> SHUTDOWN 变化
5. RuleState[20]/[18]/[21] 是否变 TRUE
6. 如果 Rule 没变，看 DeferredRuleEvaluation
```
这个顺序比盲目看 ActionList 更有效。
