# DcmDsp 配置说明

本文基于当前仓库中的 DCM 配置和代码整理，说明 `DcmConfigSet/DcmDsp` 下各类配置的作用、生成代码位置，以及和 ASW/RTE 代码的关系。

> 注意：截图中 `DcmDspMaxDidToRead` 显示为 `5`，但当前仓库生成 ARXML `BasicSoftware/ecu_config/bsw/gen/RTA_BIP_Dcm_EcucValues.arxml` 中该值为 `20`。下文以当前仓库代码和 ARXML 为准。

## 1. 配置和代码入口

主要配置文件：

| 文件 | 作用 |
| --- | --- |
| `BasicSoftware/ecu_config/bsw/gen/RTA_BIP_Dcm_EcucValues.arxml` | 当前生成的 DCM Ecuc 配置，`DcmDsp` 的 DID、Routine、Session、Security 等主要配置在这里。 |
| `BasicSoftware/ecu_config/bsw/static/Diag/RTA_Dcm_EcucValues.arxml` | 静态诊断片段，包含一个 `DcmDspMemory` 下载内存范围配置。 |
| `BasicSoftware/src/bsw/Dcm/Dcm_Lcfg_DspUds.c` | DcmDsp 生成配置表和 RTE 调用包装函数。 |
| `BasicSoftware/src/bsw/Dcm/Dcm_Cfg_DspUds.h` | DcmDsp 功能开关、数量宏、DID/Routine/DTC 等编译开关。 |
| `BasicSoftware/src/bsw/Dcm/Dcm_Cfg_DslDsd.h` | UDS 服务级开关，例如 RequestDownload、ReadMemoryByAddress 是否启用。 |
| `BasicSoftware/src/bsw/Dcm/Dcm_Lcfg_Dsd.c` | DSD 服务表，决定 SID 是否启用、允许会话/安全等级、服务处理函数。 |
| `BasicSoftware/src/rte/gen/Rte.c` | RTE 生成层，连接 DCM 到 ASW runnable；未连接端口会返回 `RTE_E_UNCONNECTED`。 |
| `ASW/SWC/Dcm_Swc/src/Dcm_Swc.c` | 项目自定义 DID、IOControl、部分 Routine 的业务实现。 |
| `ASW/SWC/DiagUT/src/DiagUT.c` | SecurityAccess 和部分 Routine 的业务实现。 |

典型调用链：

```text
诊断仪请求
  -> Dsl 接收报文
  -> Dsd 根据 SID 查服务表
  -> Dsp 服务处理函数
  -> Dcm_Lcfg_DspUds.c 中的 DID/Routine/Session/Security 配置表
  -> Rte_Call 或 Rte_Read/Rte_Write
  -> ASW/SWC/Dcm_Swc 或 ASW/SWC/DiagUT 中的业务函数
```

## 2. DcmDsp 顶层配置

当前 `DcmDsp` 下实际配置的主要容器数量：

| 容器 | 数量 | 说明 |
| --- | ---: | --- |
| `DcmDspData` | 60 | DID 或 IOControl 使用的数据元素。多数为 `UINT8_N`、同步 Client/Server 端口。 |
| `DcmDspDid` | 52 | 配置 52 个 DID/CPID，包括 RDBI、WDBI 和 IOCBI。 |
| `DcmDspDidInfo` | 52 | 每个 DID 的读/写/IOControl 访问权限。 |
| `DcmDspDidSignal` | 8 | 只配置了 8 个额外信号拆分，主要用于 `0x010B`、`0x0112`、`0xD001` 等结构化 DID。 |
| `DcmDspRoutine` | 5 | 配置 5 个 RoutineControl RID。 |
| `DcmDspSession` | 1 | 内含 3 个 session row。 |
| `DcmDspSecurity` | 1 | 内含 2 个 security row。 |
| `DcmDspCommonAuthorization` | 2 | 复用的会话授权组合。 |
| `DcmDspComControl` | 1 | CommunicationControl 的通道配置。 |
| `DcmDspControlDTCSetting` | 1 | ControlDTCSetting 配置。 |
| `DcmRbDspReadDTC` | 1 | ReadDTCInformation 的读取数量限制。 |

顶层关键参数：

| 参数 | 当前值 | 代码含义 |
| --- | --- | --- |
| `DcmDspDataDefaultEndianness` | `OPAQUE` | 数据默认不按 CPU endian 自动解释，DID 数据按字节数组透传。 |
| `DcmDspEnableObdMirror` | `false` | 不启用 OBD mirror。 |
| `DcmDspMaxDidToRead` | `20` | 单次 `0x22 ReadDataByIdentifier` 最多读取 20 个 DID。 |
| `DcmRbDspSesSecUsedInProtocol` | `UDS` | Dsp 的 session/security 授权用于 UDS 协议。 |

## 3. DSD 中实际启用的服务

是否能处理某个 SID，最终由 `Dcm_Lcfg_Dsd.c` 的服务表决定。当前服务表中有 12 个 UDS 服务：

| SID | 服务 | 处理函数 | 允许会话 | 备注 |
| --- | --- | --- | --- | --- |
| `0x10` | DiagnosticSessionControl | `Dcm_Prv_DspDiagnosticSessionControl` | Default/Programming/Extended | 子功能 `0x01/0x02/0x03`。 |
| `0x11` | ECUReset | `Dcm_Prv_DspEcuReset` | Default/Programming/Extended | 子功能 `0x01/0x03`。 |
| `0x14` | ClearDiagnosticInformation | `Dcm_Prv_DspClearDiagnosticInformation` | Default/Extended | 没有单独的 DcmDsp 子配置实例，但服务已启用。 |
| `0x19` | ReadDTCInformation | `Dcm_Prv_DspReadDTCInformation` | Default/Extended | 启用子功能 `0x01/0x02/0x04/0x06/0x0A`。 |
| `0x22` | ReadDataByIdentifier | `Dcm_Prv_DspReadDataByIdentifier` | Default/Programming/Extended | 具体 DID 权限由 `DcmDspDidInfo` 再限制。 |
| `0x27` | SecurityAccess | `Dcm_Prv_DspSecurityAccess` | Extended/Programming | L1 使用 `0x01/0x02`，L3 使用 `0x05/0x06`。 |
| `0x28` | CommunicationControl | `Dcm_Prv_DspCommunicationControl` | Programming/Extended | 4 个子功能。 |
| `0x2E` | WriteDataByIdentifier | `Dcm_Prv_DspWriteDataByIdentifier` | Extended | 服务级安全掩码限制为 L1。 |
| `0x2F` | InputOutputControlByIdentifier | `Dcm_Prv_DspInputOutputControlByIdentifier` | Extended | 具体 CPID 由 DID 配置决定。 |
| `0x31` | RoutineControl | `Dcm_Prv_DspRoutineControl` | Programming/Extended | 具体 RID 权限由 Routine 子功能配置决定。 |
| `0x3E` | TesterPresent | `Dcm_Prv_DspTesterPresent` | Default/Programming/Extended | 常规在线保持。 |
| `0x85` | ControlDTCSetting | `Dcm_Prv_DspControlDTCSetting` | Default/Extended | 子功能开关 DTC 设置。 |

## 4. Session 配置

生成代码位置：`Dcm_Lcfg_DspUds.c` 中 `Dcm_Dsp_Session[DCM_CFG_DSP_NUMSESSIONS]`。

| Session | Level | P2ServerMax | P2StarServerMax | Boot 行为 | 说明 |
| --- | ---: | ---: | ---: | --- | --- |
| `DEFAULT_SESSION` | `1` | `0.05s` | `2.0s` | `DCM_NO_BOOT` | 默认会话。 |
| `PROGRAMMING_SESSION` | `2` | `0.05s` | `2.0s` | `DCM_OEM_BOOT` | 编程会话，可用于刷写相关授权和部分 DID。 |
| `EXTENDED_DIAGNOSTIC_SESSION` | `3` | `0.05s` | `2.0s` | `DCM_NO_BOOT` | 扩展诊断会话，多数诊断控制、写 DID、Routine 依赖它。 |

代码中 P2/P2* 以微秒保存，例如 `0.05s` 生成成 `50000us`，`2.0s` 生成成 `2000000us`。

## 5. SecurityAccess 配置

生成代码位置：`Dcm_Lcfg_DspUds.c` 中 `Dcm_Dsp_Security[DCM_CFG_DSP_NUMSECURITY]`。

| Security row | Level | SeedSize | KeySize | DelayTime | DelayOnBoot | NumAttDelay | UsePort | 应用实现 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | --- | --- |
| `Unlocked_L1` | `1` | `4` | `4` | `10.0s` | `10.0s` | `3` | `USE_ASYNCH_CLIENT_SERVER` | `DiagUT.c` 中 `SecurityAccess_Unlocked_L1_GetSeed`、`CompareKey`。 |
| `Unlocked_L3` | `3` | `4` | `4` | `10.0s` | `10.0s` | `3` | `USE_ASYNCH_CLIENT_SERVER` | `DiagUT.c` 中 `SecurityAccess_Unlocked_L3_GetSeed`、`CompareKey`。 |

配置未启用 attempt counter 持久化函数，`DcmDspSecurityAttemptCounterEnabled=false`，所以失败计数主要由 DCM RAM 状态和 delay 策略控制。

实际代码调用关系：

```text
0x27 SecurityAccess
  -> Dcm_Dsp_Security 配置确认 level、seed/key 长度、delay
  -> Rte_Call_SecurityAccess_Unlocked_L1_GetSeed / CompareKey
  -> Rte_Call_SecurityAccess_Unlocked_L3_GetSeed / CompareKey
  -> ASW/SWC/DiagUT/src/DiagUT.c
```

## 6. CommonAuthorization 配置

`DcmDspCommonAuthorization` 是权限组合，主要被 DID/Routine 的访问配置引用。

| 名称 | 允许 session | Security 引用 | 用途 |
| --- | --- | --- | --- |
| `AP_Comb_0002` | `EXTENDED_DIAGNOSTIC_SESSION`、`PROGRAMMING_SESSION` | 空 | 编程/扩展会话复用授权。 |
| `AP_Extended` | `EXTENDED_DIAGNOSTIC_SESSION` | 空 | 只允许扩展会话的复用授权。 |

这里没有配置 security level 引用，表示它们只表达 session 条件；安全等级通常在 DID 写权限或服务级掩码中单独限制。

## 7. DID、Data、DidInfo 和 DidSignal

DID 配置由四类容器组合起来生效：

| 容器                | 作用                                           | 生成代码                                  |
| ----------------- | -------------------------------------------- | ------------------------------------- |
| `DcmDspData`      | 定义数据长度、类型、端口类型、读写/IOControl 回调引用。            | `Dcm_DspDataInfo_st[]` 和 RTE wrapper。 |
| `DcmDspDidInfo`   | 定义读、写、IOControl 是否允许，以及允许的 session/security。 | DID 控制信息表。                            |
| `DcmDspDid`       | 定义 DID identifier、DID 总长度、引用 `DidInfo`。      | `Dcm_DIDConfig[]`。                    |
| `DcmDspDidSignal` | 定义 DID 内部数据元素偏移，引用 `DcmDspData`。             | DID signal/data mapping。              |

当前多数 `DcmDspData` 都是：

| 参数 | 常见值 | 含义 |
| --- | --- | --- |
| `DcmDspDataType` | `UINT8_N` | 以字节数组传递。 |
| `DcmDspDataUsePort` | `USE_DATA_SYNCH_CLIENT_SERVER` | DCM 通过同步 RTE Client/Server 调用应用读写函数。 |
| `DcmDspDataEndianness` | `OPAQUE` | 不做 endian 转换。 |
| `DcmDspDidByteOffset` | `0` | 单元素 DID 基本都从 offset 0 开始。 |

### 7.2 DID 业务代码模式

`Dcm_Swc.c` 中的 DID 函数大多按以下模式实现：

```text
RE_DataServices_DID_xxxx_Read_func
  -> Rte_Read_PRP_DIDxxxxNvBlock_...
  -> 拷贝到 DCM 输出 buffer

RE_DataServices_DID_xxxx_Write_func
  -> 校验/拷贝 tester 输入
  -> Rte_Write_PRP_DIDxxxxNvBlock_...
```

`0xF100` 和 `0xF101` 不是从 NvBlock 读取，而是返回代码中定义的常量：

| DID | 数据来源 |
| --- | --- |
| `0xF100` | `ECU_SOFTWARE_VERSION_NUMBER` |
| `0xF101` | `ECU_HARDWARE_VERSION_NUMBER` |

当前 RTE 明确未连接的 DataServices 包括：

```text
0x010B_ReadData
0x0112_ReadData
0x000A_ReadData / WriteData
0x000B_ReadData / ReturnControlToECU / ShortTermAdjustment
0xD001_ReadData
0xD901_FreezeCurrentState / ReadData / ResetToDefault / ReturnControlToECU / ShortTermAdjustment
0xE101_ReadData
0xF192_ReadData
0xF194_ReadData
0xFD0C_FreezeCurrentState / ResetToDefault
```

这些端口在 `Rte.c` 中返回 `RTE_E_UNCONNECTED`。如果测试这些 DID/CPID，请先确认它们是否应当接入 ASW runnable，否则服务会因为应用端口未连接而失败。

## 8. RoutineControl 配置

生成代码位置：

| 代码区域 | 说明 |
| --- | --- |
| `Dcm_Dsp_RC_..._Func` | 每个 RID 的 wrapper，内部调用 RTE。 |
| `Dcm_DspRoutine_cast[]` | Routine 总表，记录 RID、可用子功能、授权和 wrapper。 |
| `Dcm_DspRoutineSignal_cast[]` | Routine 输入/输出信号长度。 |

当前 RID：

| RID | 配置名 | 子功能 | 授权 | RTE/应用状态 |
| --- | --- | --- | --- | --- |
| `0x000C` | `DcmDspRoutine_0xC` | Start/Stop/RequestResults | 依 Routine 子功能配置 | 生成代码会调用 RTE，但当前未定位到 ASW 实现。 |
| `0xD003` | `DcmDspRoutine_0xD003` | Start | 依 Routine 子功能配置 | `Dcm_Swc.c` 中有 `RoutineServices_Routine_Control_0xD003_Start_func`。 |
| `0xABFF` | `DcmDspRoutine_0xABFF` | Start/RequestResults | `AP_Extended` | `DiagUT.c` 中实现 Start 和 RequestResults。 |
| `0xABFE` | `DcmDspRoutine_0xABFE` | Start | `AP_Extended` | `DiagUT.c` 中实现 Start。 |
| `0x0203` | `DcmDspRoutine_0x203` | Start | `AP_Extended` | `DiagUT.c` 中实现 Start。 |

Routine 信号配置示例：

| RID | Signal | 长度 |
| --- | --- | ---: |
| `0x000C` | StartIn | 32 bit |
| `0x000C` | StartOut | 40 bit |
| `0x000C` | StopIn | 48 bit |
| `0x000C` | StopOut | 56 bit |
| `0x000C` | RequestResultsOut | 24 bit |
| `0xD003` | StartOut | 64 bit |
| `0xABFF` | RequestResultsOut | 8 bit |
| `0x0203` | StartOut | 8 bit |

当前 `Dcm_Cfg_DspUds.h` 中 `DCM_CFG_DSP_NUMISRIDAVAIL=0`，表示没有配置 RID availability 函数。Routine 是否可用主要由 session/security、子功能序列和业务函数返回值决定。

## 9. DTC、ComControl 和 ECUReset 相关配置

### 9.1 ControlDTCSetting

`DcmDspControlDTCSetting` 中：

| 参数 | 当前值 | 含义 |
| --- | --- | --- |
| `DcmSupportDTCSettingControlOptionRecord` | `false` | `0x85` 不支持额外 option record。 |

DSD 中 `0x85` 已启用，允许 Default/Extended session。

### 9.2 ReadDTCInformation

`DcmRbDspReadDTC` 中：

| 参数 | 当前值 | 含义 |
| --- | ---: | --- |
| `DcmRbDspReadDTCMaxNumDTCRead` | 1 | 单次最多读取 1 个 DTC。 |
| `DcmRbDspReadDTCMaxNumRecordRead` | 1 | 单次最多读取 1 个记录。 |

当前启用的 `0x19` 子功能为：

```text
0x01 reportNumberOfDTCByStatusMask
0x02 reportDTCByStatusMask
0x04 reportDTCSnapshotRecordByDTCNumber
0x06 reportDTCExtendedDataRecordByDTCNumber
0x0A reportSupportedDTC
```

### 9.3 CommunicationControl

`DcmDspComControl` 配置了一个 all-channel mode info，引用：

```text
ComMConfigSet/Can_Network_Channel_Can_Network
```

DSD 中 `0x28` 已启用，允许 Programming/Extended session，子功能数量为 4。

### 9.4 ECUReset

编辑器中有 `DcmRbDspEcuResetRow` 节点，但当前主 ARXML 没有看到单独的 DcmDsp reset row 实例。DSD 服务表中 `0x11` 已启用，支持子功能 `0x01` 和 `0x03`，允许所有 session。

## 10. Memory、Transfer 和 FileTransfer

静态 ARXML 中存在一个 `DcmDspMemory` 片段：

| 配置 | 值 |
| --- | --- |
| `DcmDspMemoryIdInfo` | `DcmDspMemoryIdInfo` |
| `DcmRbDspDownloadMemoryRangeInfo` low | `0` |
| `DcmRbDspDownloadMemoryRangeInfo` high | `4294967294` |

但是当前生成宏显示相关服务关闭：

```text
DCM_CFG_DSP_REQUESTDOWNLOAD_ENABLED      OFF
DCM_CFG_DSP_REQUESTUPLOAD_ENABLED        OFF
DCM_CFG_DSP_TRANSFERDATA_ENABLED         OFF
DCM_CFG_DSP_REQUESTTRANSFEREXIT_ENABLED  OFF
DCM_CFG_DSP_READMEMORYBYADDRESS_ENABLED  OFF
DCM_CFG_DSP_WRITEMEMORYBYADDRESS_ENABLED OFF
```

同时 `Dcm_Lcfg_DspUds.c` 中未生成 RequestDownload 的范围配置表。因此当前这段 memory range 配置没有被实际服务使用。

如果后续启用下载/上传链路，典型代码路径会变成：

```text
0x34 RequestDownload
  -> Dcm_RequestDownloadConfig_cast 范围匹配
  -> DcmAppl_Dcm_ProcessRequestDownload
0x36 TransferData
  -> DcmAppl_Dcm_CheckPermissionForTransferData
```
