# BRS的作用是什么？
![](assets/2026.06.05%20CAN驱动笔记/file-20260605173127341.png)


`BRS` 全称是 **Bit Rate Switch**，只用于 **CAN FD**。

它的作用是：

> **允许 CAN FD 报文在仲裁完成后，把数据段切到更高的波特率发送。**

例如你当前工程的 CAN1：

```text
Nominal phase / 仲裁段：500 kbit/s
Data phase / 数据段：2 Mbit/s
BRS = true
```

**开启 BRS 后，一帧 CAN FD 大概是这样：**

```text
SOF、ID、仲裁相关位       500 kbit/s
BRS 之后的数据段          2 Mbit/s
CRC 相关高速部分          2 Mbit/s
ACK、EOF 等再回到 nominal
```

好处：
**仲裁阶段低速，保证多节点仲裁可靠**
**数据阶段高速，提高 payload 传输效率**

**如果 `BRS = false`：**
**整帧 CAN FD 都用 nominal bitrate**
**比如全程 500 kbit/s**


它仍然可以是 CAN FD 帧，也仍然可以发超过 8 字节的数据，但不会切高速。

注意几点：

1. 只有 CAN FD 才有 BRS，Classic CAN 没有。
2. 总线上所有参与该 CAN FD 报文通信的节点都要支持对应的 data bitrate。
3. CAN 分析仪也要配置成 CAN FD + BRS，否则可能看到错误帧。
4. BRS 开启后更容易暴露物理层问题，比如线束、终端电阻、收发器能力、TDC 配置不合适。

一句话记忆：

> **`BRS` 是 CAN FD 的“数据段加速开关”。仲裁段慢慢抢总线，抢赢以后数据段高速发。**


# ==什么是仲裁段和数据段==

在一帧 CAN/CAN FD 报文里，可以粗略分成两类时间区域：

```text
仲裁段：决定“谁有资格继续发”
数据段：真正搬运 payload 数据
```

**仲裁段**就是多个节点可能同时开始发送时，用来抢总线优先级的部分。CAN 总线是“显性 0 压过隐性 1”，所以节点一边发一边读总线。如果自己发的是 1，但读到总线是 0，就说明有更高优先级的帧在发，自己退出仲裁。仲裁主要看 CAN ID，ID 数值越小，优先级越高。

仲裁段里通常包括：

```text
SOF
CAN ID
RTR / IDE 等控制位的一部分
```

**数据段**就是仲裁赢了的那个节点继续发送真正的数据内容，也就是 payload：

```text
DLC
Data bytes
CRC 相关部分
```

对 **Classic CAN** 来说，仲裁段和数据段速度一样，例如整帧都是 500 kbit/s。

对 **CAN FD + BRS** 来说，速度会切换：

```text
仲裁段：Nominal bitrate，例如 500 kbit/s
数据段：Data bitrate，例如 2 Mbit/s
```

**为什么 CAN FD 不能一开始就全程高速？因为仲裁时可能多个节点同时发，必须保证全网节点都能可靠同步、比较电平、退出仲裁，所以仲裁段通常保持较低速率。等仲裁结束后，只剩一个节点在发，就可以切到更高的数据段速率来提高吞吐量。**

# CAN和CANFD的位时序分解是一样的吗？

基本分解思想是一样的，但 **CAN FD 有两套位时序**。

你图里这种分解：

```text
1 bit = SS + PTS + PBS1 + PBS2
      = SyncSeg + PropSeg + PhaseSeg1 + PhaseSeg2
```

这个对 **Classic CAN** 和 **CAN FD** 都成立。

区别在这里：

| 类型            | 位时序数量 | 说明                                               |
| ------------- | ----: | ------------------------------------------------ |
| Classic CAN   |   1 套 | 全帧都用同一个 bit timing                               |
| CAN FD 不开 BRS |   1 套 | 虽然是 FD 帧，但不切高速，整帧仍用 nominal bit timing           |
| CAN FD 开 BRS  |   2 套 | **仲裁段用 nominal bit timing，数据段用 data bit timing** |

CAN FD 开 BRS 时可以理解成：

```text
仲裁段 / 控制段前部：
    Nominal bit timing
    例如 500 kbit/s

数据段 / CRC 相关高速部分：
    Data bit timing
    例如 2 Mbit/s
```

每一套内部仍然这样分：

```text
SyncSeg + PropSeg + PhaseSeg1 + PhaseSeg2
```

只是两套参数不同：

```text
Nominal phase:
    Prescaler_N
    PropSeg_N
    Seg1_N
    Seg2_N
    SJW_N

Data phase:
    Prescaler_D
    PropSeg_D
    Seg1_D
    Seg2_D
    SJW_D
```

结合你当前工程的 CAN1：

```text
Nominal phase:
40 MHz / 5 / (1 + 9 + 3 + 3)
= 500 kbit/s

Data phase:
40 MHz / 1 / (1 + 11 + 4 + 4)
= 2 Mbit/s
```

所以一句话：

> **CAN 和 CAN FD 的位时序分解形式一样，都是 SyncSeg + PropSeg + PhaseSeg1 + PhaseSeg2；但 CAN FD 如果开启 BRS，会有 nominal phase 和 data phase 两套位时序。**

另外 CAN FD 数据段速度高，通常还会用 `TDC`，也就是 Transceiver Delay Compensation，补偿收发器延迟。Classic CAN 一般不需要这么关注这个。



# TDC是怎么补偿的？
![](assets/2026.06.05%20CAN驱动笔记/file-20260605173127340.png)
1. ==**高速 data phase 下，CAN transceiver 和 PCB 线路延迟会让采样点偏移。**
2. **==TDC 用 Secondary Sample Point 补偿 transmitter delay。**
3. **==EB 里的 `CanControllerSspOffset = 17` 最终生成到 baudrate 配置里。**
4. ==**如果 FD 2 Mbit/s 只在某些节点上不稳定，除了波特率，还要重点查 TDC、transceiver FD 能力、线束长度、终端电阻==**
`TDC` 全称是 **Transceiver Delay Compensation**，也叫收发器延迟补偿。

它补偿的不是总线本身，而是 **发送节点从 TXD 发出去，到自己 RXD 看到这个变化之间的延迟**。

可以这样理解：

```text
MCU FlexCAN_TX
    |
    v
CAN Transceiver
    |
    v
CANH/CANL 总线
    |
    v
CAN Transceiver 回读
    |
    v
MCU FlexCAN_RX
```

这一路会有延迟，叫 loop delay / transceiver delay。

## 1. 为什么 CAN FD 需要 TDC

Classic CAN 速度低，比如 500 kbit/s：

```text
1 bit = 2 us
```

收发器延迟假设 100 ns 到 200 ns，占比不大。

但 CAN FD data phase 可能是 2 Mbit/s、5 Mbit/s：

```text
2 Mbit/s: 1 bit = 500 ns
5 Mbit/s: 1 bit = 200 ns
```

这时 100 ns 的收发器延迟就很明显了。

如果发送节点还是按普通 sample point 去检查自己发出去的 bit，可能会出现：

```text
我已经切到下一个 bit 了
但 RXD 回读的还是上一个 bit
```

于是控制器误以为发生 bit error。

## 2. TDC 怎么补偿

TDC 的核心动作是：

> **在 CAN FD 数据段里，发送节点不用普通采样点检查自己的回读信号，而是使用一个延后的 Secondary Sample Point，也就是 SSP。**

普通情况：

```text
bit start
   |
   |------ normal sample point
   |
bit end
```

开启 TDC 后：

```text
bit start
   |
   |------ normal sample point
   |
   |------------- secondary sample point, SSP
   |
bit end
```

SSP 会考虑 TXD 到 RXD 的回环延迟。

更形象一点：

```text
TXD 边沿：  |________________
             t0

RXD 边沿：       |___________
                 t0 + transceiver delay

普通采样点：   ^   可能太早
TDC/SSP：          ^   延后采样，避开收发器延迟
```

## 3. FlexCAN 里面相关字段

在 S32K324 FlexCAN 里，TDC 主要和 `FDCTRL` 有关：

| 字段 | 作用 |
|---|---|
| `TDCEN` | 使能 Transceiver Delay Compensation |
| `TDCOFF` | TDC offset，决定 SSP 的位置 |
| `TDCVAL` | 控制器测得或使用的 delay value |

在 EB 里通常对应：

```text
CanControllerSspOffset
Transceiver Delay Compensation Enable
```

你当前工程 CAN1 是：

```text
CAN FD enabled = true
BRS = true
Data phase = 2 Mbit/s
TDC enabled = true
SspOffset = 17
```

也就是说，CAN1 在 FD 高速数据段会用 TDC 来调整发送节点的回读采样点。

## 4. TDC 不是改变总线波形

这个点很重要：
**TDC 不会让 CANH/CANL 变快**
**TDC 不会修复终端电阻、线束、收发器能力问题**
**TDC 只是调整发送节点内部的回读采样点**


接收节点仍然按自己的 data phase bit timing 采样。

## 5. 调试时怎么看

如果 CAN FD 仲裁段正常，但一到数据段就错误，重点查：

```text
BRS 是否两端一致
Data bitrate 是否一致
TDC 是否开启
TDC offset 是否合适
Transceiver 是否支持该 FD 速率
线束长度和终端电阻是否合理
CAN 分析仪是否配置 CAN FD + BRS
```

一句话记忆：

> **==`TDC` 是 CAN FD 高速数据段里，发送节点为了补偿 TXD 到 RXD 回读延迟而引入的“延后采样点”。==**



# JUMP WIDTH是什么？

![](assets/2026.06.05%20CAN驱动笔记/file-20260605173127339.png)
**在重新同步的时候，PBS1 和 PBS2 中增加或减少的这段时间长度被定义为“重新同步补偿宽度SJW* (reSynchronization Jump Width)”。一般来说 CAN 控制器会限定 SJW 的最大值，如限定了最大 SJW=3Tq 时，单次同步调整的时候不能增加或减少超过 3Tq 的时间长度，若有需要，控制器会通过多次小幅度调整来实现同步。当控制器设置的 SJW 极限值较大时，可以吸收的误差加大，但通讯的速度会下降**


# 第二采样点的时刻是怎么确定的？
![](assets/2026.06.05%20CAN驱动笔记/file-20260605173127338.png)
17代表的是**以时间量子（Time Quantum, Tq）为单位的额外补偿偏移量**。
![](assets/2026.06.05%20CAN驱动笔记/file-20260605173127337.png)
不是直接由“第一采样点 80%”推出第二采样点。

你这个 `Transmitter Delay Compensation Offset = 17` 的单位是 **CAN FD data phase 的 Tq**。

当前配置：

```text
SyncSeg = 1
PropSeg = 11
PhaseSeg1 = 4
PhaseSeg2 = 4
Prescaler = 1
FlexCAN clock = 40 MHz
```

所以 data phase：

```text
总 Tq = 1 + 11 + 4 + 4 = 20 Tq

1 Tq = 1 / 40 MHz = 25 ns

1 bit = 20 Tq = 500 ns
Data bitrate = 2 Mbit/s
```

第一采样点：

```text
第一采样点 = (1 + 11 + 4) / 20
          = 16 / 20
          = 80%

也就是第 16 Tq
= 16 * 25 ns
= 400 ns
```

TDC offset = 17 表示：

```text
TDCOFF = 17 Tq
       = 17 * 25 ns
       = 425 ns
```

**但第二采样点 SSP 不是简单等于 17 Tq。FlexCAN 的思路是**：

```text
第二采样点 SSP = 测得的收发器回环延迟 + TDCOFF
```

也就是：

```text
SSP = TDCVAL + TDCOFF
```

如果暂时先忽略测得的 **transceiver loop delay**，单看 offset：

```text
17 Tq / 20 Tq = 85%
```

所以可以粗略理解成：

```text
第一采样点：16 Tq = 80%
TDC offset：17 Tq = 85%
```

但严格来说：

```text
第二采样点 = transceiver loop delay + 17 Tq
```

举例：

```text
如果硬件测得 loop delay = 2 Tq
那么 SSP = 2 + 17 = 19 Tq
        = 95%

如果硬件测得 loop delay = 1 Tq
那么 SSP = 1 + 17 = 18 Tq
        = 90%
```

所以这个 `17` 的含义不是百分比，而是：

> 在 CAN FD 高速数据段里，用来把发送节点的回读比较点往后推的 Tq 偏移量。

你这组配置里，`17` 选得比较接近 bit 后段，因为 data phase 一个 bit 总共只有 `20 Tq`。


# 什么是收发器的回环延迟？

实际项目里有两种办法：

```text
1. 用示波器量 TXD -> RXD 的延迟
2. 让 FlexCAN 自己测，然后读 TDCVAL
```

最常用、最直观的是第一种。

## 1. 示波器怎么量

测的不是 CANH/CANL 之间的传播延迟，而是本节点自己的回读延迟：

```text
MCU CAN_TXD  ->  收发器 TXD 输入
                  |
                  v
              CANH/CANL 总线
                  |
                  v
收发器 RXD 输出 -> MCU CAN_RXD
```

你要量的是：

```text
CAN_TXD 边沿 到 CAN_RXD 对应边沿 的时间差
```

接法：

```text
示波器 CH1：MCU CAN_TXD / 收发器 TXD
示波器 CH2：MCU CAN_RXD / 收发器 RXD
GND：板上地
```

然后让 MCU 发送 CAN FD + BRS 报文，数据区最好用容易产生跳变的 pattern：

```text
0x55 0xAA 0x55 0xAA ...
```

**这样数据段里会有很多边沿，方便测量。**

示意：

```text
TXD:  ____|‾‾‾‾‾‾
             ^
             t_tx

RXD:  ________|‾‾‾‾
                 ^
                 t_rx

loop delay = t_rx - t_tx
```

假设你量到：

```text
TXD 到 RXD 延迟 = 80 ns
```

当前 data phase：

```text
1 Tq = 25 ns
```

那么：

```text
loop delay = 80 ns / 25 ns = 3.2 Tq
```

可以按约 `3 Tq` 或 `4 Tq` 理解。

## 2. 那 EB 里的 offset 怎么选

你当前配置：

```text
1 bit = 20 Tq
第一采样点 = 16 Tq = 80%
TDC offset = 17 Tq
```

FlexCAN 的第二采样点大致是：

```text
SSP = 测得的 loop delay + TDCOFF
```

比如：

```text
loop delay = 3 Tq
TDCOFF = 17 Tq

SSP = 3 + 17 = 20 Tq
```

这就太靠后，甚至接近 bit 边界了，可能不理想。

所以实际调试时不是盲填一个数，而是要看：

```text
TDCOFF + loop delay < 一个 data bit 的总 Tq
```

一般希望 SSP 落在 data bit 的后半段，但不要越过 bit 末尾。

## 3. FlexCAN 自己测

S32K3 FlexCAN 有 TDC value 字段，驱动里也有：

```c
FlexCAN_Ip_GetTDCValue(instance)
```

它读的是硬件测得的 TDC value。实际调试可以：

```text
1. 开启 CAN FD + BRS + TDC
2. 发送 CAN FD 报文
3. 读 TDCVAL
4. 看 TDCFAIL 是否置位
```

如果 `TDCFAIL = 1`，说明 TDC 机制测量或补偿超范围，配置就要重新看。

## 4. 项目里推荐做法

实际项目一般这样做：

```text
1. 先按收发器 datasheet 估算 loop delay
2. 用示波器测 TXD -> RXD 实际延迟
3. 换算成 data phase Tq
4. 配 TDCOFF
5. 发 CAN FD + BRS 压力报文
6. 观察 error frame、bus off、TDCFAIL、误码
7. 再微调 TDCOFF
```

一句话：

> **收发器回环延迟就是本节点 `TXD 边沿` 到本节点 `RXD 回读边沿` 的时间差；实际项目里用示波器量，或者读 FlexCAN 的 `TDCVAL` 辅助确认。**


## 5 怎么用示波器测 TXD -> RXD 实际延迟 ，
一个节点MCU端两个引脚TX RX,经过收发器后，CANH \ CANL，如果向外发送的话,TXD有波形的变化，RXD同时也要有波形的变化吧？收发器再解析发送出去？
对，你理解得差不多了：**本节点向外发送时，TXD 有变化，RXD 也应该几乎同步有变化，只是 RXD 会晚一点点。**

但这里有个关键点：

> **==收发器不是“发送完再解析回来”，而是 一边根据 TXD 驱动 CANH/CANL，一边用内部接收比较器实时监听 CANH/CANL，并从 RXD 输出总线状态==。**

可以画成这样：

```text
MCU CAN_TXD
   |
   v
Transceiver TXD 输入
   |
   v
Transceiver Driver 驱动 CANH/CANL
   |
   v
CAN 总线电平变化
   |
   v
Transceiver Receiver 比较 CANH-CANL
   |
   v
Transceiver RXD 输出
   |
   v
MCU CAN_RXD
```

所以发送时：

```text
TXD 先变
CANH/CANL 后变
RXD 再跟着变
```

这个时间差就是你要量的：

```text
TXD -> RXD loop delay
```

### 示波器怎么接

推荐接在收发器附近：

```text
CH1：收发器 TXD 引脚
CH2：收发器 RXD 引脚
GND：板上 GND
```

也可以接 MCU 端的 CAN_TX / CAN_RX，但更推荐收发器引脚附近，少一点 PCB 走线影响。

如果有 4 通道，可以再加：

```text
CH3：CANH
CH4：CANL
```

或者用 math：

```text
CANH - CANL
```

### 正常波形关系

以常见高速 CAN 收发器为例：

```text
TXD = 0  ->  dominant  ->  CANH/CANL 拉开差分  ->  RXD = 0
TXD = 1  ->  recessive ->  CANH/CANL 回到隐性   ->  RXD = 1
```

所以 TXD 和 RXD 通常是同极性的：

```text
TXD:  ___|‾‾‾‾‾
          ^
          t1

RXD:  _____|‾‾‾‾
            ^
            t2

delay = t2 - t1
```

### 实际操作步骤

1. 让 CAN 收发器处于 Normal mode，不要在 standby/sleep。
2. 总线接好终端电阻，最好有另一个节点或 CAN 分析仪提供 ACK。
3. MCU 周期发送 CAN FD + BRS 报文。
4. Data payload 用容易产生跳变的数据，例如：

```text
55 AA 55 AA 55 AA ...
```

5. 示波器触发 CH1，也就是 TXD 的边沿。
6. 放大到几十 ns/div，比如 `20 ns/div`、`50 ns/div`。
7. 用 cursor 量：

```text
TXD 边沿到 RXD 对应边沿的时间差
```

### 举个数字

假设你测到：

```text
TXD -> RXD = 80 ns
```

你当前 CAN FD data phase：

```text
FlexCAN clock = 40 MHz
Prescaler = 1
1 Tq = 25 ns
```

换算：

```text
80 ns / 25 ns = 3.2 Tq
```

**可以理解成收发器回环延迟大约是 `3~4 Tq`。**

### 注意

最好不要只接一个孤立节点测，因为没有 ACK 时 CAN 控制器会反复报错重发，波形会乱一些。虽然仍然能看到 TXD/RXD 延迟，但用 CAN 分析仪或第二个节点 ACK，波形会更干净。

一句话：

> 发送时 RXD 本来就应该跟着变化，因为 RXD 是收发器实时监听总线后的输出。示波器量 TXD 边沿到 RXD 边沿的延迟，就是实际 TXD -> RXD 回环延迟。

## 必须懂的部分

这张表是 **FlexCAN Message Buffer 的硬件内存结构**。你要知道一个 MB 大概分三块：

```text
0x0: 控制/状态 word
0x4: ID word
0x8: DATA word 0
0xC: DATA word 1
...
```

也就是：

```text
CS + ID + DATA
```

### 1. CODE

`CODE` 表示这个 MB 当前状态。

比如：

```text
Rx empty
Rx full
Tx inactive
Tx data frame
Tx abort
```

你不用背具体编码，但要知道：

> Driver 通过 CODE 判断这个邮箱是空的、收到了、正在发、发完了。

### 2. DLC

`DLC` 表示数据长度。

Classic CAN：

```text
DLC 0..8
```

CAN FD：

```text
DLC 可以表示 12 / 16 / 20 / 24 / 32 / 48 / 64 byte
```

这和 EB 里的：

```text
Can Object Payload Length
```

有关。

### 3. IDE / RTR

| 位     | 含义       |
| ----- | -------- |
| `IDE` | 标准帧还是扩展帧 |

| 对比项        | Classic CAN MB                     | CAN FD MB                                 |
| ---------- | ---------------------------------- | ----------------------------------------- |
| 用途         | 普通 CAN 报文                          | CAN FD 报文，也可兼容普通 CAN                      |
| 最大 payload | 8 byte                             | 64 byte                                   |
| 控制位        | `CODE`、`DLC`、`IDE`、`RTR`、timestamp | 增加 `EDL/FDF`、`BRS`、`ESI`                  |

| RAM 占用     | 小                                  | payload 越大，占用越大                           |

| 额外能力       | 无高速数据段                             | 可通过 `BRS` 使用高速 data phase                 |
| 典型工程对象     | Classic CAN Rx/Tx mailbox          | CAN FD Rx/Tx mailbox                      |

简单看结构：

```text
Classic CAN MB:
  CS word:
    CODE / IDE / RTR / DLC / timestamp
  ID word:
    Standard ID or Extended ID
  DATA:
    byte 0..7
```

```text
CAN FD MB:
  CS word:
    EDL/FDF / BRS / ESI / CODE / IDE / RTR / DLC / timestamp
  ID word:
    Standard ID or Extended ID
  DATA:
    byte 0..最多 63
```


> Classic CAN MB 是 8 字节邮箱；CAN FD MB 是可变大邮箱，最多 64 字节，并且多了 `EDL/BRS/ESI` 来支持 FD 和高速数据段。


# legacy RX FIFO是什么
![](assets/2026.06.05%20CAN驱动笔记/file-20260605173127333%201.png)
`Legacy RX FIFO` 是 FlexCAN 里一种 **老式接收 FIFO 机制**。

它和普通 Rx MB 的区别是：

```text
普通 Rx MB：每个邮箱单独接收一个或一类 ID
Legacy RX FIFO：一块固定 FIFO 区域 + 一张 ID Filter Table，符合过滤条件的报文先进先出排队
```

## 1. 普通 Rx MB 是什么

普通接收邮箱大概是这样：

```text
Rx MB0  接收 ID 0x100
Rx MB1  接收 ID 0x101
Rx MB2  接收 ID 0x200
Rx MB3  接收 ID 0x300
...
```

每个 MB 都有自己的：

```text
CS + ID + DATA
```

优点是简单、确定，每个报文落在哪个 MB 很清楚。

缺点是：如果报文 ID 很多，MB 会用得很快。

## 2. Legacy RX FIFO 是什么

Legacy RX FIFO 开启后，FlexCAN 会把前面一段 MB RAM 改成特殊布局：

```text
RX FIFO 区域
Reserved 区域
ID Filter Table
后面的普通 MB
```

你截图里就是这个意思：

```text
0x80  FIFO 输出报文的 CS
0x84  FIFO 输出报文的 ID
0x88  FIFO 输出报文 Data 0..3
0x8C  FIFO 输出报文 Data 4..7
0xE0  ID Filter Table Element 0
0xE4  ID Filter Table Element 1
...
```

它不是每个 ID 一个 MB，而是：

```text
ID Filter Table 判断是否接收
        |
        v
符合条件的报文进入 RX FIFO
        |
        v
软件从 FIFO 里按顺序读出来
```

## 3. 为什么叫 Legacy

因为 NXP FlexCAN 后来还有 **Enhanced Rx FIFO**。所以老的 FIFO 机制叫：

```text
Legacy RX FIFO
```

可以理解成：

```text
Legacy RX FIFO = 旧版接收 FIFO
Enhanced RX FIFO = 新版增强接收 FIFO
```

它们和普通 MB 都是不同的接收方式。

## 4. Legacy RX FIFO 的特点

| 项目     | Legacy RX FIFO                 |
| ------ | ------------------------------ |
| 数据长度   | 主要面向 Classic CAN 8 byte        |
| 接收方式   | FIFO，先进先出                      |
| 过滤方式   | ID Filter Table                |
| RAM 占用 | 会占用固定一段 MB RAM                 |
| 优点     | **适合接收很多 ID，软件按 FIFO 取**       |
| 缺点     | 灵活性不如独立 MB，FD 支持/大 payload 不适合 |
| 名字含义   | 老式 Rx FIFO                     |

## 5. 和 CAN FD 的关系

重点：

> Legacy RX FIFO 主要是 Classic CAN 时代的接收 FIFO，不适合拿来做 CAN FD 64-byte payload 接收。

因为你截图里也能看到，FIFO 数据区只有：

```text
Data Byte 0..7
```

也就是 8 byte。

==**CAN FD 如果要接收 32 byte、64 byte payload，更适合用**：
**普通 Rx MB**
**或芯片支持时用 Enhanced Rx FIFO**==


## 6. 当前工程有没有用

你当前工程 `CanController_0 / FLEXCAN_1` 生成配置里：

```c

Enhanced Rx FIFO enabled = FALSE
```

所以当前工程 **没有使用 Legacy RX FIFO**。

当前工程主要用的是：

```text
普通 Rx MB
```

例如：

```text
CAN1:
  Rx MB0..MB14
  Tx MB15..MB18

CAN2:
  Rx MB0
  Tx MB1
```

## 一句话总结

> `Legacy RX FIFO` 是 FlexCAN 的旧版接收队列模式：用一块固定 FIFO 区域加 ID Filter Table 批量接收 Classic CAN 报文；它会占用前面一段 MB RAM，当前工程没启用，CAN FD 场景通常更关注普通 MB 或 Enhanced Rx FIFO。


# enhanced Rx FIFO又是什么
![](assets/2026.06.05%20CAN驱动笔记/file-20260605173127333.png)
`Enhanced Rx FIFO` 可以理解成 FlexCAN 的 **新版增强接收 FIFO**。

它和 `Legacy RX FIFO` 一样，都是“用 FIFO 队列接收报文”，但能力更强，尤其是更适合 **CAN FD**。

一句话：

> `Enhanced Rx FIFO` 是 FlexCAN 新版接收队列，可以接收 CAN FD 大 payload，带更强的过滤、ID hit、timestamp 等信息；但不是每个 FlexCAN 实例都有。

## 1. 它解决什么问题

普通 Rx MB 是这样：

```text
一个 Rx MB 接一个 ID 或一类 ID
```

如果你要接很多报文 ID，就要配很多 Rx MB。

Legacy Rx FIFO 是这样：

```text
固定 8-byte FIFO + ID Filter Table
```

但它偏 Classic CAN，数据区主要是 8 byte，不适合 CAN FD 64 byte。

Enhanced Rx FIFO 则是：

```text
增强 FIFO entry
  CS / ID / DATA 0..63
  ID HIT
  timestamp
```

它可以把多种 ID 的报文先进先出地放到 FIFO 里，而且每个 FIFO element 可以支持更大的 CAN FD payload。

## 2. 你截图这张表说明什么

你截图里的结构从：

```text
0x2000
```

开始，说明 Enhanced Rx FIFO 有独立的 FIFO memory area。

结构大概是：

```text
Enhanced Rx FIFO element:
  0x2000: CS word
          EDL / BRS / ESI / IDE / RTR / DLC / timestamp
  0x2004: ID word
  0x2008..0x2044:
          Data byte 0..63
  IH_OFF:
          ID HIT
  TS_OFF:
          HR timestamp
```

重点看几个字段：

| 字段 | 含义 |
|---|---|
| `EDL` / `FDF` | 这是 CAN FD frame |
| `BRS` | 该帧是否使用 bit rate switch |
| `ESI` | Error State Indicator |
| `DLC` | 数据长度 |
| `ID` | 收到的 CAN ID |
| `Data byte 0..63` | 最多 64 byte payload |
| `ID HIT` | 命中了哪个 filter element |
| `HR TIME STAMP` | 高分辨率时间戳 |

所以它比 Legacy Rx FIFO 明显强很多。

## 3. 和 Legacy Rx FIFO 区别

| 对比项 | Legacy Rx FIFO | Enhanced Rx FIFO |
|---|---|---|
| 定位 | 旧版 FIFO | 新版增强 FIFO |
| CAN FD 支持 | 不适合 FD 大数据 | 支持 CAN FD |
| Payload | 主要 8 byte | 可到 64 byte |
| Filter | Legacy ID Filter Table | Enhanced Rx FIFO filter elements |
| Timestamp | 普通 timestamp | 可带 high-resolution timestamp |
| ID hit 信息 | 有限制 | 明确记录 ID HIT |
| RAM 区域 | 占用前面一段 MB RAM | 有增强 FIFO 专用区域 |
| 适合场景 | Classic CAN 多 ID 接收 | CAN FD、多 ID、大 payload、高吞吐接收 |

## 4. 和普通 Rx MB 区别

| 对比项 | 普通 Rx MB | Enhanced Rx FIFO |
|---|---|---|
| 接收方式 | 指定邮箱接收 | 队列先进先出 |
| 配置方式 | 每个 MB 配 filter / ID | 配一组 FIFO filter |
| 报文顺序 | 看各 MB 状态 | FIFO 顺序天然保留 |
| 适合报文 | 少量关键 ID、确定性强 | 大量 ID、连续接收 |
| 调试方式 | 看 MB index | 看 FIFO status、ID HIT |
| RAM 使用 | 每个 MB 单独占用 | FIFO 元素统一管理 |

```text
FlexCAN_1..5 不支持 Enhanced Rx FIFO
```

你现在不用深入，除非遇到这些需求：

2. 需要 CAN FD 64-byte 报文排队接收。

## 一句话总结

> `Enhanced Rx FIFO` 是 FlexCAN 的新版增强接收队列，支持 CAN FD 大 payload、更多过滤和时间戳信息；它比 Legacy Rx FIFO 强，但 S32K324 当前只有 `FlexCAN_0` 支持，而你的工程用的是 `FlexCAN_1/2`，所以当前不用深入。


第一，它们主要是为了解决 **大量报文 ID 的接收**，不是发送。

```text
Legacy Rx FIFO
Enhanced Rx FIFO
```

名字里都有 `Rx`，所以重点是接收。发送一般还是靠 Tx MB/Tx mailbox，不用 Rx FIFO。

第二，不是“用同一块地址来收发”，而是：

> 用一块固定的 FIFO 接收区域，加一组过滤表，把多个 ID 命中的报文按顺序排队给软件读。

可以这样理解。

## 普通 Rx MB

普通方式是：

```text
ID 0x100 -> Rx MB0
ID 0x101 -> Rx MB1
ID 0x200 -> Rx MB2
ID 0x300 -> Rx MB3
```

每个 ID 或每类 ID 占一个邮箱。ID 多了以后，MB 很快不够。

## Rx FIFO

FIFO 方式是：

```text
Filter Table:
  接收 ID 0x100
  接收 ID 0x101
  接收 ID 0x200
  接收 ID 0x300
  ...

所有命中的报文
      |
      v
同一个 Rx FIFO 队列
      |
      v
软件按先进先出顺序读取
```

**所以它不是“每个 ID 一个地址”，而是很多 ID 共用一个 FIFO 队列。**

更准确地说：

```text
多个 ID filter -> 一个 FIFO 接收队列 -> 软件从固定 FIFO 读入口取报文
```

## 为什么适合大量 ID

因为你不用给每个 ID 单独分一个 MB。

例如有 40 个低优先级状态报文：

普通 MB：

```text
可能需要很多 Rx MB
```

Rx FIFO：

```text
配置 40 个 filter element
报文进同一个 FIFO
软件循环读取
```

这样节省 Rx MB，也更适合“很多 ID 都要收，但每个都不是特别关键”的场景。

## Legacy 和 Enhanced 的共同点

共同点：

```text
都是接收 FIFO
都有 filter table
都是多个 ID 共用一个队列
软件按顺序读
可以减少普通 Rx MB 的占用
```

## 它们的区别

```text
Legacy Rx FIFO:
  老版本
  主要适合 Classic CAN
  payload 基本按 8 byte 理解
  会占用 FlexCAN 前面一段 MB RAM

Enhanced Rx FIFO:
  新版本
  支持 CAN FD 大 payload
  可到 64 byte
  filter/timestamp/ID hit 能力更强
  不是每个 FlexCAN 实例都有
```

# 标准帧和扩展帧是区别CAN和CAN FD？
![](assets/2026.06.05%20CAN驱动笔记/file-20260605173127327.png)

# 什么是标准帧和扩展帧？
`标准帧` 和 `扩展帧` 说的是 **CAN ID 的长度不同**，不是说 CAN 还是 CAN FD。

|类型|CAN ID 长度|ID 范围|例子|
|---|---|---|---|
|标准帧 Standard Frame|11 bit|`0x000` 到 `0x7FF`|`0x72B`、`0x7AB`、`0x342`|
|扩展帧 Extended Frame|29 bit|`0x00000000` 到 `0x1FFFFFFF`|`0x18DAF110`、`0x18DA10F1`|

## 1. 标准帧是什么

标准帧使用 **11 位 CAN ID**。

```text
Standard ID = 11 bit
范围：0x000 ~ 0x7FF
```

例如：

```text
0x123
0x332
0x7DF
```

这些都是标准 ID。

你工程里很多配置就是标准 ID，例如：

```text
0x1D2
0x332
0x7DF
```

## 2. 扩展帧是什么

扩展帧使用 **29 位 CAN ID**。

```text
Extended ID = 29 bit
范围：0x00000000 ~ 0x1FFFFFFF
```

例如：

```text
0x18DAF110
0x18FEF100
0x0CF00400
```

**这些就是扩展 ID，商用车 J1939 里很常见。**

## 3. 为什么会有两种

早期 CAN 只有 11 位 ID，最多：

```text
2^11 = 2048 个 ID
```

后来有些协议觉得 ID 不够用，就扩展成 29 位：

```text
2^29 个 ID 空间
```

所以叫：

```text
标准帧：11-bit ID
扩展帧：29-bit ID
```

## 4. EB 里的 STANDARD 是什么意思

你截图里的：

```text
Can ID Message Type = STANDARD
```

意思是：

> 这个邮箱发送或接收 11-bit ID 的报文。

如果配置成：

```text
EXTENDED
```

就是 29-bit ID 报文。

## 5. 和 CAN FD 的关系

这两个概念是两个维度：

```text
维度 1：ID 长度
  STANDARD = 11-bit ID
  EXTENDED = 29-bit ID

维度 2：帧格式
  Classic CAN = 最多 8 byte
  CAN FD = 最多 64 byte
```

所以可以组合：

| ID 类型 | 帧格式 | 例子 |
|---|---|---|
| 标准帧 | Classic CAN | 11-bit ID，最多 8 byte |
| 扩展帧 | Classic CAN | 29-bit ID，最多 8 byte |
| 标准帧 | CAN FD | 11-bit ID，最多 64 byte |
| 扩展帧 | CAN FD | 29-bit ID，最多 64 byte |

所以：

```text
STANDARD 不等于 Classic CAN
EXTENDED 也不等于 CAN FD
```

## 6. 直观记忆

可以这样记：

```text
标准帧 / 扩展帧：问的是 ID 有多长
CAN / CAN FD：问的是数据区能放多少、能不能切高速
```

一句话：

> 标准帧是 11 位 ID，扩展帧是 29 位 ID；它们只描述 ID 长度，和是不是 CAN FD 不是同一个概念。




# 什么是CAN busoff?

![](assets/2026.06.05%20CAN驱动笔记/file-20260605173208790.png)
==`CAN BusOff` 是 CAN 控制器的一种 **严重错误保护状态**。==

一句话：

> ==当一个节点在 CAN 总线上连续出错太多次，CAN 控制器会认为“我可能正在破坏总线”，于是主动退出总线通信，这个状态就叫 `BusOff`。==

## 为什么会 BusOff

CAN 控制器内部有两个错误计数器：

```text
TEC = Transmit Error Counter，发送错误计数器
REC = Receive Error Counter，接收错误计数器
```

当发送错误不断增加，尤其是：

```text
TEC >= 256
```

节点就进入：

```text
BusOff
```

进入 BusOff 后，这个节点通常：

```text
不再发送报文
不再 ACK
不再参与正常通信
需要软件或硬件恢复
```

## 常见导致 BusOff 的原因

最常见不是软件函数错，而是总线通信条件不对：

```text
波特率不一致
采样点不合适
CANH/CANL 接反
没有终端电阻
只有一个节点发送，没有其他节点 ACK
收发器处于 standby/sleep
CAN FD / BRS 配置不一致
Data bitrate 不一致
线束/EMC/物理层问题
```

例如只有一个节点在发，没有任何节点应答：

```text
发送节点发一帧
没有收到 ACK
发送错误计数增加
不断重发
TEC 越来越高
最后 BusOff
```

## EB 里这个配置是什么意思

你截图里的：

```text
Can BusOff Processing Type = INTERRUPT
```

意思是：

> 当 FlexCAN 进入 BusOff 时，用中断方式通知 CAN Driver。

也就是流程：

```text
FlexCAN 检测到 BusOff
    |
    v
触发 BusOff interrupt
    |
    v
Can_43_FLEXCAN Driver 处理中断
    |
    v
调用 CanIf_ControllerBusOff()
    |
    v
上层 CanIf / BswM / ComM 决定怎么恢复
```

如果这里配置成 `POLLING`，就不是立即中断通知，而是靠周期调用：

```text
Can_MainFunction_BusOff()
```

去轮询检查。

## INTERRUPT 和 POLLING 区别

| 配置 | 含义 | 特点 |
|---|---|---|
| `INTERRUPT` | BusOff 发生后立即中断上报 | 响应快，适合实时处理 |
| `POLLING` | 周期函数检查 BusOff | 实现简单，但有延迟 |

你工程里配的是：

```text
Rx = INTERRUPT
Tx = INTERRUPT
BusOff = INTERRUPT
```

说明 CAN 事件主要靠中断及时处理。

## BusOff 后怎么恢复

通常有两类：

### 1. 自动恢复

你工程里之前看到：

```text
CanAutoBusOffRecovery = true
```

表示底层支持自动 bus-off recovery。

但注意：AUTOSAR 项目里最终能不能恢复，还要看上层策略，比如：

```text
CanIf
BswM
ComM
EcuM
Dem
```

### 2. 软件控制恢复

常见流程：

```text
CanIf_ControllerBusOff
    |
    v
上层停止 controller
等待一段恢复时间
重新 SetControllerMode STARTED
恢复通信
```

## 调试时怎么看

如果怀疑 BusOff：

```text
1. 看 CanIf_ControllerBusOff 是否被调用
2. 看 FlexCAN error/status register
3. 看 TEC/REC 计数
4. 用 CAN 分析仪看 ACK error / bit error / form error
5. 示波器看 CANH/CANL
6. 检查终端电阻、波特率、BRS、收发器 standby
```

一句话记忆：

> BusOff 是 CAN 节点连续通信错误太多后的自我隔离状态；EB 里的 `Can BusOff Processing Type = INTERRUPT` 表示一旦 BusOff，CAN Driver 通过中断尽快上报给上层处理。



# CAN 和 CAN FD 的区别
CAN 和 CAN FD 的核心区别：**CAN FD 能发更长、更快的数据帧**。

|对比项|Classical CAN|CAN FD|
|---|---|---|
|单帧最大数据|8 字节|64 字节|
|DLC|0 到 8|0 到 15|
|数据段速率|固定，和仲裁段一样|可切到更高速率|
|BRS 位|没有|有，表示数据段加速|
|CRC|较短|更强 CRC|
|兼容性|老 CAN 节点可识别|老 CAN 节点不能直接接收 CAN FD 帧|
|典型用途|普通信号、控制报文|大数据量诊断、刷写、标定|

最直观：

```
CAN:
最多 8 byte
例如：06 50 03 00 32 00 C8 00

CAN FD:
最多 64 byte
例如：00 23 22 F1 86 F1 90 ... 一帧放 35 byte UDS 数据
```

你现在的诊断请求：

```
00 23 22 F1 86 F1 90 ...
```

**长度超过 8 字节，所以它必须走 CAN FD，否则 Classical CAN 单帧放不下。**

注意不要混：

```
标准帧 / 扩展帧：说的是 CAN ID 长度，11 bit 还是 29 bit
CAN / CAN FD：说的是帧格式和数据长度能力
```

所以可以组合：

```
标准 CAN 帧：11bit ID + 最多 8 byte
扩展 CAN 帧：29bit ID + 最多 8 byte
标准 CAN FD 帧：11bit ID + 最多 64 byte
扩展 CAN FD 帧：29bit ID + 最多 64 byte
```

你当前常见的是：

```
0x72B / 0x7AB：11bit 标准 ID
CAN FD：因为 Data length 可以到 35/48/64
```
