ComM 的作用一句话：
# ComM
**ComM 是 AUTOSAR 中负责“通信需求仲裁与网络通信模式控制”的 BSW 模块。**
![](assets/2026.06.24%20bsw中ISOLAR配置--ComM(系统服务层)/file-20260625091059786.png)

更精确地说：

**ComM 接收 SWC/BSW 用户对通信的请求，按 Channel 统一仲裁，并通过 BusSM/Nm/EcuM/BswM 等模块控制网络在 `NO_COMMUNICATION`、`SILENT_COMMUNICATION`、`FULL_COMMUNICATION` 等状态间切换，从而在通信可用性和功耗之间做管理。**

1. ComM 模块负责 PNC 的状态机切换，并提供各个 PNC 的当前状态
2. 协调多个 PNC 状态切换请求（来自于不同网段，或者来自于内部请求）

<mark style="background: #FF5582A6;">它本身**不负责报文收发**，也**不做信号/PDU 路由**；这些分别属于 `Com`、`PduR`、`CanIf` 等模块。ComM 管的是：**这个网络现在该不该通信、通信到什么程度、由谁请求、何时释放。**</mark>

![](assets/2026.06.24%20bsw中ISOLAR配置--ComM(系统服务层)/file-20260625090633280.png)


| `ComMChannel "Can_Network_Channel_Can_Network"`               | 一个逻辑通信通道，通常对应一个 CAN 网络。ComM 会对它管理 `NO_COMMUNICATION / SILENT_COMMUNICATION / FULL_COMMUNICATION`。 |
| ------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |

** `ComMUser` 请求通信 → 通过 `ComMUserPerChannel` 关联到 `ComMChannel` → ComM 仲裁后控制该 Channel 进入对应通信模式 → 通过 BusSM/Nm/BswM 等影响实际网络状态。**

所以你这份配置可以理解为：  
**这个 ECU 配了 2 个 ComM 通道，一个普通 CAN 网络，一个 XCP CAN 网络；每个通道各有一个对应 User 可以请求通信；另外还配置了 1 个 PNC 用于局部网络管理。**


## `ComMNetworkManagement`

![](assets/2026.06.24%20bsw中ISOLAR配置--ComM(系统服务层)/file-20260625090633271.png)


| `ComMNetworkManagement` | 定义该 Channel 如何和网络管理交互，例如是否使用 Nm、如何配合唤醒/休眠/BusSM。对 CAN 来说通常关联 CanSM/Nm 行为。 |
| ----------------------- | ------------------------------------------------------------------------- |

** `ComMNetworkManagement` 就是在告诉 ComM：这个通信通道要不要使用网络管理 Nm，以及 ComM 该怎么和 Nm 配合唤醒、保持、释放网络。**

它不是 `CanNm` 本身，而是 **ComM 对某个 `ComMChannel` 的网络管理策略配置**。

按你截图里的配置：

| 配置项                                 | 通俗解释                                                                                                                               |
| ----------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| `ShortName = ComMNetworkManagement` | 这个配置容器的名字，本身不影响功能。                                                                                                                 |
| `ComMNmVariant = FULL`              | 这条 **CAN 通道使用完整的网络管理**。也就是说，当有用户请求通信时，ComM 会调用 Nm，例如 `Nm_NetworkRequest()`，让网络保持唤醒；释放通信后，再交给 Nm 协调整个网络进入睡眠。                        |
| `ComMNmLightTimeout = 10`           | 轻量网络管理相关的超时时间，单位是秒。可以理解为“进入可睡眠状态后，不要立刻睡，先等一段时间”。你这里是 10 秒。当前 `ComMNmVariant = FULL` 时，**主要睡眠/唤醒节奏通常由 Nm/CanNm 配置决定，这个值不是最核心的控制点。** |
| `ComMPncNmRequest = false`          | **如果 PNC 状态请求通信，不额外触发 `Nm_NetworkRequest()`。也就是 Partial Network 的请求不会主动通过 Nm 去拉起整个网络。**                                            |
|                                     |                                                                                                                                    |
|                                     |                                                                                                                                    |

一句话总结：

** `ComMNetworkManagement` 决定这个 ComM Channel 是“自己简单管通信保持时间”，还是“**通过 Nm 正式参与整车网络的唤醒和休眠管理**”。你这里配置的是 `FULL`，表示它走完整 Nm 管理。**

## ComMPnc_0

![](assets/2026.06.24%20bsw中ISOLAR配置--ComM(系统服务层)/file-20260625090633269.png)

PNC 可以这样理解：

**ComMChannel 管的是“整条网络”；PNC 管的是“这条网络里面某一组功能/节点”。**

比如一条 CAN 网络上挂了很多 ECU：

```
CAN 网络
├─ ECU A：底盘功能
├─ ECU B：车身功能
├─ ECU C：诊断/XCP
├─ ECU D：舒适功能
```

如果没有 PNC，只要某个功能需要通信，通常就是：

```
请求 CAN 通信 -> 整个 CAN Channel 进入 FULL_COMMUNICATION
```

也就是粒度比较粗：**这条 CAN 网络要么起来，要么睡。**

PNC 的作用是把这个粒度变细：

```
请求 PNC 8 -> 只表示“PNC 8 这一组功能需要通信”
```

其他不属于 PNC 8 的节点，可以不被长期保持唤醒，或者尽快回到低功耗状态。

所以 PNC 的核心用途是：

<mark style="background: #FF5582A6;">**在同一条网络上，只唤醒/保持真正需要工作的那一组节点，减少无关 ECU 被唤醒，从而省电。**</mark>

结合你截图：

|配置项|含义|
|---|---|
|`ComMPnc_0`|定义了一个局部网络簇，也就是一个 PNC。|
|`ComMPncId = 8`|这个 PNC 的编号是 8。网络上的其他 ECU 如果也配置 PNC ID 8，就表示它们属于同一个局部功能组。|
|`ComMChannelPerPnc = Can_Network_Channel_Can_Network`|这个 PNC 工作在普通 CAN 通道上，不是 XCP 通道。|
|`ComMPncMainFunctionPeriod = 0.02`|ComM 每 20ms 处理一次这个 PNC 的状态机。|
|`ComMPncWakeupSleepRequestEnabled = false`|这个 PNC 状态变化时，不额外走“带唤醒请求”的 PNC 模式通知。|

最关键的一点：

**PNC 不是一条新的 CAN，也不是一个新的报文通道；它只是 CAN/Nm 报文里的一个“功能组请求标志”。**

可以把它理解成：

```
ComMChannel：我要不要打开这条 CAN 网络？
PNC：打开这条 CAN 网络时，到底是哪一组功能真的需要保持通信？
```

你当前配置里，`ComMPnc_0` 的 ID 是 `8`，并且只关联到 `Can_Network_Channel_Can_Network`。所以它表达的是：

**普通 CAN 网络上有一个编号为 8 的局部功能组，ComM 可以单独管理这个功能组的请求、保持和释放。**

| 配置项                   | 作用                                                        |
| --------------------- | --------------------------------------------------------- |
| `ComMPncs [1]`        | Partial Network Cluster 列表。只有使用 Partial Networking 时才重要。  |
| `ComMPnc "ComMPnc_0"` | 一个部分网络簇。用于只唤醒/保持某一组通信节点，而不是整个网络都保持通信。                     |
| `ComMPncComSignal`    | PNC 和 COM 信号的关联，通常用于承载 Partial Network 请求/状态信息，例如 PN 信息位。 |
|                       |                                                           |
|                       |                                                           |


### PNC 的深刻理解
![](assets/2026.06.24%20bsw中ISOLAR配置--ComM(系统服务层)/file-20260625093846170.png)/file-20260625093846170.png)
这张图在讲一件事：
**软件功能不是按 ECU 唤醒，而是按“功能组 PNC”唤醒。**
从上到下看。
**1. 最上面：SWC 之间的功能关系**
上面一排是一个 `CompositionType`，里面有多个 `SW-C`：
- 绿色：属于 `VFC1`
- 深蓝色：属于 `VFC2`
- 浅蓝色：属于 `VFC3`
- 小白方块：通信端口 `Communication Port`

这里表达的是：这些 SWC 从功能角度互相通信，组成不同的车辆功能簇。  
可以把 `VFC` 理解成“一个功能集合”，比如：

```
VFC1：功能组 1
VFC2：功能组 2
VFC3：功能组 3
```

它不是物理 ECU，也不是 CAN 通道，而是软件功能视角。

**2. 中间：VFC 映射到 PNC**

中间这块是关键：

```
VFC1 / VFC2 / VFC3  ->  PNC1 / PNC2
```

意思是：

**某个软件功能组需要通信时，不是直接说“唤醒整车网络”，而是说“我需要某个 PNC”。**

比如图里大致表达：

```
VFC1 -> PNC1
VFC2 -> PNC1
VFC3 -> PNC2
```

也就是说：

- 如果 `VFC1` 或 `VFC2` 工作，需要保持 `PNC1`
- 如果 `VFC3` 工作，需要保持 `PNC2`

**PNC 的作用就是把“功能需要通信”变成“网络上哪些 ECU 需要被保持唤醒”。**

**3. 最下面：SWC 被部署到不同 ECU**

下面有 3 个 ECU：

```
ECU A：SW-C1、SW-C2
ECU B：SW-C3、SW-C4、SW-C5
ECU C：SW-C6、SW-C7
```

每个 ECU 内部都有：

```
SWC
↓
RTE
↓
Basic Software
↓
ECU Hardware
```

这些 ECU 又都挂在同一条 CAN Bus 上。

重点来了：

虽然它们在同一条 CAN 上，但不是每次通信都必须让所有 ECU 都保持活跃。

**4. PNC 1 和 PNC 2 表示不同的“局部唤醒范围”**

图底部的 `PNC1` 和 `PNC2` 线表示：

```
PNC1 覆盖一组 ECU/功能
PNC2 覆盖另一组 ECU/功能
```

举个通俗例子：

```
PNC1：只涉及 ECU A 和 ECU B
PNC2：只涉及 ECU B 和 ECU C
```

那么：

- 当 `PNC1` 被请求时，只需要让参与 `PNC1` 的 ECU 保持通信
- 当 `PNC2` 被请求时，只需要让参与 `PNC2` 的 ECU 保持通信
- 不相关的 ECU 可以继续睡眠或尽快回睡

这就是 Partial Networking 的价值。

**5. 没有 PNC 时会怎样**

没有 PNC 的情况下，逻辑通常是：

```
某个功能要通信
-> 唤醒 CAN 网络
-> 所有挂在这个 CAN 上的 ECU 都可能被唤醒/保持
```

这样比较浪费电。

**6. 有 PNC 后会怎样**

有 PNC 后，逻辑变成：

```
某个功能要通信
-> 判断它属于哪个 PNC
-> 只请求这个 PNC
-> 只有相关 ECU/功能保持通信
```

所以 PNC 本质上是：

**同一条 CAN 上的“局部功能唤醒标签”。**

它不是新建一条 CAN，也不是新建一个物理网络，而是在 Nm/PN 信息里告诉其他 ECU：

```
现在 PNC1 需要工作
或者
现在 PNC2 需要工作
```

相关 ECU 看到自己属于这个 PNC，就保持唤醒；不相关的 ECU 不必一直陪着耗电。

对应到你工程里的配置：

```
ComMPnc_0
ComMPncId = 8
ComMChannelPerPnc = Can_Network_Channel_Can_Network
```

意思就是：

**在普通 CAN 网络上定义了一个编号为 8 的 PNC。以后如果这个 PNC 被请求，ComM/Nm 会按 PNC 8 的逻辑去管理这组局部网络功能。**

- `ComMWakeupInhibitionEnabled = false`：不启用唤醒抑制。
- `ComMGlobalNvMBlockDescriptor`：你这里没配，表示 ComM 不把全局状态存到 NvM。

一句话总结你这页的配置：

**ComM 开了 PNC、开了 RTE、开了错误检测和模式限制，但没开唤醒抑制、没开同步唤醒、没开 PNC 网关，也没用 NvM 保存 ComM 状态。**


# 可以让 ComM 的 Channel 状态机切换的事件
总结一下可以让 ComM 的 Channel 状态机切换的事件可以来自于：
1. 来自 User 的请求 ComM_RequestComMode()函数调用
2. 来 自 Dcm 模 块 的 ComM_DCM_ActiveDiagnostic() 和 ComM_DCM_InactiveDiagnostic()函数调用
  当诊断仪与 Dcm 模块通信时，需要保证通道的正常可用状态，不能进入休眠。这时 Dcm 模块通过在适当的时机调用 ComM 模块的 ComM_DCM_ActiveDiagnostic()来请求通道保持在 FULL_COM 状态 。 完成诊断通信后 ， Dcm 模块再调用 ComM_DCM_InactiveDiagnostic()接口释放对通道的使用。这时如果没有其他用户再使用该通道，则通道最终将进入 NOCOM 模式，通道关闭，所以在前面介绍 Channel 状态管理状态切换的时候就提到了 Dcm 模块是让 ComM Channel 状态切换的一个重要来源。
3. 来 自 Nm 模 块 的 网 络 状 态 通 知 ComM_Nm_NetworkMode() 与 ComM_Nm_PrepareBusSleepMode()和 ComM_Nm_BusSleepMode()函数调用
4. 来自 Nm 的 ComM_Nm_RestartIndication()和 ComM_Nm_NetworkStartIndication()
5. 来自 EcuM 模块的唤醒通知 ComM_EcuM_WakeUpIndication()
  当某个通信通道发生唤醒事件后，EcuM 会调用 ComM_WakeupIndication()函数通知 ComM 模块。该通知会作为通道正常工作的一个触发源，将该通道状态切换到 FULLCOM 模式。这时 ComM 会调用网络管理模块的 Nm_PassiveStartup()唤醒网络管理模块。
6. ComM 模块内部定时器等
