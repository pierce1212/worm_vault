##  通过 0 x 72 B 给 ECU 发送 00 23 22 F 1 86 F 1 90 F 1 8 A F 1 8 C F 1 8 B F 1 93 F 1 95 F 1 88 F 1 91 F 1 97 F 1 84 F 1 00 F 1 01 FC 0 A FD 0 C F 1 87 F 1 02 00 00 00 00 00 00 00 00 00 00 00。周期 90 ms。 ✅ 2026-06-29
这个报文的意思是：**通过物理请求 CAN ID `0x72B`，周期 90 ms 发送一个 CAN FD Single Frame 的 `0x22 ReadDataByIdentifier` 请求。**

关键拆开看：

```
00 23 22 F1 86 F1 90 ...
```

不是说 UDS 请求从 `00 23` 开始。真正给 Dcm 的 UDS 是从 `22` 开始。

| 字节                | 含义                            |
| ----------------- | ----------------------------- |
| `00`              | CAN FD Single Frame 扩展长度格式    |
| `23`              | UDS payload 长度，`0x23 = 35` 字节 |
| `22`              | UDS SID：ReadDataByIdentifier  |
| `F1 86 F1 90 ...` | 一次读取多个 DID                    |
| 后面的 `00`          | padding，不属于 UDS 数据            |

所以 Dcm 实际收到的是：

```
22 F1 86 F1 90 F1 8A F1 8C F1 8B F1 93 F1 95
F1 88 F1 91 F1 97 F1 84 F1 00 F1 01 FC 0A FD 0C
F1 87 F1 02
```

它走的链路是：

```
0x72B
-> CanIf
-> PduR
-> CanTpRxNPdu
-> CanTpRxNSdu RX_UdsRxPhy
-> PduR
-> Dcm Rx PDU 0
-> SID 0x22
```

注意 `90 ms` 周期比较密。如果 ECU 的 `0x22` 响应还没发完，下一帧又来了，可能出现 busy、丢请求、响应错位，或者 Dcm 还没收到 TxConfirmation 就又收到新请求。

## 发送的数据中 00 23 到底是什么？

00 23 是 **CanTp 协议头**，不是 UDS 数据。CANoe 作为“发送原始 CAN FD 帧”的工具，会把整帧都发到总线上.
ECU 收到后，处理顺序是：
CanIf 收到整帧 00 23 22 F 1 86 ...
-> PduR 转给 CanTp
-> CanTp 解析 00 23
-> CanTp 知道后面 UDS 长度是 0 x 23 = 35 字节
-> CanTp 去掉 CanTp 头和 padding
-> 把真正 UDS 数据交给 Dcm。
CANoe 发整帧；CanTp 剥掉 00 23；Dcm 只看 22 后面的 UDS。


## TRACE 这里怎么看？为什么每次都有两个 0 X 72 B
![](assets/2026.06.29%20UDS故障排查/file-20260629154823639.png)

你这张 trace 其实**有 `00 23` **，在红框第一行的 Data 里：

```
[00 23] 22 F1 86 F1 90 ...
```

只是 CANoe 同时做了两层显示，所以看起来像“下面那行没有 `00 23`”。

你红框里有两行：

```
7.42870   72B   UdsRxPhy          [00 23] 22 F1 86 ...   SF   Tx
7.42870   72B   Unknown action... 22 F1 86 ...           req
```


第一行：

```
[00 23] 22 F1 86 ...
```

这是**真实发到 CAN 总线上的 CAN FD 帧数据**。`00 23` 是 CanTp Single Frame 头。

第二行：

```
22 F1 86 ...
```

这是 CANoe 诊断解释器把 CanTp 头解析掉以后，显示出来的 **UDS request payload**。

所以不是没有 `00 23`，而是：

```
总线原始帧：   00 23 22 F1 86 ...
UDS解释视图：       22 F1 86 ...
```

CANoe trace 里 `Data length = 35 48` 也能说明这点：

- `35`：UDS payload 长度，也就是 `0x23`
- `48`：CAN FD 实际 DLC 映射后的帧长度
- 多出来的部分是 CanTp 头和 padding

一句话：**发给 ECU 的物理 CAN 帧包含 `00 23`；Dcm 最终收到的 UDS 数据不包含 `00 23`。**

## 22 F 1 86 F 1 90 F 1 8 A F 1 8 C F 1 8 B F 1 93 F 1 95 F 1 88 F 1 91 F 1 97 F 1 84 F 1 00 F 1 01 FC 0 A FD 0 C F 1 87 F 1 02 00 00 00 00 00 00 00 00 00 00 00 这个具体什么含义

这个是一个 **UDS 0 x 22 ReadDataByIdentifier 请求**，意思是：一次性读取多个 DID。

你这串里真正有效的 UDS 数据应该是前 35 字节：

```
22 F1 86 F1 90 F1 8A F1 8C F1 8B F1 93 F1 95
F1 88 F1 91 F1 97 F1 84 F1 00 F1 01 FC 0A FD 0C
F1 87 F1 02
```

后面的：

```
00 00 00 00 00 00 00 00 00 00 00
```

是 CAN FD padding，不属于 UDS 请求内容。

拆开看：

| 字节      | 含义                           |
| ------- | ---------------------------- |
| `22`    | UDS SID：ReadDataByIdentifier |
| `F1 86` | DID 1                        |
| `F1 90` | DID 2                        |
| `F1 8A` | DID 3                        |
| `F1 8C` | DID 4                        |
| `F1 8B` | DID 5                        |
| `F1 93` | DID 6                        |
| `F1 95` | DID 7                        |
| `F1 88` | DID 8                        |
| `F1 91` | DID 9                        |
| `F1 97` | DID 10                       |
| `F1 84` | DID 11                       |
| `F1 00` | DID 12                       |
| `F1 01` | DID 13                       |
| `FC 0A` | DID 14                       |
| `FD 0C` | DID 15                       |
| `F1 87` | DID 16                       |
| `F1 02` | DID 17                       |

所以它的含义是：

```
读取 17 个 DID：
F186, F190, F18A, F18C, F18B, F193, F195, F188,
F191, F197, F184, F100, F101, FC0A, FD0C, F187, F102
```

UDS 格式是：

```
22 DID_H DID_L DID_H DID_L ...
```

正响应格式会是：

```
62 DID_H DID_L data DID_H DID_L data ...
```

如果某个 DID 不支持，可能返回：

```
7F 22 31
```

其中：

| 响应         | 含义                                |
| ---------- | --------------------------------- |
| `62`       | `0x22` 的正响应                       |
| `7F 22 31` | RequestOutOfRange，DID 不支持或当前条件不允许 |
| `7F 22 13` | 报文长度或格式错误                         |
| `7F 22 33` | SecurityAccessDenied，需要解锁安全等级     |
| `7F 22 78` | ResponsePending，Dcm 还在处理          |

**注意：这个请求一次读 17 个 DID，响应可能很长，ECU 大概率会用 CanTp 多帧回复。**


## 正确的7 AB 响应报文

这个红框是 ECU 通过 `0x7AB` 回给 tester 的 **UDS 0 x 22 正响应的 First Frame**。

红框数据：

```
10 DE 62 F1 86 01 F1 90 41 35 35 35 35 35 35 35 35 35 ...
```

拆开看：

| 字节                | 含义                                               |
| ----------------- | ------------------------------------------------ |
| `10 DE`           | CanTp First Frame，响应总长度是 `0x0DE = 222` 字节        |
| `62`              | UDS 正响应 SID，表示对 `0x22 ReadDataByIdentifier` 的正响应 |
| `F1 86`           | 第一个 DID                                          |
| `01`              | `F186` 的数据值，长度由 Dcm DID 配置决定                     |
| `F1 90`           | 第二个 DID                                          |
| `41 35 35 35 ...` | `F190` 的数据内容，ASCII 看起来是 `A555555...`             |

所以这行不是完整响应，只是响应的第一帧。

**因为响应长度是 222 字节，超过单帧容量，所以 ECU 必须走 CanTp 多帧：**

```
ECU -> Tester: First Frame
Tester -> ECU: Flow Control CTS
ECU -> Tester: Consecutive Frame 1
ECU -> Tester: Consecutive Frame 2
...
```

你截图下面也能看到 tester 发了 FC：

```
72B ... [30 08 0A ...]  FC.CTS  Tx
```

这个 Flow Control 的意思是：

|字节|含义|
|---|---|
| `30` |FlowControl ContinueToSend|
| `08` |Block Size = 8，ECU 每发 8 个 CF 后要等下一次 FC|
| `0A` |STmin = 10 ms，ECU 连续帧之间至少间隔 10 ms|

整体链路就是：

```
Tester -> ECU:
0x72B  00 23 22 ...   读 17 个 DID

ECU -> Tester:
0x7AB  10 DE 62 ...   First Frame，总响应 222 字节

Tester -> ECU:
0x72B  30 08 0A ...   Flow Control，允许 ECU 继续发

ECU -> Tester:
0x7AB  21 ...
0x7AB  22 ...
0x7AB  23 ...
...
```

所以红框这行代表：**ECU 已经正确进入 0 x 22 服务处理，并开始返回 222 字节的多帧正响应。**

## 多帧响应中的流控帧
对，这里又发了一个 `0x72B`，但它不是新的 `22` 读 DID 请求，而是 **Flow Control 帧**。

你选中的这行：

```
ID: 72B
Name: UdsRxPhy
Data: [30 08 0A 00 00 00 00 00]
Event Type: FC.CTS
Dir: Tx
```

含义是 tester 发给 ECU：

```
30 08 0A
```

这是 CanTp 的 Flow Control，专门用于控制 ECU 后续连续帧怎么发。

拆开：

| 字节      | 含义                                        |
| ------- | ----------------------------------------- |
| `30`    | Flow Control：Continue To Send，允许 ECU 继续发  |
| `08`    | Block Size = 8，ECU 最多连续发 8 个 CF 后要等下一次 FC |
| `0A`    | STmin = 10 ms，ECU 连续帧之间至少间隔 10 ms         |
| 后面 `00` | padding                                   |

为什么还是 `0x72B`？

因为方向是 tester -> ECU。你这个工程里：

```
tester -> ECU 请求 ID = 0x72B
ECU -> tester 响应 ID = 0x7AB
```

Flow Control 是 tester 控制 ECU 继续发响应，所以它也走 tester -> ECU 方向，也就是 `0x72B`。

完整过程是：

```
1. tester -> ECU
   0x72B: 00 23 22 ...
   读 DID 请求

2. ECU -> tester
   0x7AB: 10 DE 62 ...
   First Frame，告诉 tester：我要回 222 字节

3. tester -> ECU
   0x72B: 30 08 0A ...
   Flow Control CTS，告诉 ECU：继续发，每 8 帧停一次，间隔 10 ms

4. ECU -> tester
   0x7AB: 21 ...
   0x7AB: 22 ...
   0x7AB: 23 ...
   连续帧 Consecutive Frame
```

所以这里的 `72B` 不是“又发了一次 22 请求”，而是 **CanTp 流控帧 FC**。

## 流控帧后发了三帧连续帧第四帧是什么？
![](assets/2026.06.29%20UDS故障排查/file-20260629164427427.png)
红框里流控帧后看到的是 **3 个 CAN TP 连续帧 + 1 个 CANoe 诊断解析汇总行**。

具体是：

```
0x7AB [21] ...   CF 第 1 帧
0x7AB [22] ...   CF 第 2 帧
0x7AB [23] ...   CF 第 3 帧
0x7AB 62 F1 86 ... pos
```

你问的“第四帧”其实不是第四个连续帧。

这一行：

```
7AB  Unknown action::pos   62 F1 86 01 F1 90 ...
```

是 CANoe 把前面的：

```
First Frame + CF1 + CF2 + CF3
```

重组以后，解析出来的 **UDS 正响应内容**。

也就是说它不是总线上又发了一帧 CAN 报文，而是 CANoe 的诊断解释层显示：

```
我已经从多帧里拼出一个 UDS 正响应了
```

`pos` 表示 positive response，正响应。

为什么只发了 3 个 CF 就结束？

因为这个响应虽然 First Frame 里显示总 UDS 长度是：

```
10 DE
```

也就是：

```
0x0DE = 222 字节
```

但你的 trace 这里每个 CAN FD CF 可以带很多数据，不是 Classic CAN 的 7 字节。CAN FD 下，一个 CF 可以携带最多 63 字节左右的数据。

大概算一下：

```
总 UDS 响应长度 = 222
First Frame 已带 62 字节左右
剩余约 160 字节
3 个 CF 每帧最多 63 字节 = 189 字节
```

所以 3 个 CF 足够把剩下的数据发完。

总结：

```
[21] = 第 1 个 Consecutive Frame
[22] = 第 2 个 Consecutive Frame
[23] = 第 3 个 Consecutive Frame
pos  = CANoe 拼包后的 UDS 正响应显示，不是第四帧
```

# CANoe 发了一帧 CAN FD 报文后，ECU 还没有响应完全的时候，CANoe 又再次发了一帧报文，这样 ECU 是怎么响应的呢？是等待当前响应的连续帧都响应完了之后，再响应这次发的报文吗？
CANoe发了一帧CAN FD报文后，ECU还没有响应完全的时候，CANoe又再次发了一帧报文，这样ECU是怎么响应的呢？是等待当前响应的连续帧都响应完了之后，再响应这次发的报文吗？
![](assets/2026.06.29%20UDS故障排查/file-20260701140427043.png)
一般不会“排队等当前响应发完再处理第二个请求”。

按你这个工程的典型行为：

```
同一个诊断连接：0x72B 请求 / 0x7AB 响应
ECU 正在发 0x7AB 多帧响应时
CANoe 又发 0x72B 新请求
```

ECU 通常有几种处理方式：

| 情况                 | ECU 行为                                |
| ------------------ | ------------------------------------- |
| DCM/CanTp 忙，不能接新请求 | 新请求被拒绝或丢弃                             |
| DCM 能识别“第二请求”      | 回负响应 `7F 22 21`，意思是 BusyRepeatRequest |
| 当前响应还没发完           | 一般继续把当前 `0x7AB` 的 CF 发完               |
| 配了并行请求/队列          | 才可能并行或排队，但你这个工程不像是这种配置                |

结合你截图，ECU 的实际行为是：

```
1. ECU 发 0x7AB 首帧 FF
2. CANoe 回 0x72B 流控 FC.CTS
3. CANoe 又发了一帧新的 0x72B 请求
4. ECU 继续发 0x7AB 连续帧 CF
5. CANoe 最后拼出绿色 pos 响应
```

这说明：**第二个请求没有打断当前响应。ECU 优先把当前多帧响应继续发完。**

但是重点是：**它不一定会在发完后再处理你插进来的第二个请求**。在你这个工程里更可能是：

```
当前请求正在处理/发送响应
第二个 0x22 请求到来
DCM/CanTp 认为忙
第二个请求被拒绝、忽略，或者返回 NRC 0x21
```

你现在的周期请求如果太快，就会出现这种现象：

```
请求1 -> ECU 开始多帧响应
请求2 -> 插进来，但 ECU 忙
请求1响应继续完成
请求2不一定有独立响应
请求3又来了
```

所以正确做法是：**不要固定周期硬发这种长 `22` 请求**。应该等本次完整响应结束后再发下一次。

判断完整响应结束的方法：

1. Trace 里看到绿色 `pos 62 F1 86 ...`，说明 CANoe 已经重组完成。
2. 或者看到最后一个 CF 后，`CanTp_TxConfirmation` /诊断响应完成。
3. 再发下一次 `0x72B` 请求。

你的这个请求返回长度是 `0xDE = 222` 字节，多帧响应需要时间。周期如果设置得太短，比如 90 ms，就很容易撞上当前响应。建议先改成：

```
1000ms 或 2000ms
```

确认稳定后再慢慢缩短。



# CANOE 中我发送 1003 后，Dcm 会按 S 3 Server 时间自动退回 Default Session。打开 `Tester Present on` 后，诊断工具会周期性发送：3 E 00 也就是 `TesterPresent` 服务告诉 ECU：我这个 tester 还在线，不要退出当前 session。这个 3 E 的周期是在哪里确定的？


![](assets/2026.06.29%20UDS故障排查/file-20260701160603464.png)


|CDD 配置项|当前值|作用|
|---|---|---|
|`S3Client`|`4000 ms`|CANoe/tester 发送 TesterPresent 的周期或保持会话相关时间。|
|`S3Server`|`5000 ms`|ECU 允许非默认会话空闲的最大时间。超过这个时间没收到诊断请求，ECU 回 Default Session。|
|`TesterPresentPhys`|`0x3E00`|物理寻址 TesterPresent 请求内容。|
|`TesterPresentFunc`|`0x3E80`|功能寻址 TesterPresent 请求内容，`0x80` 表示 suppress positive response。|
|`P2Client`|`100 ms`|CANoe 发请求后等待首个响应的时间。|
|`P2Server`|`100 ms`|CDD 里描述 ECU 应在多久内首响。注意你 ECU 工程 DCM 里现在是 50ms。|
|`P2*Client`|`5500 ms`|CANoe 收到 `7F xx 78` 后继续等待最终响应的时间。|
|`P2*Server`|`2000 ms`|CDD 里描述 ECU pending 后的服务端时间。|
|`P3ClientPhys`|`100 ms`|物理请求之间的最小间隔。|
|`P3ClientFunc`|`100 ms`|功能请求之间的最小间隔。|
也就是说 CANoe 打开 `Tester Present on` 后，大概率每 4 s 发一次：

```
3E 00
```

因为你的 `S3Server = 5000ms`，所以 4 s 周期理论上能保住扩展会话：

```
S3Client 4000ms < S3Server 5000ms
```

但工程经验上，`4000ms` 离 `5000ms` 太近了，余量只有 1 s。如果总线忙、CANoe 卡顿、ECU 忙、窗口调度慢，就可能超过 5 s，ECU 自动回默认会话。

更稳的配置是：

```
S3Client = 2000ms
S3Server = 5000ms
```

也就是标准表里推荐的 2 s。

另外你现在 CDD 和 ECU 工程有一个不一致点：

```
CDD: P2Server = 100ms
ECU DCM: P2Server = 50ms
```

这个不一定导致问题，因为 tester 等 100 ms，ECU 承诺 50 ms，tester 更宽松。但如果你希望 CDD 和 ECU 完全一致，最好同步。最关键的是 `S3Client` 建议改成 2000 ms。
