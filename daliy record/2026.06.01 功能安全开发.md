## 什么是composition?
`Composition` 可以理解成 AUTOSAR 里的“组件装配图”。

单个 `SWC` 只是一个组件类型，比如：

```text
SafetyMgr
INP_SWC
OUTP_SWC
WdgMUT_SysCore
```

它定义了这个软件组件有什么 runnable、P-Port、R-Port、数据、接口。  
但它本身还不是“系统里真的有一个实例在运行”。

`Composition` 做的是把这些 SWC **实例化并连接起来**，比如：

```text
TopLevelComposition
  CPT_INP_SWC
  CPT_OUTP_SWC
  CPT_SafetyMgr
  CPT_WdgMUT_SysCore
```

这里的 `CPT_SafetyMgr` 就是 `SafetyMgr` 这个 SWC Type 的一个实例。

Composition 主要包含三类东西：

```text
Component Prototypes
```

表示有哪些组件实例，例如 `CPT_SafetyMgr`。

```text
Assembly Connectors
```

表示组件和组件之间的端口连接，例如 `SafetyMgr` 读取 `INP_SWC` 的状态，或者调用 `Dem/WdgM` 服务。

```text
Delegation Connectors
```

表示把内部组件端口暴露到 Composition 外部。

简单类比：

```text
SWC Type      = 类
Component Prototype = 对象/实例
Composition   = 系统装配图/对象关系图
Connector     = 对象之间的连线
```

所以你创建了 `SafetyMgr` SWC 后，还要把它放到 composition 里。否则 RTE 不知道系统里真的有一个 `SafetyMgr` 实例需要调度。

## 我通过isolar生成SafetyMgr.c代码，为什么没有RE_SafetyMgr_Init

对，你这个判断是对的。

如果 `IE_SafetyMgr_Init` 要在 **StartOS() 之前** 执行，那它**不应该用 RTE InitEvent**。

`InitEvent` 虽然名字叫 Init，但它仍然属于 RTE/OS 调度体系：

```text
InitEvent -> RteEventToTaskMapping -> OsTask -> RTE 调用 Runnable
```

所以它一定依赖 OS Task。OS 没启动前，RTE Event 不会被调度，Task 也不存在。

你这个工程启动链路大概是：

```text
main()
  -> EcuM_Init()
     -> EcuM_AL_DriverInitZero()
     -> EcuM_DeterminePbConfiguration()
     -> EcuM_AL_DriverInitOne()
        -> Mcu_Init()
        -> Gpt_Init()
        -> ...
        -> Target_Init()
     -> EcuM_Prv_StartOS()
        -> StartOS()
```

所以如果你要做 **OS 前安全初始化**，应该放在：

```text
EcuM_AL_DriverInitZero()
```

或者：

```text
EcuM_AL_DriverInitOne()
```

或者你们项目已有的：

```text
Target_Init()
```

我看你当前工程里 `Target_Init()` 已经有安全相关入口：

```c
Bist_Sequence();
eMcem_SPDInit();
SCST_ForegroundTest();
```

这说明你们项目目前就是把一部分 safety startup 放在 OS 前做的。

建议这样分：

```text
OS 前：
Safety_PreOsInit()
BIST / LBIST / MBIST result check
eMcem init
SCST foreground test
clock / FCCU / safety base early init
```

```text
OS 后，RTE 里：
RE_SafetyMgr_10ms
RE_SafetyMgr_100ms
SCST background test
周期性诊断监控
故障状态汇总
DEM / NvM / Com 相关上报
```

所以 `RE_SafetyMgr_Init` 如果只是为了 OS 前初始化，可以删掉对应的 `InitEvent`，或者保留但不要用于 OS 前功能。更推荐新建一个普通 C 函数，例如：

```c
void SafetyMgr_PreOsInit(void)
{
    Bist_Sequence();
    eMcem_SPDInit();
    SCST_ForegroundTest();
}
```

然后从 `Target_Init()` 或 `EcuM_AL_DriverInitOne()` 里调用。

关键原则：**StartOS 前不要依赖 RTE Runnable；StartOS 后才用 SWC/RTE。**


# BIST测试项的意义

对，这一页里每一项 `BIST_MBIST_x` 代表一个 **MBIST 分区**，也就是芯片内部某一类存储器/存储阵列的自检分组。

注意：你截图所在页是 **`Unrecoverable_MBIST`**，它不是“启用哪些测试”的主配置，而是配置：**这些 MBIST 分区如果失败，按不可恢复故障处理**。不可恢复故障通常会走 `MC_RGM` 复位路径；可恢复/非关键故障一般映射到 `FCCU`。

S32K324 对应关系如下：

| 项目 | 测试对象 | 含义 |
|---|---|---|
| `BIST_MBIST_0` | `SYS0_RAMS` | 系统 RAM 分区 0 |
| `BIST_MBIST_1` | `SYS1_RAMS` | 系统 RAM 分区 1 |
| `BIST_MBIST_2` | `DMA_TCD_RAM` | DMA TCD 描述符 RAM |
| `BIST_MBIST_3` | `CM7_0_TOP` | Cortex-M7 Core0 相关 RAM/TCM/cache RAM 分区 |
| `BIST_MBIST_4` | `CM7_1_TOP` | Cortex-M7 Core1 相关 RAM/TCM/cache RAM 分区 |
| `BIST_MBIST_5` | `FLEX_CAN_RAMS` | FlexCAN 模块内部 RAM |
| `BIST_MBIST_6` | `QSPI_PERI_RAMS` | QSPI 外设相关 RAM |
| `BIST_MBIST_7` | `EMAC_TSN_RAM` | Ethernet TSN 相关 RAM |
| `BIST_MBIST_8` | `EMAC_RAMS` | Ethernet MAC 相关 RAM |
| `BIST_MBIST_9` | `b03_ETF_RAMS` | Trace/ETF 相关 RAM |
| `BIST_MBIST_10` | `HSE_RAMS` | HSE 安全引擎 RAM |
| `BIST_MBIST_11` | `HSE_ROMS` | HSE 安全引擎 ROM |

另外还有：

| 项目 | 测试对象 | 含义 |
|---|---|---|
| `BIST_LBIST_0` | `LBIST` | 芯片逻辑电路自检，不是 RAM 测试 |

简单理解：

```text
MBIST = Memory Built-In Self-Test
测试 SRAM / TCM / cache RAM / 外设 RAM / HSE RAM-ROM 等

LBIST = Logic Built-In Self-Test
测试芯片数字逻辑电路
```

NXP 的 AN14969 里说明：S32K3 的 BIST 分区分为 MBIST 分区和 LBIST 分区；S32K314/S32K324/S32K344 的 MBIST ID `0..11` 对应上面这些分区。[NXP AN14969](https://www.nxp.jp/docs/en/application-note/AN14969.pdf)

对你现在配置来说，建议先不要随便删这些 `Unrecoverable_MBIST_x`。如果某个 RAM 分区 BIST 失败，而应用还继续使用那块 RAM，安全上是不成立的。通常 startup safety boot 阶段全部按 unrecoverable 处理比较保守。

# BIST又有两种模式，分别是off-line和on-line，其中off-line就是不需要软件的干预，硬件自己完成BIST测试，而on-line则需要软件来控制，S32K3只支持on-line模式。


![](assets/2026.06.01%20功能安全开发/file-20260602133851608.png)


按 embedded-c-mcu 的理解，这段可以比较明确地读成：

**S32K3 这个芯片的 STCU2 只支持 Online BIST sequence 这一类。**

因为 54.7.5 说 “supports these types”，但表里只有一项：`Online BIST sequence`。它的定义也很明确：

> 运行时，由软件请求；应用软件通过写 STCU2 寄存器配置并启动 BIST。

所以严格按这段 RM，**不能说 S32K3 支持 Offline BIST sequence**。至少在这个芯片的 STCU2 章节里，NXP 没列 offline 类型。

但这里有个容易混的点：

**DCF/DCD 上电自动触发** 不等于 RM 表里的 “Offline BIST sequence”。  
它更像是：上电/复位阶段，硬件根据 DCF/DCD 把配置加载进 STCU2，然后触发 STCU2 的 BIST 流程。底层仍然是 STCU2 的这套 Online BIST sequence/寄存器机制，只是“写寄存器和启动”的动作不是 APP 代码显式做的，而是在 reset boot sequence 中由 DCF/DCD 机制完成。

所以我的理解是：

- **BIST sequence type 层面**：S32K3 只列了 `Online BIST sequence`。
- **触发方式层面**：可以是 APP 软件写 STCU2 寄存器触发，也可能通过 DCF/DCD 在上电复位阶段加载/触发。
- **安全影响层面**：即使叫 Online，MBIST/LBIST 仍然是侵入式的，尤其 MBIST 仍可能破坏 RAM 内容，不能理解成“系统正常运行中无影响后台自检”。

54.7.6 的意思是：即使是 Online BIST，也不是你想怎么组合都行，NXP 只支持某些芯片特定的 BIST sequence/channel 组合，要看后面的 chip-specific STCU2 information。

# 如果是通过DCF/DCD来进行触发，应该怎么做
如果要通过 **DCF/DCD 上电触发 STCU2 BIST**，流程不是在 APP 里加 `Bist_Run()`，而是把 STCU2 相关配置做成 **DCF records**，烧进 **UTEST Flash**。上电/POR 时芯片 reset sequence 会处理这些 DCF records，把配置加载到 STCU2 并触发 BIST；APP 启动后只读结果。

**推荐流程**
1. 准备依据  
   需要用 S32K3 RM 的 DCF/DCF client 说明，通常还会有一个 NXP 提供的 `S32K3xx_DCF_clients.xlsx` 或类似表格。它会告诉你哪个 DCF client 对应 STCU2、哪些 bit/field 控制 BIST。不要自己拍脑袋写 DCF word。

2. 配置 STCU2 BIST sequence  
   先决定跑哪些 BIST：
   - 只 MBIST？
   - MBIST + LBIST？
   - 哪些 reset domain/channel？
   - 是否需要 self-test done 后 functional reset？
   - 是否配置 `RESET_B`/pin safe state 行为？

   你工程当前软件 `Bist_Run()` 会写这些 STCU2 寄存器作为参考：
   - `STCU_WDG`
   - `STCU_CFG`
   - `STCU_ALGOSEL`
   - `STCU_BSTART`
   - `STCU_MB_CTRL[n]`
   - `STCU_LB_CTRL`
   - `STCU_LB_PCS`
   - `STCU_LB_MISREL/MISREH`

   但这些值不能直接等同于 DCF records，DCF 要按 RM 的 DCF client 格式编码。

3. 生成 DCF/DCD records  
   用 NXP 工具或 EB tresos/S32 Design Studio 的配置能力生成，或者按 DCF client 表手工生成 hex/srec。关键是生成后的地址必须落在 UTEST 区域。

   S32K324 UTEST 地址在你工程里是：
   - `0x1B000000 - 0x1B001FFF`

   见 [C40_Ip_Cfg.h](E:/github/ECAS_RTA_S32K324GHS_Heating/BasicSoftware/integration/mcal/src/gen/include/C40_Ip_Cfg.h:182)。

4. 烧录 UTEST  
   用调试器、S32 Flash Tool、J-Flash、Trace32、PEmicro 或量产烧录器把 DCF records 写入 UTEST。注意：这通常不是 APP hex 的一部分。你当前工程里我解析过 `output` 和 `boot` 下的 hex，都没有看到 UTEST 地址段，所以现在这些镜像本身看不出已经带了 DCF/DCD。

5. 上电验证  
   必须真正断电上电 POR。然后在 APP 早期、`Bist_Run()` 之前停住，读：
   - `MC_RGM.DES.F_POR`
   - `MC_RGM.FES.ST_DONE`
   - `STCU_RUNSW = 0x403A0004`
   - `STCU_ERR_STAT = 0x403A0024`
   - `STCU_LBESW0 = 0x403A005C`
   - `STCU_MBESW0 = 0x403A014C`
   - `STCU_LBSSW0 = 0x403A004C`
   - `STCU_MBSSW0 = 0x403A010C`

   APP 里再调用 `Bist_GetExecStatus(BIST_SAFETYBOOT_CFG)`，如果 DCF BIST 成功跑过，应该不是 `BIST_NORUN`。

**你代码上要配合的点**
当前 [MC_bist.c](E:/github/ECAS_RTA_S32K324GHS_Heating/CDD/SafeTpack/Bist/src/MC_bist.c:98) 在 `BIST_NORUN` 时会调用 `Bist_Run()`。验证 DCF/DCD 时要临时避免这一步，或者至少先记录状态，否则软件触发会掩盖“上电 DCF 有没有真正触发”。

一句话：**DCF/DCD 触发 BIST = 把 STCU2 startup BIST 配置烧到 UTEST；POR 时硬件处理；APP 只读状态，不再主动 `Bist_Run()`。**


# 当时BIST测试需要用到时钟，也就是需要Mcu_Init，我觉得是不是放在这里比较好

![](assets/2026.06.01%20功能安全开发/file-20260602144030068.png)

对，你这个判断基本是对的：**如果是 APP 软件触发 STCU2 BIST，位置应该放在 `Mcu_Init()` 和时钟初始化完成之后，而且越靠前越好。**

但更准确一点，不建议放在 `Mcu_Init()` 前，也不建议夹在 `Mcu_Init()` 和 `Mcu_Integration_InitClock()` 中间。应该放在：

```c
(void)Mcu_Init(...);
(void)Mcu_Integration_InitClock();

/* 这里比较合适：Bist_Sequence(); */

(void)Gpt_Init(...);
(void)VectorTableSysCore_Init();
(void)Port_Init(...);
...
```

原因是：

1. **STCU2 BIST 需要 MCU/clock 条件**  
   你工程的 `Mcu_Integration_InitClock()` 里会：
   - `Mcu_SetMode()`
   - `Mcu_InitClock()`
   - 等 PLL lock
   - `Mcu_DistributePllClock()`

   所以 BIST 应该等这里完成后再跑。

2. **要早于 GPT、Port、Mcl、Dio、Adc、Spi、Fls、Can、Wdg、Pwm 等初始化**  
   因为 MBIST 对 RAM/存储阵列可能是破坏性的，跑完后不能假设这些模块的 RAM 状态、驱动状态、buffer、global variable 还可靠。

3. **不要等到 `Target_Init()` 之后**  
   当前 `Target_Init()` 里面还会启动 timer、I2C、SPI async mode、ADC 等。BIST 放到这些后面就太晚了。

4. **尤其要早于 `Wdg_Init()`**  
   否则 BIST 期间普通 watchdog 可能干扰流程。

所以我的建议位置是：

**`EcuM_AL_DriverInitOne()` 中，`Mcu_Integration_InitClock()` 之后，`Gpt_Init()` 之前。**

不过还有一个关键前提：如果 `Bist_Run()` 触发的是 MBIST，那么测试完成后应该依赖 STCU2 的 self-test done reset，复位后重新初始化 RAM/BSS/stack/BSW；不要在同一次执行流里继续跑后面的 init。否则 RAM 被破坏后继续初始化，风险很大。

另外我刚才看了当前工程，`SafetyMgr_PreOsInit()` 现在好像没有实际被调用到，只剩定义和 `extern`。所以如果你要启用，确实需要重新挂到一个明确位置，而不是假设它已经在 `Target_Init()` 里跑。
