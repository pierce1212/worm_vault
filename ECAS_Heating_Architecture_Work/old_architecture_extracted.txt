1 Introduction/概述
1. Purpose/目的
    本文档在《SoftwareRequirements_v1.1》的基础上，对软件的总体结构框架及软件处
理机制、流程进行描述；设计和接口规范则对软件中的各功能模块、接口和采用的数据
结构进行定义和描述。该文档为软件的详细设计提供依据，并为软件开发任务的分配提
供输入。
2. Scope/适用范围
    本文档适用于本产品/项目软件架构设计和详细设计阶段。
3. StackHolders/读者对象
     ➢ 项目软件开发和测试人员。

COMM
|编号                |控制器          |ASIL等级         |
|SUM2-JGSJ-MM-002    |SUM2            |B                |


    ComM作为通信的总开关，管理着CAN的通信网络，主要有三种状态：Full
Communication（既接收，又发送）、Silent（只接收，不发送）和No
Comunication（不接收，不发送）。
    根据吉利基础技术规范《Base  Tech   Software   Requirement   Specification
SUM》和SUM2功能的要求，总结出通讯管理的如下要求：
    1. 上电初始化，SUM2两路CAN都处在No Comunication状态。
    2.       初始化后，XCP进入Full       Communication状态，Shutdown后进入No
       Communication状态。
    3.
       初始化后，ChassisCANFD2如果是Easyload唤醒或者周期性唤醒或者网络管理报文唤
       醒，进入Full Communication状态。否则，返回No Comunication状态。
    4.    ChassisCANFD2在Full    Communication状态，由于Bussleep，进入Slient
       Communication状态，如果有网络管理报文，再进入Full
       Communication状态，否则，进入No Comunication状态。
    5. 硬欠压状态时，SUM2两路CAN进入No Comunication状态。
    6. 网络唤醒报文ID范围：0x500-0x53f。

4. 诊断管理
|编号                |控制器          |ASIL等级         |
|SUM2-JGSJ-DEM-001   |SUM2            |B                |


    诊断管理架构如下图：
[pic]

    DCM(故障诊断通信管理)，主要负责确保诊断通信数据流，及包括安全访问在内的诊
断状态控制。当总线给该ECU发送UDS协议中定义的诊断请求指令时，DCM会调用DEM、SW
C或者是其他基础软件模块提供的接口进行如诊断请求判断、诊断功能执行以及响应的反
馈。
    DEM(故障诊断事件管理)，负责直接处理与DTC相关的诊断服务，在ECU运行过程中，
一旦出现疑似故障时，会直接调用DEM的SetEventStatus来进行故障设置（Prefailed）
，debounce后即可完成将诊断故障数据的确认。
    DCM和DEM的开发主要是根据《SSDS_SUM2_APP.xlsx》提供的服务和参数进行开发。
5. Nvm存储策略
|编号                |控制器          |ASIL等级         |
|SUM2-JGSJ-NVM-001   |SUM2            |B                |


    Nvm模块的下面可以分为EA、EEPROM和FEE、DFLASH、MEMIF等模块。
                                    [pic]
    根据需求，NVM设计的主要如下：
   1. NVM模块的block使用冗余备份功能，8bit的CRC。
   2. NVM模块的block分为立即写入和下电写入两种模式。
   3. NVM上电读功能，通过函数Nvm_ReadAll在启动阶段读取所有BLOCK的数据。
   4. 启动默认值功能，在block的数据不存在或者CRC错误时，自动返回默认值。
    SUM2用了FEE、DFLASH两个模块。S32K324的DFLASH共256k，整个存储区域可以擦除1
25000次，使用期限是10年。使用FEE的控制逻辑，可以使每个BLOCK达到100万的擦写次
数。如下图
[pic]

    [pic]
    Memory                                                         Operation
Module负责控制器内存逻辑块配置、分段处理和非易失性数据管理。Security
Module负责安全访问算法和数据完整性算法。Runtime
Environment实现诊断协议的回调处理和控制整个系统的运行。上层的Bootloader
Manager负责应用程序和Bootloader启动处理及Stay    in     Boot功能。Bootloader
Application实现时钟配置、时间片处理及中断重映射功能。
                                                                        SUM2
BT软件采用了分段管理的技术，把APP文件和CAL文件划分到不通的区域，可以单独进行
更新。APP文件存储范围是0x47A800-0x7B0000,       CAL文件的存储范围是0x7B8800-
0x7D4000.
   SUM2  BT软件采用了两级Bootloader,  PBL(Primary  Bootloader),SBL(Secondary
Bootloader)，当需要更新APP或CAL文件时，把SBL下载到RAM区域，运行SBL软件才可以
擦除、写入PFLASH区域的内存。SBL可以防止APP软件无意之中破坏PFLASH区域数据，保
证数据安全。
                                                                        SUM2
BT软件数据完整性算法采用了非对称加密RSA2048算法，保证了数据的安全性。

8.10.3 功能安全软件架构动态设计Dynamic Design of Software Architecture

   软件架构中动态设计关注的是软件组件相关项的控制逻辑和相互关系，是对静态要素
的组合描述。可以通过行为视图来描述架构动态行为，例如数据流，控制流，不同状态
切换等。常见的行为视图包括，控制流图，真值表，时序图，状态图等。
● 数据流图 ——描述数据如何由输入逐步流向输出的过程。
● 控制流图 ——描述由输入经过一系列控制动作到输出的过程。
● 状态机图 ——描述系统不同状态之间的转换关系。
● 真值表 ——描述一个复杂的组合逻辑关系。
● 序列图 ——描述不同组成部分通过信息交互的时序关系。
● 结构图 ——描述组件之间的层次关系。
动态行为常见的描述方式主要有以下三种：
There are three main ways in which dynamic behaviour is commonly described:
1. 数据流图（Data Flow Diagram）；
2. 时序图（Sequence Diagram）；
3. 结构图（Structure Diagram）
本设计方案主要通过以上三种方式来描述L946项目软件架构的动态行为。

   RTAOS使用单一堆栈模型，这说明所有任务和ISR运行在单一堆栈上。当一个任务运行
时，他的堆栈使用量会正常的增加和减少，当一个任务被抢占时，高优先级任务的堆栈
使用将继续在同一个堆栈上进行，就像标准函数一样。当一个任务终止时，他正在使用
的堆栈空间将被回收，然后重新用于下一个优先级更高的任务运行
